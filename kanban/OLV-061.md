## Why

This is the most cross-cutting ticket in MVP. It's the proof that all
prior domain modules compose correctly. Get the steps in the right order
and the right transaction shape, and everything downstream gets easier.

## Scope

`POST /api/orders` — exactly the 8-step pipeline from PRD §8.3, in order:

1. **회원 검증**: caller is authenticated; status is ACTIVE.
2. **상품 판매 상태 검증**: every `productOptionId` resolves to an option
   whose product is `ON_SALE`. Reject otherwise.
3. **재고 검증 + 선점**: call `InventoryService.reserve(orderId=preview,
   items, ttl=15min)`. NB: reserve before persisting the order so we can
   bail without a half-written order.
4. **쿠폰 검증**: if `couponId` set, call `CouponService.validate()`.
5. **포인트 검증**: if `usePointAmount > 0`, check
   `PointService.spendableBalance(memberId, now) >= usePointAmount`.
6. **최종 금액 계산**:
   - subtotal = Σ (option.sale_price × quantity), snapshot from products.
   - coupon_discount = `coupon.apply(subtotal)`.
   - point_discount = `usePointAmount`.
   - shipping_fee = flat 3000원 if `subtotal < 30000`, else 0 (configurable).
   - grand_total = max(0, subtotal − coupon − point + shipping).
   - All BigDecimal, scale 0, HALF_UP.
7. **주문 생성**: insert `orders`, `order_items` (with snapshots),
   `order_price_summaries`, initial `order_status_histories` row
   `(null → CREATED)`. State transitions to `PAYMENT_PENDING` after the
   payment intent is generated in step 8.
8. **결제 요청 정보 생성**: insert a `payments` row with status `READY`,
   `requested_amount = grand_total`, generate a fresh `idempotency_key`
   (UUID). Return `{orderNo, paymentKey: payments.id, amount: grand_total,
   pgCheckoutPayload}` to the client so they can launch the PG SDK.

Request body (PRD §8.3 example shape):

```json
{
  "items": [{"productOptionId": 10, "quantity": 2}],
  "couponId": 5,
  "usePointAmount": 1000,
  "deliveryAddressId": 3
}
```

If any step fails AFTER step 3 (reservation), explicitly call
`InventoryService.release(orderIdPreview)` in the catch block.

Publish `OrderCreatedEvent(orderId)` in the same txn (via outbox).

## Acceptance Criteria

- [ ] Happy path: 3-line cart → 201 with order_no, all snapshots stored,
      reservations held, payment row READY.
- [ ] Stock-out on one of the lines → 422 + ZERO reservations / orders /
      payments persisted (verify via DB inspection in the test).
- [ ] Invalid coupon → 422 + ZERO reservations persisted.
- [ ] Insufficient points → 422 + ZERO reservations persisted.
- [ ] Concurrent identical-cart orders by 50 users on a 30-stock SKU:
      exactly 30 succeed (smoke from OLV-031 still holds end-to-end).
- [ ] Audit log + `order_status_histories` row written.
- [ ] Idempotency: replaying the same request body does NOT create a
      second order (key off a request-supplied `Idempotency-Key` header
      stored in a 24h Redis cache).

## Hints

- The "preview orderId" problem: generate the orderNo BEFORE persistence
  (sequence call) so the reservation row can carry it. The order PK is
  generated on insert.
- Wrap steps 3-8 in a single `@Transactional` so a partial failure rolls
  back. Reservations live in their own transaction inside the inventory
  service; ensure release is called explicitly on any throw.
- Use a `@TransactionalEventListener(phase = AFTER_COMMIT)` for
  `OrderCreatedEvent` publication so subscribers see committed state.
