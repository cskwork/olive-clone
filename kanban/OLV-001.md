## Why

This is the first ticket. Without a buildable Gradle project no other
ticket can land. Establish the modular-monolith package layout from PRD
§5.1 so subsequent tickets just add packages, not project skeleton.

## Scope

Initialize a single-module Gradle (Kotlin DSL) Spring Boot 3.x application
running on Java 21 with the package layout from `llm-wiki/00-architecture-overview.md`.

## Acceptance Criteria

- [ ] `./gradlew bootRun` starts on port 8080 and serves `GET /actuator/health`
      → `{"status":"UP"}`.
- [ ] `./gradlew test` passes (one trivial smoke test is fine).
- [ ] `./gradlew build` produces `build/libs/commerce-backend-*.jar`.
- [ ] Source tree contains empty package directories with package-info.java
      stubs:
      `com.olive.commerce.{member,product,search,cart,order,payment,inventory,promotion,delivery,review,admin,common}`.
- [ ] `gradle/wrapper/gradle-wrapper.properties` pinned to a current Gradle
      version (8.x or higher).
- [ ] `.gitignore` excludes `build/`, `.gradle/`, `out/`, `*.log`, `.idea/`,
      `.env*`, `application-local.yml`.
- [ ] `README.md` (project root) shows: prereqs, `./gradlew bootRun`,
      `./gradlew test`, link to `WORKFLOW.md` for the agent pipeline.

## Hints

- Use Spring Initializr-equivalent dependencies: Spring Web, Spring Data JPA,
  Spring Security, Validation, Actuator, Lombok (optional), Spring Boot Test.
- Defer DB driver to OLV-002 — leave JPA disabled at bootstrap time
  (`spring.autoconfigure.exclude=DataSourceAutoConfiguration` for now), so
  `bootRun` works without a database. OLV-002 will reverse this.
- Group: `com.olive`, artifact: `commerce-backend`, version: `0.1.0`.
- Add a single `BootstrapTest` that asserts the Spring context loads.

## Domain Brief

**무엇**: 본 티켓 이후 모든 도메인(회원·상품·주문 등 12개)이 추가될 Spring Boot 3 + Java 21 모듈러 모놀리식 프로젝트의 빈 골격을 만든다.
**왜**: 골격이 없으면 어떤 후속 티켓도 빌드/실행할 수 없다 — 즉 PRD §5.1의 “단일 Gradle 루트, 단일 애플리케이션, 단일 스키마” 전제를 코드로 못 박아야 OLV-002 이후가 패키지 추가만으로 진행된다.
**As-Is → To-Be**:
- As-Is: 저장소에 Java 코드 0줄, 빌드 도구 0개. `./gradlew bootRun`은 “file not found”.
- To-Be: `./gradlew bootRun`이 8080 포트에서 떠서 `/actuator/health`가 `{"status":"UP"}` 반환, `./gradlew test`/`build`가 통과.

- 잠금된 스택(PRD §18, llm-wiki/00-architecture-overview.md): Java 21, Spring Boot 3.x, Gradle Kotlin DSL, JPA(이번 티켓엔 비활성), Security, Validation, Actuator, Lombok, JUnit 5.
- 도메인 패키지 12개를 `com.olive.commerce.<domain>` 하위에 빈 package-info.java로 선언 — `@SpringBootApplication`이 `com.olive.commerce`에 위치하면 후속 티켓이 자동 스캔된다.
- 이번 티켓에서 DB는 **의도적으로 비활성** — `spring.autoconfigure.exclude=DataSourceAutoConfiguration`으로 빠지고, OLV-002가 한 줄을 지워 다시 켠다(kanban/OLV-002.md §Hints).
- 위험: `spring-boot-starter-security`를 추가하면 모든 엔드포인트가 기본 401 — actuator/health 200 AC를 만족시키려면 `SecurityFilterChain` bean에서 명시적 permitAll이 필요.
- 환경 사전 작업(이미 완료): `brew install openjdk@21 gradle` → JDK 21.0.11 + Gradle 9.5.0 설치. wrapper 생성 후 wrapper 버전은 안정 라인인 8.10으로 고정.

_세부: docs/OLV-001/explore/details.md_

## Plan Candidates

- **A. 단일 Gradle 모듈 + 패키지 분리(티켓 명시)**: 가장 가볍다. `com.olive.commerce.<domain>` 하위 패키지로 도메인을 격리, 후속 티켓이 패키지만 추가. 도메인 경계 강제는 패키지 컨벤션과 Hard rule(이벤트만 통신)에 의존.
- **B. Spring Modulith 도입**: `org.springframework.modulith:spring-modulith-starter-core`를 처음부터 넣어 패키지간 의존을 컴파일 타임에 검증. 장점: 경계 위반 자동 감지. 단점: 학습 곡선 + 부트스트랩 단계의 과한 의존성 — PRD에 명시되지 않음.
- **C. Gradle 멀티 모듈(subproject)**: 도메인마다 `:member`, `:product` subproject. 장점: 강한 빌드 격리. 단점: 12개 build.gradle.kts 폭증 + 단일 스키마/단일 App.run() 가정이 깨짐 — `“single Gradle root, single Spring Boot application”`(00-architecture-overview.md L11) 인바리언트 위반.

## Recommendation

**Option A 채택.** 티켓 §Scope("single-module Gradle … sub-packages per domain")와 wiki 인바리언트가 직접 요구. B/C는 부트스트랩 범위 밖이며 “Touch only what the ticket requires”(WORKFLOW.md Hard rules) 위배.
**첫 실패 테스트**: `BootstrapTest`(`@SpringBootTest`) — `contextLoads` 1개 메서드. 부트스트랩 단계에서 컨텍스트가 떠야 통과; 의존성 누락/Security 자동설정 충돌이 즉시 잡힌다. 의존성/main 클래스/SecurityFilterChain이 없으면 RED, 갖추면 GREEN.

## Learnings

**무엇**: 부트스트랩에서 발견한, Explore brief에는 없던 세 가지 invariant를 wiki에 못박았다.
**왜**: 다음 도메인 티켓이 같은 함정에 다시 빠지지 않도록 — 특히 회원 도메인(OLV-010)이 자동설정 exclude를 거꾸로 풀 때 “왜 끊었는지”를 추적할 수 있어야 한다.
**As-Is → To-Be**:
- As-Is: wiki는 “seed” 결정만 보유, OLV-001 단계의 실 구현 결정 미반영.
- To-Be: 00/98에 OLV-001 결정 로그 3건 추가, INDEX 갱신일 OLV-001로 갱신.

- starter-security만 추가해도 `InMemoryUserDetailsManager`가 임의 비밀번호를 **로그에 노출** — SecurityFilterChain만으로는 못 막는다. 자동설정 exclude가 유일한 깔끔한 해법(`98-security.md`).
- `endpoint.health.probes.enabled=true`로 켜면 `/actuator/health/liveness`·`/readiness`가 무료로 노출 — OLV-130(K8s probe)가 이 위로 직행 가능(`00-architecture-overview.md`).
- Explore에서 Gradle 9.5 설치 후 wrapper 버전을 8.10.2로 명시한 결정은 정답이었다 — Spring Boot 3.3.x의 안정 라인과 일치, 빌드 deprecation 경고가 informational에 그쳤다.
- `spring-boot-starter-data-jpa`를 의존성에는 포함하되 자동설정만 끄는 패턴은 OLV-002의 “한 줄 삭제로 활성화”가 자연스럽게 흐르도록 한다 — Hard rule “Touch only what the ticket requires”와 후속 티켓 진입 비용을 동시에 만족.

### Reasoning
- **Option A(단일 Gradle + 패키지 분리)** 채택 — 티켓 §Scope와 wiki 인바리언트(`single Gradle root, single Spring Boot application`)가 직접 요구. Spring Modulith/멀티모듈은 부트스트랩 범위 밖이며 Hard rule “Touch only what the ticket requires”에 위배.
- **트레이드오프**: 도메인 경계 강제는 패키지 컨벤션 + “이벤트로만 통신” 규약에 의존 — 컴파일러 강제는 없다. 후속 단계에서 ArchUnit 또는 Spring Modulith 도입을 검토 가능(현재 deferred).
- **DB는 의도적으로 비활성** (`DataSourceAutoConfiguration` 외 2건 exclude) — OLV-002에서 Postgres+Flyway 도입과 함께 한 줄씩 제거. 부트스트랩 단계 한정 임시 invariant.
- **HIGH 이슈 1건 자체 수정**: starter-security가 임의 비밀번호를 로그에 출력하던 동작을 `UserDetailsServiceAutoConfiguration` exclude로 제거 (`98-security.md` Decision log).
- **Follow-up(의도적 보류)**:
  - PostgreSQL/Flyway/Testcontainers → kanban/OLV-002.md.
  - 회원 도메인·실제 `UserDetailsService` → kanban/OLV-010.md (이때 `UserDetailsServiceAutoConfiguration` exclude 한 줄 제거).
  - K8s liveness/readiness probe 매핑 → kanban/OLV-130 계열(observability).
