## Why

Refunds are the second money-touching path. Closes out the order
lifecycle and is necessary for any post-MVP CS workflow.

## Scope

User endpoint:

- `POST /api/me/orders/{orderNo}/refund-request` — `{reason, items: [{orderItemId, quantity}]}`.
  Allowed only for orders in `DELIVERED` (or admin-approved at any state).
  Sets `orders.status=REFUND_REQUESTED`, inserts a `refunds` row with
  `status=REQUESTED`.

Admin endpoints (require `ORDER_ADMIN`):

- `GET /api/admin/refunds?status=&page=&size=` — list pending requests.
- `POST /api/admin/refunds/{refundId}/approve` — calls
  `PgClient.refund()`, on success sets `refunds.status=APPROVED`,
  `orders.status=REFUNDED`, restores points used, restocks inventory
  (`InventoryService.adjust(optionId, +qty, "REFUND", adminId)`), publishes
  `OrderRefundedEvent`.
- `POST /api/admin/refunds/{refundId}/reject` — `{reason}`. Sets
  `refunds.status=FAILED`, order returns to its prior status.

## Acceptance Criteria

- [ ] User can request refund only for own order in DELIVERED status.
- [ ] Admin approve: PG refund called once, order REFUNDED, inventory
      restocked, points cancel-rewound, audit log row written.
- [ ] Approve is idempotent — replay does not double-call PG nor double-stock.
- [ ] Refund amount cannot exceed `orders.final_payment_amount` minus
      prior refund amounts (partial refund support).

## Learnings

1. **Map.of() null 제약**: `Map.of()`는 null 값을 허용하지 않아 NPE 발생 가능. HashMap 사용 필요
2. **테스트용 setId 패턴**: JPA @GeneratedValue 엔티티는 테스트에서 ID를 수동 설정할 방법 필요
3. **환불 멱등성**: 상태 기반 early return으로 중복 PG 호출 방지 (이미 APPROVED 상태면 no-op)
4. **단일 트랜잭션 중요성**: 환불 승인 시 PG 호출, 재고 복구, 포인트 환원을 원자적으로 처리

- RefundService가 환불 요청/승인/거절 완전 구현
- 사용자 API: `POST /api/me/orders/{orderNo}/refund-request`
- 관리자 API: `GET /api/admin/refunds`, `POST /api/admin/refunds/{id}/approve`, `POST /api/admin/refunds/{id}/reject`

- 상태 기반 멱등성으로 PG 중복 호출 방지 (안전성)
