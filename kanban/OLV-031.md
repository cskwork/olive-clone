## Why

The single most-likely place to ship a critical bug. Concurrency is hard;
the PRD spends §10 on it for a reason. Land the locking strategy + a
property test before any code calls reserve().

## Scope

Service:

- `InventoryService.reserve(orderId, items: List<{optionId, qty}>, ttl)`
  — atomic across all items. If any line lacks stock, NONE are reserved.
- `InventoryService.commit(orderId)` — flips reservations to COMMITTED,
  decrements total_quantity, writes history rows.
- `InventoryService.release(orderId, reason)` — decrements
  reserved_quantity, sets reservation RELEASED, writes history rows.
- `InventoryService.adjust(optionId, delta, reason, adminId)` — admin
  manual stock movement, audit-logged.

Locking strategy (per `llm-wiki/30-inventory-domain.md`):

- Default: Redis `Redisson` `RLock("lock:inv:{optionId}")`, lease 5s, max
  wait 2s. Lock per option_id; sort the option_id list before locking to
  prevent deadlock across multi-line orders.
- Fallback when Redis is unavailable (configurable feature flag
  `inventory.lock.fallbackToDb=true`): `SELECT ... FOR UPDATE` on the
  `inventories` row inside `@Transactional`.

Admin endpoints under `/api/admin/inventories`:

- `GET /api/admin/inventories?productId=` — list per-option current state.
- `POST /api/admin/inventories/{optionId}/adjust` — body
  `{delta, reason}`.

## Acceptance Criteria

- [ ] Concurrent test: spin 50 virtual threads each reserving 1 unit of
      a 30-stock option. Exactly 30 succeed, 20 get
      `INSUFFICIENT_INVENTORY`. No row ever shows
      `available_quantity < 0`.
- [ ] Reserve with TTL 1s → wait 2s → release expired runs (call the
      batch step manually here; OLV-120 schedules it). Available returns
      to original.
- [ ] Commit then release on the same order is idempotent (release after
      commit is a no-op, not an error).
- [ ] Audit log entries land in `inventory_histories` for every change.
- [ ] Redis-down fallback test: stop Redis container, set the feature
      flag, repeat the concurrent test against the DB lock path. Same
      invariant holds.

## Hints

- `Redisson`'s `tryLock(waitTime, leaseTime, unit)` is the right primitive.
- For multi-line atomicity, acquire all locks first (sorted), then validate
  inventory, then mutate. Release locks in reverse order in `finally`.
- The QA stage of THIS ticket is the most important — write the
  concurrent test as the first thing you do.

## Domain Brief

**무엇**: 재고 도메인의 핵심 서비스(InventoryService)를 구현하고, Redis 분산 락(Redisson RLock)으로 동시성을 제어하며, Admin CRUD API를 제공. 50개 스레드 동시성 테스트로 정합성을 검증.
**왜**: Inventory는 가장 동시성에 민감한 도메인. "The single most-likely place to ship a critical bug" — 락 전략과 property test를 먼저 깔지 않으면 reserve() 호출 시점에 race condition이 터짐.
**As-Is → To-Be**:
- As-Is: V4 스키마만 존재 — InventoryService/Repository/Controller 0개, Redisson 빈 0개, 동시성 테스트 0개.
- To-Be: RedissonClient 빈 + InventoryService (reserve/commit/release/adjust) + Admin API + 50스레드 동시성 테스트로 AC 5건 모두 검증.

핵심 인용 5건:
- `llm-wiki/30-inventory-domain.md:20-24`: **Default = Redisson RLock** (key=`lock:inv:{optionId}`, lease 5s, wait 2s). **Fallback = SELECT...FOR UPDATE** (Redis 다운 시).
- `llm-wiki/03-infra-baseline.md:50-53`: OLV-030의 Redisson은 `RedisConnectionFactory` 위에 별도 빈으로 쌓아 올림 — Redis 자체는 baseline 보장.
- `llm-wiki/01-common-conventions.md:24`: `BusinessException(ErrorCode, String detail)`만 throw. `INSUFFICIENT_INVENTORY` 이미 정의됨.
- PRD §10.2: Multi-line orders는 **sorted option_id 순서로 락 획득** (deadlock 방지). Lock 해제는 역순 finally.
- 티켓 §Hints 줄 38: "multi-line atomicity" = 모든 락 획득 → inventory 검증 → mutation → 락 해제.

## Plan Candidates

**Option A — Redisson + InventoryService + TDD (권장)**: Redisson 3.37.0 빈 추가 → InventoryService (reserve/commit/release/adjust) → Admin API → 50스레드 동시성 테스트. **첫 테스트는 concurrent test** (50 threads, 30 stock). `INSUFFICIENT_INVENTORY`는 이미 ErrorCode에 존재. Fallback은 feature flag로 분기.
- 장점: TDD 순서 준수(Hint 강조), 락 전략이 Service에 집중, DB 폴백이 별도 경로로 분리.
- 단점: Redisson 설정이 처음이라 learning curve 존재.

**Option B — Service + API 먼저, 나중에 락 추가**: Service/Controller 먼저 구현 → 동시성 테스트 실패 확인 → Redisson 추가.
- 장점: 구현 먼저 보여줌.
- 단점: (1) 티켓 §Why 명시("Locking strategy + property test before any code calls reserve()") 위반, (2) 나중에 락 추가 시 기존 코드 리팩토 필요.

**Option C — Testcontainers 통합 없이 동시성 테스트**: 실제 Redis 대신 mock Redisson 사용.
- 장점: 테스트가 빠름.
- 단점: (1) "실제 Redis 다운" 시나리오(AC5)를 검증 불가, (2) Testcontainers baseline(Olv-003)과 패턴 불일치.
