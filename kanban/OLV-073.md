## Why

Real PGs push state changes asynchronously. PRD §6.6 / §20.4 mandates the
webhook is authoritative — client success can lie, network can drop the
confirm response, but the webhook is the truth. Build it now while the
confirm flow is fresh.

## Scope

`POST /api/payments/webhook` — PG-side push. Body shape per PG spec
(documented in `MockPgClient`). Behaviour:

1. Verify webhook signature (mock signs with HMAC-SHA256 of body using a
   shared secret in config — real PGs use their own scheme).
2. Look up payment by `paymentKey`.
3. Persist a `payment_transactions` row of kind `WEBHOOK` with the raw
   PG body — even if we end up no-oping below.
4. Reconcile state per the webhook event:
   - `APPROVED`: if our order is still `PAYMENT_PENDING`, run the same
     post-confirm steps as OLV-072 (commit inventory, mark coupon used,
     etc.). If already `PAID`, no-op + log "duplicate".
   - `FAILED` / `CANCELED`: set `payments.status` accordingly, set
     `orders.status=PAYMENT_PENDING` (still cancellable by user) or
     `FAILED` if past PG retry window. Do NOT release reservations here
     — the batch (OLV-120) owns that policy.
   - `REFUNDED`: only legal if our order is in `REFUND_REQUESTED`; set
     `orders.status=REFUNDED`, `payments.status=REFUNDED`.
5. Always 200 — return the same body twice yields the same effect.

## Acceptance Criteria

- [ ] Webhook with invalid signature → 401.
- [ ] APPROVED webhook for already-PAID order → 200, no second commit
      (idempotent).
- [ ] APPROVED webhook for an order that the client never managed to
      confirm (network failure simulated) → order flips to PAID via this
      path; outcome identical to confirm path.
- [ ] Replay protection: same `paymentKey + status` webhook posted twice
      yields no double-effect.

## Hints

- Don't trust webhook timestamps — use server-side `now()` for state
  transitions; webhook body is for verification only.
- Use a Redis SETNX with TTL on `webhook:dedup:{paymentKey}:{status}` to
  collapse rapid retries.

## Domain Brief

**무엇**: PG 웹훅 핸들러 구현 — 비동기 PG 상태 변경 푸시를 받아 결제/주문 상태를 재조정하는 API

**왜**: 클라이언트 성공 응답은 거짓일 수 있고, 네트워크는 confirm 응답을 유실할 수 있음. PRD §6.6/§20.4에 따라 웹훅이 진짜 결제 상태의 소스 of truth임

**As-Is → To-Be**:
- As-Is: PaymentService.confirmPayment()는 동기 요청만 처리. PG 비동기 webhook은 처리 불가
- To-Be: POST /api/payments/webhook이 서명 검증 → 중복 방지 → 상태 재조정(트랜잭션 기록) → 항상 200 반환

---

**기존 자산**:
- `payment_transactions` 테이블 (V9): WEBHOOK kind 지원, `PaymentTransaction.webhook()` 팩토리 메서드 존재
- `PaymentRepository`: `paymentKey`로 조회 가능 (`idx_payments_payment_key` 인덱스)
- `ErrorCode.PG_WEBHOOK_INVALID`: 400 반환용
- `StringRedisTemplate`: Redis SETNX deduplication용
- OLV-072 `processSuccessfulPayment()`: 재사용 가능한 4-7단계 로직

**Domain 이해**:
- Payment 상태: READY → REQUESTED → APPROVED (외부 FAILED/CANCELED/REFUNDED 가능)
- Order 상태: PAYMENT_PENDING → PAID → REFUND_REQUESTED → REFUNDED
- Idempotency: `(paymentKey, status)` 조합으로 Redis dedup + DB transaction row

## Plan Candidates

### Option A: PaymentService.handleWebhook() 일체형 구현
- Webhook 로직을 PaymentService 내부에 추가
- processSuccessfulPayment() 메서드를 private에서 package-visible로 변경하여 재사용
- Redis deduplication을 webhook handler 내부에서 수행
- **장점**: 코드 응집도 높음, OLV-072와 일관된 구조
- **단점**: PaymentService가 confirm + webhook 두 책임 담당

### Option B: WebhookService 분리 + PaymentService 위임
- 별도 WebhookService 생성, 서명 검증과 dedup 전담
- 상태 변경 로직은 PaymentService.publicCompletePayment() 위임
- **장점**: 관심사 분리, webhook 특정 로직 격리
- **단점**: 추가 계층, 복잡도 증가

### Option C: Controller → Service 간소 구조
- PaymentController.webhook()에서 직접 서명 검증
- Redis dedup 후 PaymentService.webhookComplete() 호출
- **장점**: 간단한 구조
- **단점**: Controller에 비즈니스 로직 유출

## Recommendation

**선택: Option A - PaymentService 확장**

**이유**:
1. Webhook 처리는 결제 도메인의 핵심 관심사 — PaymentService가 적격
2. processSuccessfulPayment() 재사용으로 코드 중복 최소화
3. Redis dedup는 웹훅 고유 사안이나 간단한 메서드로 캡슐화 가능
4. OLV-072와 일관된 구조 유지

**첫 번째 실패 테스트**:

```java
@Test
@DisplayName("Happy path: webhook APPROVED → PAID, 모든 side effects 완료")
void webhookApproved_paymentPending_to_paid() throws Exception {
    // Given: 회원, 상품, 주문(PAYMENT_PENDING), 결제(REQUESTED)
    Long orderId = givenOrderReadyForPayment(memberId, productId);
    String paymentKey = "mock-payment-key-webhook-001";

    String webhookPayload = """
        {
            "paymentKey": "%s",
            "status": "APPROVED",
            "approvedAt": "2026-05-12T12:00:00Z",
            "approvedAmount": 35000
        }
        """.formatted(paymentKey);

    String signature = mockPgClient.signWebhook(webhookPayload);

    // When: POST /api/payments/webhook
    mockMvc.perform(post("/api/payments/webhook")
            .header("X-Webhook-Signature", signature)
            .contentType(MediaType.APPLICATION_JSON)
            .content(webhookPayload))
        .andExpect(status().isOk());

    // Then: PAID 상태, inventory committed, coupon used, outbox event
    assertOrdersStatus(orderId, "PAID");
    assertInventoryCommitted(orderId);
}
```

### 주요 변경 사항

**MockPgClient.java**:
- HMAC-SHA256 기반 웹훅 서명 생성
- `getWebhookSecret()`: 테스트용 공유 시크릿 반환

**PaymentService.handleWebhook()**:
1. Redis deduplication: `webhook:dedup:{paymentKey}:{status}` 키로 5분 TTL
2. 서명 검증 실패 시 `PG_WEBHOOK_INVALID` 에러
3. 상태별 분기:
   - `APPROVED`: `processSuccessfulPayment()` 재사용 → confirm 경로와 동일한 side effect
   - `FAILED/CANCELED`: `payment.setStatus()`만 호출, 주문은 `PAYMENT_PENDING` 유지
   - `REFUNDED`: `REFUND_REQUESTED` 상태 검증 후 `REFUNDED`로 변경
4. 항상 `payment_transactions`에 WEBHOOK kind 행 기록

**PaymentController.java**:
- `X-Webhook-Signature` 헤더로 서명 수신
- 모든 예외를 200으로 변환 (PG 재시도 허용)

## Learnings

**주요 발견**:
1. **웹훅 멱등성의 두 계층**: Redis SETNX는 빠른 거름 역할, DB의 transaction row는 최종 보장. 이 이중 보호가 중복 처리를 방지함.
2. **타임스탬프 불신**: PRD에서 명시한 대로 웹훅 타임스탬프는 검증용으로만 사용. 상태 전이는 항상 서버의 `now()` 사용.
3. **코드 재사용**: OLV-072의 `processSuccessfulPayment()`를 그대로 재사용하여 confirm과 webhook 경로의 일관성 보장.

**개선 가능한 부분**:
- MockPgClient에 하드코딩된 시크릿은 실제 환경에서는 설정에서 주입 필요
- PgClient 인터페이스에 signWebhook/verifyWebhookSignature 메서드 추가를 고려 (다양한 PG 지원)
