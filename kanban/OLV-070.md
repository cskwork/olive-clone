## Why

Payment is where the most external-facing failure modes live. Schema must
hold the idempotency key, the PG callback evidence, and the refund chain.

## Scope

Create `V8__payment.sql` matching PRD §7.7:

- `payments`: id, order_id FK UNIQUE, payment_key VARCHAR(255) (PG-side
  unique key, NULLABLE until PG returns one), pg_provider VARCHAR(50),
  method VARCHAR(50) NOT NULL (CARD | KAKAO_PAY | NAVER_PAY | TOSS_PAY |
  VIRTUAL_ACCOUNT), status VARCHAR(30) NOT NULL,
  requested_amount DECIMAL(12,2), approved_amount DECIMAL(12,2) NULLABLE,
  idempotency_key UUID UNIQUE NOT NULL, requested_at, approved_at NULLABLE,
  failed_reason VARCHAR(255) NULLABLE, created_at, updated_at.
- `payment_transactions`: id, payment_id FK, kind
  (`REQUEST | APPROVE | CANCEL | WEBHOOK | REFUND`), pg_response_json JSONB,
  http_status INT, idempotency_key UUID, created_at.
  - UNIQUE on (payment_id, kind, idempotency_key) for replay protection.
- `refunds`: id, payment_id FK, order_id FK, amount DECIMAL(12,2),
  reason VARCHAR(255), status (`REQUESTED | APPROVED | FAILED`),
  pg_refund_key VARCHAR(255) NULLABLE, requested_at, approved_at NULLABLE.

## Acceptance Criteria

- [ ] V8 applied; integration test inserts a full lifecycle (request →
      webhook → refund) chain.
- [ ] Replay: inserting a second `payment_transactions` row with the
      same `(payment_id, kind, idempotency_key)` violates the UNIQUE
      constraint.

## Hints

- Store the entire PG response body as JSONB so post-mortem debugging
  doesn't require enabling extra logging.
- `payments.idempotency_key` is set at order creation (OLV-061) so the
  client can safely retry with the same key.
