## Why

The cart is the staging area for orders. Without it, users can only do
"바로 구매" (direct buy), and our checkout UX would be incomplete from MVP
day one (PRD §3.1).

## Scope

Schema (`V5__cart.sql`):

- `carts`: id, member_id BIGINT UNIQUE FK, created_at, updated_at.
- `cart_items`: id, cart_id FK, product_option_id FK, quantity INT,
  created_at, updated_at. UNIQUE (cart_id, product_option_id).

Service + endpoints:

- `POST /api/cart/items` — `{productOptionId, quantity}`. If the option
  is already in the cart, increment. Reject if option is `STOPPED|HIDDEN`
  or `available_quantity < quantity` (return 409 with the available count).
- `GET /api/cart` — return items with: option name, current sale_price,
  on_sale flag, available_quantity, line subtotal. Mark items whose
  product is no longer ON_SALE so the UI can prompt removal.
- `PATCH /api/cart/items/{cartItemId}` — `{quantity}`. Re-validate stock.
- `DELETE /api/cart/items/{cartItemId}`.
- `POST /api/cart/merge` — called by the login flow when a session-bound
  anonymous cart should fold into the member cart.

Anonymous cart:

- Stored in Redis under `cart:anon:{sessionId}`, value is the JSON of items.
- 30-day TTL.
- Merge: union by product_option_id, sum quantities, cap at
  `available_quantity`.

## Acceptance Criteria

- [x] Round-trip integration test for add/list/update/remove.
- [x] Adding the same option twice increments instead of duplicating.
- [x] Add with quantity > stock → 409 + the available count in the body.
- [x] Anonymous → member merge: anonymous cart had option A:2, member

## Domain Brief

**무엇**: 장바구니 — 사용자 구매 의도를 저장하는 staging 영역 (비회원 Redis + 회원 DB)

**왜**: "바로 구매"만으로는 MVP UX 불완전. 장바구니는 전환율 핵심 경로이며, 로그인 시 익명→회원 카트 병합이 필요.

**As-Is → To-Be**:
- As-Is: 장바구니 없음. 사용자는 바로 구매만 가능.
- To-Be: 비회원(Redis 30일 TTL) + 회원(DB) 장바구니, 로그인 시 자동 병합, 최신 가격/재고 조회 API.

**핵심 제약** (llm-wiki/40-cart-domain.md + PRD §6.4, §8.2):
1. 카트는 "convenience storage" — 주문 시 재고/가격을 source domain에서 재검증.
2. 회원 카트: `carts`(member_id UNIQUE) + `cart_items`(cart_id, product_option_id UNIQUE).
3. 익명 카트: Redis `cart:anon:{sessionId}`, JSON, 30일 TTL.
4. Merge 로직: product_option_id별 union, 수량 합산, available_quantity로 cap.
5. API: POST /items (증분), GET / (최신 가격/상태), PATCH /items/{id}, DELETE /items/{id}, POST /merge.

**관련 스키마** (V2__member.sql, V3__product.sql, V4__inventory.sql):
- `members.id` (FK), `product_options.id/status`, `inventories.available_quantity`
- ProductOption.status: ON_SALE, SOLD_OUT, STOPPED, HIDDEN (추가 불가 상태)

## Recommendation

**옵션 1 채택**: 단순 CRUD + Merge

**이유**:
1. MVP 범위에 적합 — 확장 가능한 기본 구조
2. 기존 코드 패턴과 일치 (Member, Product, Inventory가 단순 entity+repository+service 패턴)
3. Merge 로직이 CartService에 위치하면 테스트가 용이 (Redis mock 가능)

**첫 번째 실패 테스트** (TDD 시작점):
```java
@Test
void add_sameOptionTwice_incrementsQuantity_notDuplicate() {
    // Given: option 1, quantity 2
    mockMvc.perform(post("/api/cart/items")
        .contentType(MediaType.APPLICATION_JSON)
        .content("{\"productOptionId\":1,\"quantity\":2}"))
        .andExpect(status().isOk());

    // When: add same option again
    mockMvc.perform(post("/api/cart/items")
        .contentType(MediaType.APPLICATION_JSON)
        .content("{\"productOptionId\":1,\"quantity\":3}"))
        .andExpect(status().isOk());

    // Then: quantity = 5, single item
    mockMvc.perform(get("/api/cart"))
        .andExpect(jsonPath("$.data.length()").value(1))
        .andExpect(jsonPath("$.data[0].quantity").value(5));
}
```

## As-Is → To-Be Report

### As-Is
- 장바구니 기능 없음: 사용자는 "바로 구매"만 가능
- 상품을 장바구니에 담아두지 못하고, 바로 주문해야 함
- 비회원과 회원 간 구매 경험 차이 없음

### To-Be
- **회원 장바구니 API**: POST/GET/PATCH/DELETE `/api/cart/items*`
  - 동일 옵션 추가 시 수량 증분 (중복 방지)
  - 재고 초과 시 409 Conflict + available count 응답
  - 최신 가격/상태 실시간 조회
- **익명 장바구니 API**: POST/GET/PATCH/DELETE `/api/cart/anonymous/items*`
  - Redis 30일 TTL, `cart:anon:{sessionId}` 키
  - 세션 ID 기반 (X-Session-ID 헤더 또는 JSESSIONID 쿠키)
- **병합 API**: POST `/api/cart/merge`
  - 익명 카트 → 회원 카트 자동 병합
  - 동일 옵션 수량 합산, 재고로 cap
- **Integration Test**: 7개 테스트 모두 통과

### Reasoning
- **단순 CRUD + Merge 패턴 채택**: 기존 Member/Product/Inventory 도메인과 일관성 유지
- **Redis 익명 카트**: 30일 TTL로 데이터 무한 정리 방지, 로그인 시 DB로 이동
- **최신 가격/상태 조회**: CartService에서 Product/Inventory를 조인하여 실시간 데이터 반환
- **JWT 인증 수정**: MockMvc 테스트에서 `@AuthenticationPrincipal` 지원을 위해 `UsernamePasswordAuthenticationToken`으로 `AuthenticatedUser`를 principal로 직접 설정
