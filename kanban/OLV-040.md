## Why

The cart is the staging area for orders. Without it, users can only do
"바로 구매" (direct buy), and our checkout UX would be incomplete from MVP
day one (PRD §3.1).

## Scope

Schema (`V5__cart.sql`):

- `carts`: id, member_id BIGINT UNIQUE FK, created_at, updated_at.
- `cart_items`: id, cart_id FK, product_option_id FK, quantity INT,
  created_at, updated_at. UNIQUE (cart_id, product_option_id).

Service + endpoints:

- `POST /api/cart/items` — `{productOptionId, quantity}`. If the option
  is already in the cart, increment. Reject if option is `STOPPED|HIDDEN`
  or `available_quantity < quantity` (return 409 with the available count).
- `GET /api/cart` — return items with: option name, current sale_price,
  on_sale flag, available_quantity, line subtotal. Mark items whose
  product is no longer ON_SALE so the UI can prompt removal.
- `PATCH /api/cart/items/{cartItemId}` — `{quantity}`. Re-validate stock.
- `DELETE /api/cart/items/{cartItemId}`.
- `POST /api/cart/merge` — called by the login flow when a session-bound
  anonymous cart should fold into the member cart.

Anonymous cart:

- Stored in Redis under `cart:anon:{sessionId}`, value is the JSON of items.
- 30-day TTL.
- Merge: union by product_option_id, sum quantities, cap at
  `available_quantity`.

## Acceptance Criteria

- [ ] Round-trip integration test for add/list/update/remove.
- [ ] Adding the same option twice increments instead of duplicating.
- [ ] Add with quantity > stock → 409 + the available count in the body.
- [ ] Anonymous → member merge: anonymous cart had option A:2, member
      cart had A:3 + B:1 → resulting cart has A:5 (capped at stock) +
      B:1.
- [ ] Cart items reflect the latest product price/status, not what was
      stored when added.
