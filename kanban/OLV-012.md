## Why

The order checkout flow needs `deliveryAddressId`. The user must be able to
manage their address book before they can pick one at checkout.

## Scope

Endpoints under `/api/me`:

- `GET /api/me` — current member profile (id, email, name, phone, grade).
- `PATCH /api/me` — update name, phone (no email change here — that's a
  separate verification flow, out of scope).
- `GET /api/me/addresses` — list addresses, default first.
- `POST /api/me/addresses` — add address; if `isDefault=true`, demote any
  prior default.
- `PATCH /api/me/addresses/{id}` — edit an owned address.
- `DELETE /api/me/addresses/{id}` — remove an owned address (block delete
  if it's the only address and any unshipped order references it).

## Acceptance Criteria

- [ ] All endpoints are 401 without auth, 403 if a member tries to access
      another member's address by ID.
- [ ] Setting a new default demotes the previous default in the same
      transaction.
- [ ] PII (phone, address) is masked in admin list views (covered later in
      OLV-063 admin path) but exposed in full to the owning member.
- [ ] Integration tests cover the ownership boundary (member A cannot
      read/edit member B's address).

## Domain Brief

**무엇**: 마이페이지 회원 프로필 조회/수정 + 배송지 관리 CRUD 6개 엔드포인트 구현
**왜**: 주문 checkout 시 `deliveryAddressId` 선택이 필요하며, 사용자가 자신의 배송지를 관리할 수 있어야 함
**As-Is → To-Be**:
- As-Is: `/api/me`가 단순 echo (memberId, email, name, role)만 반환; 배송지 관리 불가
- To-Be: 완전한 프로필 (phone, grade 포함) 조회/수정 + 배송지 추가/목록/수정/삭제; 소유권 검증으로 타 회원 접근 차단

### 핵심 발견 (llm-wiki/10-member-domain.md, V2__member.sql)

1. **member_addresses 테이블 이미 존재** — OLV-010에서 생성 완료 (`recipient_name`, `phone`, `zipcode`, `address_main`, `address_detail`, `is_default`)
2. **Partial Unique Index 보장** — `uniq_member_addresses_default_per_member`가 회원당 default 배송지 1건을 DB가 강제
3. **SecurityConfig 자동 401** — `/api/**` 패턴이 USER 역할 요구하므로 별도 매처 불필요
4. **소유권 검증은 애플리케이션 책임** — DB `member_id` 외래키가 있지만, URL 경로의 `{id}`가 접속자의 것인지 서비스 계층에서 명시적 검증 필요
5. **기존 MemberProfileController 확장** — OLV-011에서 placeholder로 도입; 본 티켓에서 실제 비즈니스 로직으로 교체

### 인바리언트

- `is_default=true`인 새 배송지 추가 시 기존 default 배송지는 **같은 트랜잭션**에서 `false`로 변경 (race 방지)
- DELETE는 **유일 배송지** + **미발송 주문 참조** 시 차단 (PRD §6.1)
- PII는 소유 회원에게만 전체 노출; admin 마스킹은 OLV-063으로 연기

## Plan Candidates

### Option A: 단일 컨트롤러 + 서비스 분리

```
MemberProfileController (프로필 + 배송지 모두 담당)
  ├─ GET /api/me          → MemberService.getProfile(id)
  ├─ PATCH /api/me        → MemberService.updateProfile(id, req)
  ├─ GET /api/me/addresses→ AddressService.listByMember(id)
  ├─ POST /api/me/addresses → AddressService.add(memberId, req)
  ├─ PATCH /api/me/addresses/{id} → AddressService.update(id, memberId, req)
  └─ DELETE /api/me/addresses/{id} → AddressService.delete(id, memberId)
```

- 장점: 관심사 분리; 각 컨트롤러가 단일 책임
- 단점: 경로가 `/api/me`와 `/api/me/addresses`로 나뉨 (자연스러운 분리)

## Recommendation

**선택**: Option B (컨트롤러 분리) + Option C (DTO + Repository 패턴)

**이유**:
1. **관심사 분리** — 프로필(회원 본인 정보)과 주소(배송지 목록)는 명확히 다른 도메인; 향후 배송지 관련 로직(우편번호 검색, 배송 가능 여부 확인 등)이 추가될 때 독립된 컨트롤러가 유리
2. **기존 MemberProfileController 최소 변경** — OLV-011 이미 `/api/me` GET를 구현; PATCH만 추가하면 됨
3. **Repository 패턴 일관성** — 다른 도메인(Product, Cart 등)이 Entity + Repository + Service + Controller 구조를 따르므로 동일 패턴 유지

## Learnings

1. **도메인 엔티티에 비즈니스 메서드 추가** — `Member.updateProfile(name, phone)`을 추가하여 컨트롤러에서 리플렉션 없이 상태 수정 가능. 이는 캡슐화를 강화하고 테스트 용이성을 높임
2. **@Transactional 서비스 메서드의 중요성** — 기본 배송지 변경 시 `clearDefaultByMemberId` → `save` 순서가 한 트랜잭션 내에서 실행되어야 원자성 보장. `@Transactional` 없이는 두 쿼리 사이의 경쟁 상태(Race Condition) 발생 가능
3. **Partial Unique Index 활용** — `WHERE is_default = TRUE` 조건부 유니크 인덱스로 "회원당 기본 배송지 1개"를 DB 레벨에서 강제. 애플리케이션 로직으로 방어하는 것보다 훨씬 안전함

## As-Is -> To-Be Report

### As-Is
- `/api/me` GET 요청 시 `memberId`, `email`, `name`, `role`만 echo 반환 (OLV-011 placeholder)
- 배송지 관리 불가 — `member_addresses` 테이블만 존재, API 없음
- 회원 본인 정보 수정(name, phone) 불가

### To-Be
- `GET /api/me` — `phone`, `grade` 포함된 완전한 프로필 반환
- `PATCH /api/me` — name, phone 수정 가능 (phone null 시 기존 값 유지)
- `GET /api/me/addresses` — 배송지 목록, 기본 배송지 우선 정렬
- `POST /api/me/addresses` — 배송지 추가; `isDefault=true` 시 기존 default 자동 해제
- `PATCH /api/me/addresses/{id}` — 소유한 배송지만 수정
- `DELETE /api/me/addresses/{id}` — 소유한 배송지만 삭제 (유일 배송지 보호)

### Reasoning
- **컨트롤러 분리** — 프로필과 배송지는 명확히 다른 도메인. 향후 배송지 관련 로직(우편번호 검색, 배송 가능 여부) 추가 시 독립된 컨트롤러가 유리
- **Repository 패턴 일관성** — 다른 도메인(Product, Cart)과 동일한 Entity + Repository + Service + Controller 구조 유지
- **소유권 검증 명시화** — DB FK만으로는 부족, URL 경로 `{id}`가 접속자의 것인지 서비스 계층에서 명시적 검증
