## Why

Every product, search, cart, and order ticket downstream queries this
schema. Land it once, properly indexed, before any product API ticket.

## Scope

Create `V3__product.sql` matching PRD §7.2-7.3:

- `products`: id, brand_id NULLABLE FK, name, description TEXT, status
  (`DRAFT|ON_SALE|SOLD_OUT|STOPPED|HIDDEN`), base_price DECIMAL(12,2),
  sale_price DECIMAL(12,2) NULLABLE, created_at, updated_at.
- `product_options`: id, product_id FK, option_name (e.g., "1호 색상"),
  option_price DECIMAL(12,2), status, created_at, updated_at.
- `product_images`: id, product_id FK, url, sort_order, is_thumbnail BOOLEAN.
- `brands`: id, name UNIQUE, slug, logo_url NULLABLE, status.
- `categories`: id, parent_id NULLABLE FK self, name, slug, sort_order, depth.
- `product_category_mapping`: product_id FK, category_id FK, PRIMARY KEY
  (product_id, category_id).

Indexes:

- `products(status, brand_id)` for category-list queries.
- `products(name text_pattern_ops)` for the Phase-1 LIKE search.
- `product_options(product_id)`.
- `product_images(product_id, sort_order)`.

Seed: insert one demo brand, three top-level categories
(스킨케어, 메이크업, 헤어/바디), and one demo product with two options for
manual smoke testing.

## Acceptance Criteria

- [ ] `./gradlew flywayInfo` shows V3 applied.
- [ ] Repository test inserts a product with two options + three category
      mappings and reads them back.
- [ ] `EXPLAIN` on `SELECT * FROM products WHERE status='ON_SALE' AND name
      LIKE '선크림%'` uses the `text_pattern_ops` index.

## Hints

- `option_name` keeps it simple — no need for option-type/option-value
  normalization at MVP. PRD §6.2 calls option as a single string.
- Use Postgres `CITEXT` for `brands.name` if case-insensitive uniqueness
  matters later; not required at MVP.

## Domain Brief

**무엇**: 상품 도메인의 핵심 테이블 6개(products, product_options, product_images, brands, categories, product_category_mapping)과 인덱스를 생성하고, 시드 데이터를 위한 Flyway V3 마이그레이션을 작성한다.
**왜**: 검색/장바구니/주문 티켓(OLV-022/OLV-040/OLV-030)이 모두 이 스키마를 조회하므로, Product API 티켓보다 먼저 제대로 인덱스된 기반을 깔아두지 않으면 후속 티켓마다 schema 수정이 번갈아 일어남.
**As-Is → To-Be**:
- As-Is: V2__member.sql까지만 적용된 상태 — products 브랜드/카테고리 테이블 0개, Repository 테스트 불가.
- To-Be: V3 적용 후 6개 테이블 + 4개 인덱스 + 시드 데이터 존재, ProductSchemaIntegrationTest가 INSERT→SELECT 왕복 검증.

- 인바리언트(`llm-wiki/20-product-domain.md:8-30`): **Product와 Option은 분리 테이블**, status는 5가지 enum(`DRAFT|ON_SALE|SOLD_OUT|STOPPED|HIDDEN`), 돈은 `DECIMAL(12,2)` + Java `BigDecimal`, 이미지는 URL만 보관.
- V2__member.sql 패턴: `set_updated_at()` 트리거 재사용, FK 제약에 `ON DELETE CASCADE`는 Category 자식 row 보존을 위해 일부 생략.
- 위험 R1: `brands.name` UNIQUE 인덱스 한글 이름 삽입 시 대소문자 구분 — MVP는 VARCHAR + UNIQUE로 충분(CITEXT는 훗날).
- 위험 R2: `product_category_mapping`(product_id, category_id) composite PK는 한 제품이 같은 카테고리를 중복 맵핑하는 것을 막음 — INSERT 시 중복 체크 필요.

## Plan Candidates

- **A. V3__product.sql 단일 마이그레이션 (티켓 Scope 그대로)**: 6개 테이블 + 4개 인덱스 + 시드 데이터를 V3 한 파일에 담아 `flywayInfo` V3 Success로 끝. `ProductSchemaIntegrationTest`(@DataJpaTest + @ServiceConnection)가 INSERT/SELECT 왕복 + 인덱스 사용(`EXPLAIN`) 검증. **권장**.
- **B. V3_products + V4_brands + V5_categories 분리**: 도메인별로 쪼개면 각각 독립적으로 revert 가능. 단점: 티켓의 "한 번에 깔아둔다"는 목적(§Why)과 위배 — 후속 티켓이 3개 마이그레이션을 모두 기다려야 함.
- **C. JPA Entity @Entity 함께 생성**: 마이그레이션 + Entity 한 방에 완성. 단점: (1) 본 티켓 AC는 "Repository test"이지 Entity API 설계가 아님, (2) OLV-022 Product API 티켓이 상세 설계를 담당 — 일찍 만든 Entity는 후속 티켓에서 리팩토릴 유발.

## Recommendation

**Option A 채택.** 티켓 명세가 V3 단일 마이그레이션으로 명확하고, 후속 티켓(OLV-022 Product API, OLV-022 Search, OLV-040 Cart, OLV-030 Order)가 모두 이 schema를 전제로 시작하므로 분리(B)는 과공학. Entity는 API 설계 티켓(C)로 연기 — Schema-only IT가 원하는 AC 범위에 정확히 부합.
**첫 실패 테스트**: `com.olive.commerce.product.ProductSchemaIntegrationTest` — `@DataJpaTest` + `@ServiceConnection`. `em.createNativeQuery("INSERT INTO products ...")`로 demo product 1건 + options 2건 + category_mappings 3건 삽입 후 SELECT으로 왕복, `EXPLAIN`으로 `text_pattern_ops` 인덱스 사용 확인. V3 없으면 RED, 있으면 GREEN.
