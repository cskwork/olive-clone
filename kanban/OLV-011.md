## Why

This is the entry point for every user-facing flow. Until this lands,
nothing under `/api/cart`, `/api/orders`, `/api/reviews` can be exercised
end-to-end.

## Scope

Endpoints under `/api/auth`:

- `POST /api/auth/signup` — body `{email, password, name, phone}`. Returns
  `{memberId}` on 201; 409 if email duplicate; 400 on validation failure
  (email format, password ≥ 8 chars, phone format).
- `POST /api/auth/login` — body `{email, password}`. Returns
  `{accessToken, refreshToken, expiresInSec}`. Writes a `member_login_histories`
  row with `success=true`. Wrong password writes `success=false` and returns
  401. Five consecutive failures within 10 min lock the account for 15 min.
- `POST /api/auth/refresh` — body `{refreshToken}`. Returns a fresh access
  token and a rotated refresh token (revoke the old `member_refresh_tokens`
  row, insert the new one).
- `POST /api/auth/logout` — revokes all of the caller's refresh tokens.

## Acceptance Criteria

- [ ] Happy path: signup → login → use access token on a placeholder
      `/api/me` endpoint that echoes `memberId`.
- [ ] Login failure writes a row with `success=false` and the failure
      reason `BAD_CREDENTIALS` / `ACCOUNT_LOCKED`.
- [ ] Refresh rotation: old refresh token is rejected on second use
      (replay protection).
- [ ] Passwords are stored as bcrypt (cost ≥ 12); never returned in any
      response.
- [ ] Integration test (Testcontainers) covers the four endpoints above.

## Hints

- Use Spring Security's `BCryptPasswordEncoder(12)`.
- Account lock state can live in Redis with a TTL — keeps the `members`
  table read-light. Key: `lock:member:{email}`, value: failed-attempt
  count, TTL 10 min.
- The email validation regex should be permissive (`^[^@]+@[^@]+\.[^@]+$`)
  — strict RFC compliance is not worth the cost.
- Add audit log calls (`auditLogger.log("LOGIN_SUCCESS"|"LOGIN_FAILURE")`)
  per `llm-wiki/98-security.md`.

## Domain Brief

**무엇**: 회원가입·로그인·refresh·logout 4개 공개 API + `/api/me` 검증용 엔드포인트. 비밀번호는 bcrypt(12), refresh는 SHA-256 hex로만 DB 저장, 5회 실패 시 Redis 락(15분).
**왜**: 이 티켓이 풀려야 cart/orders/reviews 가 endtoend 검증 가능 — 모든 사용자 흐름의 진입점.
**As-Is → To-Be**:
- As-Is: `AuthPlaceholderController` 가 `POST /api/auth/login` 만 placeholder JSON 반환. 회원/refresh/logout/me 엔드포인트 부재.
- To-Be: signup/login/refresh/logout/me 5개 엔드포인트 동작, login 실패 → `member_login_histories` 기록, refresh 회전 + replay 차단, 5회 실패 락 동작.

이미 깔린 자산(재구현 금지): `JwtTokenProvider#issueAccess/issueRefresh` (RS256), `JwtAuthenticationConverter` (access→`AuthenticatedUser`), `members`/`member_refresh_tokens`/`member_login_histories` 5테이블, `BusinessException`+`GlobalExceptionHandler`, `AuditLogger` 빈, `RedisAutoConfiguration` `StringRedisTemplate`. 미존재: `BCryptPasswordEncoder` 빈, refresh 토큰 검증(`parseRefresh`), 잠금 정책 정책 컴포넌트, ErrorCode 4종 (EMAIL_ALREADY_USED·BAD_CREDENTIALS·ACCOUNT_LOCKED·INVALID_REFRESH_TOKEN).

## Plan Candidates

**A. 단일 `AuthService`에 4 흐름 압축** — `MemberRepository`+`RefreshTokenRepository`+`LoginAttemptGuard` 주입. 컨트롤러는 4 메서드 위임. 짧고 응집도 높음. 단점: 흐름 추가 시 service가 부풀음 (지금 4 → 향후 social-login/2FA 시 분리 필요).

**B. 흐름별 service 분리 (`SignupService`/`LoginService`/`RefreshService`/`LogoutService`)** — 각각 SRP. 단점: 4개 서비스가 같은 repo·encoder 의존. 현 시점 분리는 over-engineering — common rules의 "premature abstraction" 위반.

**C. JPA 엔티티 vs JDBC** — V2 마이그레이션은 이미 native query / DataJpaTest 패턴(MemberSchemaIT)으로 검증됨. JPA 엔티티 도입 시 `Member`/`MemberRefreshToken`/`MemberLoginHistory` 3개. 엔티티 없이 `JdbcTemplate`만 쓰면 SQL 노출 ↑. 본 모듈러 모놀리스는 JPA 표준 채택 — 엔티티 도입.

## Recommendation

**선택: A (단일 AuthService) + JPA 엔티티 + Redis 잠금 + refresh row lock 회전**. 흐름이 4개로 작고 모두 동일 의존을 공유한다 — 분리는 5번째 흐름이 들어올 때 리팩터.

핵심 변경: `MemberRole`이 `member` 패키지에 이미 있으므로 `member` 패키지에 `Member`/`MemberRefreshToken`/`MemberLoginHistory` 엔티티 + 각 repository, `auth` 패키지에 `AuthService`/`AuthController`/`LoginAttemptGuard`(Redis), `common.security`에 `PasswordEncoderConfig` (bcrypt 12), `JwtTokenProvider#parseRefresh` 추가.

**첫 실패 테스트**: `AuthApiIT#signupThenLoginThenMe_returnsMemberId` — `POST /api/auth/signup` → 201 `{memberId}`, 같은 자격으로 `POST /api/auth/login` → 200 + accessToken, 그 토큰으로 `GET /api/me` → 200 `{memberId, email, name}`. 현재는 컨트롤러 미존재 → 404.

## Learnings

**무엇**: Explore brief 와 실제 구현 사이의 갭. 4건의 비-자명한 invariant 가 구현 중에 드러났다.
**왜**: 다음 도메인 티켓이 같은 함정을 반복하지 않게.

- **login() 트랜잭션 경계**: 처음에는 `@Transactional`로 감쌌으나 BusinessException → rollback → history INSERT 사라짐. 감사 트레일이 살아남으려면 인증 실패는 트랜잭션 밖에서 commit되어야 한다.
- **dummy bcrypt hash 형식**: timing-attack 흉내는 진짜 bcrypt 형식(60자, `$2a$12$...`)이어야 silent하게 작동. 가짜 string 은 매 요청마다 Spring Security warn 노이즈.
- **`@SpringBootTest` 좁은-context IT의 회귀**: 신규 `@Service`/`@Component` 가 component scan 으로 잡히면 기존 좁은-IT 의 컨텍스트 부팅이 깨진다. `@MockBean` 으로 의존을 끊는 것이 표준. 5번째 도메인쯤 base-class 추출 검토.
- **BootstrapTest sibling Redis**: actuator/health 가 RedisHealthIndicator까지 점검 — PG 만 있으면 503. 본 티켓에서 sibling Redis 컨테이너 추가로 보강 (pre-existing 결함).

## As-Is -> To-Be Report

### As-Is

- `auth/AuthPlaceholderController` 만 존재. `POST /api/auth/login` 호출 시 `{"success":true,"data":{"status":"placeholder"}}` 만 반환.
- 회원가입·실제 로그인·refresh 회전·logout·`/api/me` 모두 부재.
- 실패한 로그인 시도가 `member_login_histories` 에 기록되지 않음 — 감사 trail 공백.
- `members` 테이블은 OLV-010 의 V2 마이그레이션으로 비어있는 상태 (5 테이블 + BRONZE/SILVER/GOLD seed 만 있음).
- `BCryptPasswordEncoder` 빈 미존재 → 비밀번호 해싱 가능 X.
- `JwtTokenProvider` 는 access 토큰만 검증 (`parseAccess`) — refresh 검증 메서드 부재.

### To-Be

- 5 엔드포인트 동작:
  - `POST /api/auth/signup` 201 `{memberId}` / 409 `EMAIL_ALREADY_USED` / 400 `VALIDATION_FAILED`
  - `POST /api/auth/login` 200 `{accessToken, refreshToken, expiresInSec}` / 401 `BAD_CREDENTIALS` / 423 `ACCOUNT_LOCKED`
  - `POST /api/auth/refresh` 200 (회전된 새 토큰 한 쌍) / 401 `INVALID_REFRESH_TOKEN` (replay 차단)
  - `POST /api/auth/logout` 200 `{revokedTokens}` (모든 active refresh revoke)
  - `GET /api/me` 200 `{memberId, email, name, role}` (인증 필요)
- 모든 로그인 시도 (성공/실패) 가 `member_login_histories` 에 row 작성, `failure_reason` ∈ {BAD_CREDENTIALS, ACCOUNT_LOCKED}.
- 5회 연속 실패 시 Redis (`auth:lock:{email}` TTL 15분) 자동 잠금 — 정확한 비밀번호도 락 활성 동안 거절.
- Refresh 토큰 회전: 동일 토큰 두 번째 사용 시 `revoked_at IS NOT NULL` 보고 401.
- 비밀번호: `BCryptPasswordEncoder(12)` 로 해싱, DB `password_hash` 는 `^\$2[ab]\$12\$.*` 형식, 응답 본문 어디에도 평문 echo X.
- `LogbackAuditLogger.log("LOGIN_SUCCESS"/"LOGIN_FAILURE", ...)` 로 모든 시도가 일자 롤링 JSON 으로 감사.

### Reasoning

- **단일 `AuthService` (Plan A)** 채택 — 4 흐름이 같은 의존을 공유, SRP 분리는 social-login/2FA 가 들어올 때까지 유보 (common-rules "premature abstraction 회피"). Plan B (4 service) 는 의존 중복.
- **JPA 엔티티 + Spring Data Repository** — V2 마이그레이션 위에 `Member`/`MemberRefreshToken`/`MemberLoginHistory`/`MemberGrade` 엔티티. `JdbcTemplate` 직접 사용은 SQL 노출이 커지고 모듈러 모놀리스의 표준 (Spring Data JPA) 에서 벗어남.
- **login() 의 `@Transactional` 의도적 누락** — BusinessException 시 history INSERT rollback 방지. 각 repo 호출은 Spring Data 의 단위 `@Transactional` 로 독립 커밋.
- **refresh 회전은 `@Lock(PESSIMISTIC_WRITE)`** — DB row lock 으로 동시 회전 race 직렬화. application-side 락 불필요.
- **Redis 잠금 정책** — 영구 컬럼 `members.status` 는 운영 SUSPENDED/DELETED 만, 일시 잠금은 Redis TTL 로 분리 (10-member-domain.md 가이드).
- **HTTP 423 Locked** — RFC 4918 의미적으로 정확. 클라이언트가 401 fallback 처리만 한다면 스펙 변경 가능.
- **dummy bcrypt 해시 캐싱** — 진짜 bcrypt 형식이어야 silent timing attack 흉내 + Spring Security warn 노이즈 회피. 생성자에서 한 번 encode 후 인스턴스 필드.
- **트레이드오프 / 후속 deferred**:
  - Email validation 은 permissive (`^[^@\s]+@[^@\s]+\.[^@\s]+$`) — 강제 RFC 준수보다 UX 우선 (티켓 §Hints).
  - timing attack 방어가 known/unknown email 사이에 완벽하지 않음 — 후속 강화 티켓.
  - `AuthPlaceholderController.java` 가 빈 파일로 남음 (rm 권한 미허용) — 후속 cleanup 티켓에서 삭제.
  - JPA 엔티티의 `created_at`/`updated_at` 은 `insertable=false, updatable=false` — DB DEFAULT/trigger 가 책임. JPA `@PrePersist`/`@PreUpdate` 미사용으로 native query 우회 일관성 보장.
