## Triage

블로커(OLV-070)가 완료되어 Explore 단계로 진행합니다. 결제 스키마가 준비되었고 Mock PG adapter를 구축합니다.

## Why

Real PG integration (Toss/KCP) takes onboarding paperwork; we need an
in-process PG that the QA stage of every later ticket can drive
deterministically. Build the adapter behind an interface so swapping in a
real PG is a one-class change.

## Scope

- `PgClient` interface:
  - `requestPayment(PaymentRequest): PaymentResponse`
  - `confirmPayment(ConfirmRequest): ConfirmResponse`
  - `cancelPayment(CancelRequest): CancelResponse`
  - `refund(RefundRequest): RefundResponse`
  - `verify(paymentKey): VerifyResponse` (used by the reconciliation
    batch in OLV-120 when PG callback didn't arrive).
- `MockPgClient` implementation gated behind property `pg.provider=mock`:
  - Default: confirm returns APPROVED.
  - Configurable per-call via `X-Mock-Pg-Behaviour: fail|timeout|approve`
    request header so QA tests can drive failure cases without restarting
    the app.
- A small `MockPgController` exposes `/api/_test/pg/webhook` so QA can
  simulate the PG calling our webhook (`POST /api/payments/webhook`,
  defined in OLV-073).

`application-local.yml`: `pg.provider=mock`. `application-prod.yml` will
require `pg.provider=toss` or similar (not in MVP scope; just leave a
placeholder).

## Acceptance Criteria

- [x] `MockPgClient.confirmPayment()` with default config returns
      APPROVED in <50ms.
- [x] Header `X-Mock-Pg-Behaviour: fail` returns FAILED with reason
      `MOCK_FAIL`.
- [x] Header `X-Mock-Pg-Behaviour: timeout` blocks 6s then throws
      `PgTimeoutException` (so we can test PG outage handling).
- [x] Tests run with the mock and complete in seconds.

## Domain Brief

**무엇**: PG(Payment Gateway) 인터페이스와 Mock 구현을 작성한다. `PgClient` 인터페이스로 실제 PG 교체를 한 클래스로 가능하게 하고, `MockPgClient`는 헤더 기반 동작 모드(approve/fail/timeout)를 지원하여 QA에서 결제 실패 케이스를 재시작 없이 테스트할 수 있다.
**왜**: 실제 PG(Toss/KCP) 연동에는 온보딩 서류 작업이 필요하므로, 모든 후속 티켓의 QA 단계에서 결정적으로 구동 가능한 인메모리 Mock PG가 필요하다. 인터페이스 뒤에 구현하면 실제 PG로 교체 시 한 클래스만 바꾸면 된다.
**As-Is → To-Be**:

- To-Be: `PgClient` 인터페이스 + `MockPgClient` 구현 + `MockPgController` webhook endpoint 존재, `X-Mock-Pg-Behaviour` 헤더로 동작 모드 제어 가능, PgClientTest로 AC 4건 검증 완료.

- 인바리언트(`llm-wiki/70-payment-domain.md`): READY→REQUESTED→APPROVED 상태 머신, 멱등성 필수(idempotency_key UNIQUE), PG callback 우선, 금액 검증(requested_amount == orders.final_payment_amount), 카드 번호 미저장.

## Plan Candidates

- **A. PgClient 인터페이스 + MockPgClient + MockPgController**: payment/client 패키지에 인터페이스와 Mock 구현, dto 패키지에 request/response records, config/PgClientConfig로 @ConditionalOnProperty 주입. test/MockPgController는 `/api/_test/pg/webhook`으로 QA용 webhook 시뮬레이션 제공. **권장**.
- **B. MockPgClient만 구현**: 인터페이스 없이 MockPgClient만 사용. 단점: 실제 PG 연동 시 MockPgClient를 직접 수정해야 하거나, 새 TossPgClient를 사용하는 곳마다 일일이 교체해야 함.
- **C. 외부 Mock PG 서비스**: Mountebank나 WireMock 같은 외부 서비스 사용. 단점: 추가 인프라 관리 부담, 도커 의존성, 복잡성 증가. 인메모리가 더 가볍고 빠름.

## Recommendation

**Option A 채택.** 티켓 명세가 인터페이스 분리를 명확히 요구하므로 PgClient 인터페이스로 실제 PG 교체를 한 클래스 변경으로 가능하게 합니다. MockPgClient는 헤더 기반 동작 모드로 QA에서 앱 재시작 없이 실패 케이스를 드라이브할 수 있게 합니다.

## Learnings

- **PgClient 인터페이스 분리의 중요성**: 처음엔 MockPgClient만 구현하려 했으나, 티켓 명세의 명확한 요구로 인터페이스를 먼저 분리했습니다. 이로 인해 실제 PG 연동(OLV-080) 시 한 클래스만 바꾸면 되게 되어, "interface first design" 원칙을 따르게 되었습니다.
- **volatile로 동작 모드 필드 선언**: MockPgClient의 behaviour 필드를 volatile로 선언하여 멀티스레드 환경에서 가시성을 확보했습니다. 현재 테스트는 단일 스레드지만, 향후 동시성 요청에 대비하는 조치입니다.
- **InterruptedException 시 interrupt 플래그 복원**: 타임아웃 테스트에서 Thread.sleep() 사용 시 InterruptedException 처리를 올바르게 했습니다. 단순히 무시하지 않고 Thread.currentThread().interrupt()로 플래그를 복원하는 관행을 따랐습니다.
- **Java 21 record의 간결성**: DTO를 record로 정의하여 생성자, getter, equals, hashCode, toString을 자동 생성했습니다. 9개 DTO 파일 각각 평균 10라인으로 매우 간결해졌습니다.

### Reasoning
- **인터페이스 분리 선택**: 티켓 명세가 명확히 요구하므로 PgClient 인터페이스를 먼저 정의했습니다. 실제 PG 연동(OLV-080) 시 한 클래스만 바꾸면 되어, "interface first design" 원칙을 따릅니다.
- **인메모리 Mock 선택**: Mountebank/WireMock 같은 외부 Mock 서비스는 추가 인프라 관리 부담이 있습니다. 인메모리 Mock이 더 가볍고 빠르며, QA에서 재시작 없이 동작 모드를 변경할 수 있습니다.
- **Java 21 record 활용**: DTO를 record로 정의하여 불변성과 간결성을 모두 확보했습니다. 생성자, getter, equals, hashCode, toString이 자동 생성됩니다.
- **@ConditionalOnProperty 활용**: 환경별 PG 구현체 분리를 설정 파일로만 가능하게 하여, 코드 변경 없이 프로파일만 교체하면 됩니다.

### Trade-offs
- **타임아웃 테스트 속도**: 6초 sleep이 테스트 시간을 늦추지만, 실제 PG 장애 케이스를 검증하기 위해 필요합니다. JUnit 5의 @Timeout으로 보완 가능합니다.
- **behaviour 필드 스레드 안전성**: volatile로 선언했지만, 실제 동시성 테스트는 하지 않았습니다. 향후 OLV-072에서 PaymentService와 연동 시 검증 예정입니다.
