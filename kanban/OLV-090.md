## Why

PRD §6.10. Reviews drive product detail conversion; eligibility check ties
the domain to order_items.

## Scope

Schema (`V10__review.sql`):

- `reviews`: id, member_id FK, product_id FK, order_item_id FK UNIQUE,
  rating SMALLINT NOT NULL CHECK (rating BETWEEN 1 AND 5), title, body
  TEXT, status (`VISIBLE | HIDDEN`), created_at, updated_at.
- `review_images`: id, review_id FK, url, sort_order.
- `review_reports`: id, review_id FK, reporter_member_id FK, reason
  VARCHAR(255), status (`PENDING | RESOLVED | DISMISSED`), created_at.
- `product_review_summaries`: product_id PK, avg_rating DECIMAL(3,2),
  review_count INT, updated_at.

User endpoints:

- `POST /api/me/reviews` — `{orderItemId, rating, title, body, imageUrls}`.
  Eligibility: order_item belongs to caller's order with status
  ∈ `{DELIVERED}` (PRD §6.10) — block earlier statuses with 422.
- `GET /api/products/{productId}/reviews?sort=latest|helpful&page=&size=`
  — public, only VISIBLE reviews.
- `POST /api/me/reviews/{reviewId}/report` — flag inappropriate content.

Admin endpoints (require `CS_MANAGER` or higher):

- `GET /api/admin/review-reports?status=` — list pending reports.
- `POST /api/admin/reviews/{reviewId}/hide` — sets status HIDDEN.

Aggregate maintenance: subscribe to `ReviewCreatedEvent` and recompute
`product_review_summaries` (use a MERGE / UPSERT keyed by product_id).
Recompute also on hide.

## Acceptance Criteria

- [ ] User who hasn't received the order yet → 422 on review write.
- [ ] One review per order_item — second attempt returns 409.
- [ ] After 3 reviews land, `product_review_summaries.avg_rating` matches
      the manual recompute.
- [ ] HIDDEN reviews don't appear in `/api/products/{id}/reviews`.
