## Why

PRD §6.10. Reviews drive product detail conversion; eligibility check ties
the domain to order_items.

## Scope

Schema (`V10__review.sql`):

- `reviews`: id, member_id FK, product_id FK, order_item_id FK UNIQUE,
  rating SMALLINT NOT NULL CHECK (rating BETWEEN 1 AND 5), title, body
  TEXT, status (`VISIBLE | HIDDEN`), created_at, updated_at.
- `review_images`: id, review_id FK, url, sort_order.
- `review_reports`: id, review_id FK, reporter_member_id FK, reason
  VARCHAR(255), status (`PENDING | RESOLVED | DISMISSED`), created_at.
- `product_review_summaries`: product_id PK, avg_rating DECIMAL(3,2),
  review_count INT, updated_at.

User endpoints:

- `POST /api/me/reviews` — `{orderItemId, rating, title, body, imageUrls}`.
  Eligibility: order_item belongs to caller's order with status
  ∈ `{DELIVERED}` (PRD §6.10) — block earlier statuses with 422.
- `GET /api/products/{productId}/reviews?sort=latest|helpful&page=&size=`
  — public, only VISIBLE reviews.
- `POST /api/me/reviews/{reviewId}/report` — flag inappropriate content.

Admin endpoints (require `CS_MANAGER` or higher):

- `GET /api/admin/review-reports?status=` — list pending reports.
- `POST /api/admin/reviews/{reviewId}/hide` — sets status HIDDEN.

Aggregate maintenance: subscribe to `ReviewCreatedEvent` and recompute
`product_review_summaries` (use a MERGE / UPSERT keyed by product_id).
Recompute also on hide.

## Acceptance Criteria

- [x] User who hasn't received the order yet → 422 on review write.
- [x] One review per order_item — second attempt returns 409.
- [x] After 3 reviews land, `product_review_summaries.avg_rating` matches
      the manual recompute.
- [x] HIDDEN reviews don't appear in `/api/products/{id}/reviews`.

## Domain Brief

**무엇**: 리뷰 도메인 구현 - 리뷰 작성/조회/신고 API + 리뷰 요약 집계 테이블 유지보수

**왜**: 리뷰는 상품 상세 전환율을 높이는 핵심 요소이며, 배송 완료(order.status=DELIVERED) 고객만 작성 가능하여 주문 도메인과 연계됩니다. 집계 테이블로 핫 읽기 경로에서 실시간 계산을 방지합니다.

**As-Is → To-Be**:
- As-Is: reviews/review_images/review_reports/product_review_summaries 테이블 0개, 리뷰 작성/조회/신고 API 없음, 상품별 평점 계산 불가
- To-Be: V10__review.sql 적용 후 4개 테이블 + ReviewService + ReviewController + ReviewAdminController + ReviewCreatedEvent 구독자로 집계 유지보수

핵심 인바리언트:
- **자격 검증**: order_item이 속한 주문의 소유자이며 상태가 DELIVERED일 때만 리뷰 작성 가능
- **중복 방지**: order_item_id UNIQUE 제약으로 하나의 주문 상품당 하나의 리뷰만 작성
- **가시성**: HIDDEN 상태 리뷰는 목록 조회에서 제외 (관리자 숨김 기능)
- **집계 테이블**: ReviewCreatedEvent 발행 시 product_review_summaries를 MERGE/UPSERT로 갱신

위험 R1: 자격 검증 누락 시 배송 전 리뷰 작성 가능
위험 R2: order_item_id UNIQUE 없으면 중복 리뷰 작성
위험 R3: 집계 계산 오류 시 상품 상세에 잘못된 평점 표시

## Plan Candidates

### Option 1 - V10 단일 마이그레이션 + Spring Event 기반 집계 (추천)
- **장점**: 원자적 스키마 적용, Spring ApplicationEvent로 간단한 fan-out, 테스트에서 이벤트 리스너 비활성화 용이
- **단점**: 이벤트 리스너 실패 시 재시도 로직 필요 (별도 스케줄러 또는 outbox)
- **복잡도**: 중간
- **적용 사례**: DeliveryCompletedEvent 패턴과 동일

### Option 2 - 마이그레이션 분리 (V10_reviews + V11_images + V12_reports + V13_summaries)
- **장점**: 변경 단위 세분화
- **단점**: 원자성 보장 어려움, 후속 티켓이 4개 마이그레이션을 모두 기다려야 함
- **복잡도**: 높음
- **적용 사례**: 순차적 마이그레이션 필요 시

### Option 3 - 실시간 계산 (집계 테이블 없음)
- **장점**: 데이터 일관성 보장 (단일 진실 공급원)
- **단점**: 핫 읽기 경로에서 reviews 테이블 전체 스캔, 성능 저하
- **복잡도**: 낮음
- **적용 사례**: 트래픽 낮은 서비스

## Recommendation

**Option 1 채택.** V10__review.sql 단일 마이그레이션으로 4개 테이블을 한 번에 적용하고, Spring ApplicationEvent(`ReviewCreatedEvent`)를 발행하여 `ReviewAggregateSubscriber`가 `product_review_summaries`를 갱신합니다. DeliveryCompletedEvent 패턴을 따라 구현 일관성을 유지합니다.

**이유**:
1. PRD §6.10에서 요구하는 집계 테이블 패턴 준수 (핫 읽기 경로 최적화)
2. order_item_id UNIQUE 제약으로 DB 레벨 중복 방지
3. Spring Event 리스너는 `@TransactionalEventListener(phase=AFTER_COMMIT)`으로 트랜잭션 커밋 후 실행

**첫 번째 실패 테스트**:
```java
@Test
@DisplayName("배송 완료 전 주문 상품에 리뷰 작성 시 422 반환")
void writeReview_beforeDelivery_returns422() {
    // given
    Long orderItemId = 1L; // PAID 상태 주문
    ReviewDtos.WriteReviewRequest request = new ReviewDtos.WriteReviewRequest(
        orderItemId, 5, "좋아요", "구매 만족", List.of()
    );

    // when/then
    assertThatThrownBy(() -> reviewService.writeReview(memberId, request))
        .isInstanceOf(BusinessException.class)
        .satisfies(e -> assertThat(((BusinessException) e).getErrorCode())
            .isEqualTo(ErrorCode.REVIEW_ELIGIBLE_ORDER_REQUIRED));
}
```

## Learnings

1. **테스트 데이터 일관성**: 외래 키 제약 조건을 준수하기 위해 테스트에서 생성하는 FK 참조 데이터(ID)를 일치시켜야 합니다. `order_items.product_option_id=1`을 사용했지만 `product_options.id=999`만 생성해서 FK 위반이 발생했습니다.

2. **DataJpaTest 격리**: `@DataJpaTest`는 JPA 관련 컴포넌트만 로드하므로, AWS S3 같은 외부 의존성 문제에 영향받지 않습니다. 전체 애플리케이션 컨텍스트를 로드하는 `@SpringBootTest`와는 다릅니다.

3. **기존 문제 노출**: 새 도메인 추가로 전체 테스트 스위트를 실행할 때 기존 코드베이스의 설정 문제(AWS S3 region)가 노출되었습니다. 하위 호환성 테스트가 중요합니다.
