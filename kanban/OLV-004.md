## Why

Every API ticket from OLV-011 onward expects a consistent envelope, a
predictable error shape, and a request-id propagated to logs. Wiring this
once means downstream tickets do not re-invent it.

## Scope

In `com.olive.commerce.common`:

- `ApiResponse<T>` record with the envelope from PRD patterns
  (`success: boolean`, `data: T?`, `error: ErrorBody?`, `meta: PageMeta?`).
- `ErrorBody` (`code: String`, `message: String`, `path: String`,
  `traceId: String`).
- `ErrorCode` enum (HTTP-aligned: `MEMBER_NOT_FOUND`, `PRODUCT_SOLD_OUT`,
  `COUPON_INVALID`, `INSUFFICIENT_INVENTORY`, `PAYMENT_AMOUNT_MISMATCH`,
  `IDEMPOTENCY_CONFLICT`, etc. — start with the obvious ones, add more
  later).
- `BusinessException(ErrorCode, String?)` and a
  `@RestControllerAdvice GlobalExceptionHandler` that maps:
  - `BusinessException` → 400/404/409/422 by `ErrorCode.httpStatus`.
  - `MethodArgumentNotValidException` → 400 with field error list.
  - Anything else → 500 (no leaked stack trace in body).
- `RequestIdFilter` (`OncePerRequestFilter`) that reads/generates
  `X-Request-Id`, sets MDC, propagates back in response header.
- `AuditLogger` interface + Logback-backed JSON impl that writes to
  `log/audit-YYYY-MM-DD.log` for: login attempts, admin mutations, payment
  events, inventory changes, coupon use.

## Acceptance Criteria

- [ ] Throwing `BusinessException(MEMBER_NOT_FOUND, "id=42")` from a test
      controller returns HTTP 404 with body
      `{"success":false,"error":{"code":"MEMBER_NOT_FOUND","message":"id=42",...}}`.
- [ ] Logs contain `traceId=<uuid>` matching the response `X-Request-Id`.
- [ ] An audit log line appears in `log/audit-*.log` after a synthetic
      `auditLogger.log("LOGIN_SUCCESS", member=42)` call.
- [ ] Unit tests for `GlobalExceptionHandler` cover the three paths above.

## Hints

- Use `slf4j` `MDC` for request-id propagation; clear it in the filter's
  `finally` block to avoid leaking across thread reuse.
- For JSON audit logs use Logback `JsonLayout` (logstash-encoder) — the
  Prometheus stack ticket (OLV-130) will scrape this format.

## Domain Brief

**무엇**: API 응답 봉투(`ApiResponse`)·표준 에러 본문(`ErrorBody`/`ErrorCode`)·요청별 traceId(`RequestIdFilter`)·감사 로그(`AuditLogger`+JSON 파일)를 `com.olive.commerce.common`에 한 번 깔아둔다.
**왜**: OLV-011 이후 모든 도메인 컨트롤러가 동일한 envelope을 반환해야 하고, 장애 디버깅·OLV-130 옵저버빌리티·PRD §16.2 감사 의무가 traceId/JSON 감사 라인 위에서 굴러간다. 지금 못 박지 않으면 도메인 티켓마다 다른 모양의 에러가 새어 나간다.
**As-Is → To-Be**:
- As-Is: 컨트롤러 0개, 예외 매핑/필터/audit 0개. 부팅하면 Spring 기본 ErrorController가 `{"timestamp", "status", "error", "path"}` 같은 비-envelope JSON을 반환. logback 설정 부재.
- To-Be: 임의 컨트롤러가 `BusinessException(MEMBER_NOT_FOUND, "id=42")`을 던지면 HTTP 404 + `{"success":false,"error":{"code":"MEMBER_NOT_FOUND",...,"traceId":"<X-Request-Id>"}}`. 응답 헤더 `X-Request-Id`와 콘솔 로그의 `traceId=...`가 일치하고, `auditLogger.log("LOGIN_SUCCESS", member=42)` 호출 시 `log/audit-YYYY-MM-DD.log`에 단일라인 JSON이 추가된다.

- 인바리언트(`llm-wiki/98-security.md:22-26`): 요청 단위 (request_id, member_id, role, method, path, status, latency_ms) 감사 — traceId/MDC를 거기 그대로 흘려보낼 수 있도록 *지금* 골격을 맞춘다.
- 인바리언트(`llm-wiki/00-architecture-overview.md:13`): common 모듈은 횡단 관심사만 — 도메인 의존성 0(컨트롤러/서비스/엔티티 미작성), 상호 결합은 record + interface만.
- 의존성 추가 단 1개: `net.logstash.logback:logstash-logback-encoder`(기본 logback 위에 JSON 인코더만 얹음). Spring Boot 3.x 기본 jackson이 record 직렬화를 지원하므로 추가 모듈 불필요.
- 위험 R1(MDC 누수): try/finally로 `MDC.remove("traceId")`. R2(SecurityException 가로채기): `GlobalExceptionHandler.handleGeneric(Exception)`은 `AuthenticationException`/`AccessDeniedException`을 명시적으로 rethrow.

## Plan Candidates

- **A. 단일 `@RestControllerAdvice` + `OncePerRequestFilter` + Logback `logstash-encoder` (티켓 Hints 라인)**: record envelope, ErrorCode→httpStatus 매핑, MDC 기반 traceId, `logback-spring.xml`에 일자 롤링 audit appender. **권장.**
- **B. Spring `ErrorController` 직접 구현(`AbstractErrorController`)**: Spring MVC가 자체 에러 디스패처로 라우팅하는 방식. 장점: forward 기반이라 비-컨트롤러 예외(예: 정적 자원 404)도 잡음. 단점: `@RestControllerAdvice` 대비 보일러가 많고 record envelope 매핑이 더 멀다. 티켓이 명시한 `@RestControllerAdvice`와 어긋남.
- **C. 별도 `audit-logger` 마이크로 모듈 + Kafka 프로듀서**: PRD §12 outbox 패턴까지 확장. 장점: 향후 OLV-130과 직접 연결. 단점: 본 티켓 범위를 벗어나며 Kafka는 잠금된 스택이 아님 (PRD §18.3 "Kafka later"). YAGNI.

## Recommendation

**Option A.** 티켓 §Scope·§Hints가 모두 A를 가리키며, B는 Spring MVC 자체 dispatcher 의존도를 키워 테스트가 무거워지고, C는 PRD §18.3·§19와 본 티켓 범위에 모두 어긋난다.

## Learnings

**무엇**: Explore 가정 vs 실제 구현 차이 4가지를 wiki 로 인계.
**왜**: 후속 도메인 티켓이 동일 함정을 재발견하지 않도록 *지금* 못 박는다.

- Explore 가정 "Spring Boot 3.x 의 jackson 이 record 를 자동 직렬화한다" → 사실. 별도 모듈 0개로 envelope 구현 가능.
- 새 발견 1: `@WebMvcTest(controllers=X)` 가 **테스트 클래스 내부 nested static @RestController** 를 자동 등록하지 않는다 — 명시적 `@Import` 필요(GlobalExceptionHandlerTest 5개 테스트 동시 실패의 원인이었다).
- 새 발견 2: `RollingFileAppender` 의 `<file>` 요소를 **생략**해야 활성 파일 = 일자 패턴 파일이 된다. `<file>audit-current.log</file>` 형태였다면 자정 rollover 전까지 IT 가 audit-YYYY-MM-DD.log 를 못 찾아 깨진다.
- 새 발견 3: `@SpringBootTest(webEnvironment=NONE)` + `SecurityFilterAutoConfiguration` 제외 조합은 `SecurityConfig` autowire(HttpSecurity) 가 깨진다 — audit IT 는 `WebEnvironment.MOCK` 으로 회피.
- 운영 메모: Docker Desktop 4.58.1 + Testcontainers 1.20.3 호환성 이슈 — daemon `/info` 응답이 비어있어 모든 strategy 가 BadRequestException. OLV-002 의 Repository IT 가 본 환경에서 미가동(별도 후속).

### Reasoning
- **`@RestControllerAdvice` + `OncePerRequestFilter` (Option A)** 채택. 티켓 §Hints 라인을 그대로 따랐고, Spring MVC `ErrorController` 직접 구현(B) 대비 보일러가 적고 record envelope 매핑이 직관적. 별도 audit 마이크로 모듈/Kafka(C) 는 PRD §18.3·§19 의 잠금 스택을 벗어나 YAGNI.
- **record + `@JsonInclude(NON_NULL)`** — Jackson 기본 동작만으로 envelope 의 nullable 필드를 제거. mixin/모듈 추가 0개.
- **`AuditLogger` 인터페이스 + `LogbackAuditLogger` 빈** — 정적 헬퍼를 피해 도메인 서비스에서 mockable 하도록. 카테고리 enum 도입은 의도적으로 *유보* — 사용처가 늘면 자연스럽게 enum 으로 승격(OLV-100/-130).
- **Trade-offs**:
  - audit 카테고리 enum 미도입 → 호출처에서 typo("LOGN_SUCCESS") 가 컴파일 타임에 잡히지 않음. 의도된 점진적 도입.
  - LogstashEncoder 가 `OLIVE_AUDIT_DIR` 같은 logback context property 를 JSON 필드에 포함 → 약간의 노이즈. OLV-130 에서 정리 가능.
  - `${olive.audit.dir:-log}` default 가 상대 경로 → K8s 배포 시 OLV-130 helm values 에서 절대 경로 주입 필요.
- **Follow-ups (intentionally deferred)**:
  - 카테고리 enum 도입 — OLV-100(payment) / OLV-130(observability) 에서 자연스럽게.
  - request audit interceptor 가 method/path/status/latency 자동 캡처 — OLV-130 범위.
  - OLV-002 Testcontainers Docker Desktop 호환성 이슈 — 별도 후속(OLV-002 의 QA 단계에서 처리).
