## Why

PRD §6.3, §8.1: this is the user-facing search endpoint. Without it,
discovery is limited to category browsing.

## Scope

Endpoints under `/api/search`:

- `GET /api/search/products?keyword=&categoryId=&brandId=&sort=&page=&size=`
  — full-text search against the OpenSearch index. Response shape mirrors
  `/api/products` for client reuse. Sort options: relevance (default),
  popular (salesCount desc), latest (createdAt desc), price asc/desc,
  rating desc.
- `GET /api/search/autocomplete?prefix=&size=10` — prefix completion on
  productName + tags + brandName. Cached in Redis 5 min by prefix.
- `GET /api/search/popular?size=10` — popular search keywords from the
  last hour, computed by a `@Scheduled` job that aggregates a Redis ZSET
  written by `/api/search/products`.

Failure handling per `llm-wiki/99-failure-handling.md`: if OpenSearch is
down, return 503 with `{"error":"검색 일시 중단"}` — never silently fall
back to DB LIKE.

## Acceptance Criteria

- [ ] Search `keyword=선크림` returns all ON_SALE products whose name or
      tags contain "선크림", sorted by relevance.
- [ ] Filter by categoryId combined with keyword returns the intersection.
- [ ] Stopping OpenSearch returns 503 with the documented body.
- [ ] Autocomplete returns prefix matches (case-insensitive).
- [ ] Popular keywords list is non-empty after seeding 100 fake searches
      via the test harness.

## Domain Brief

**무엇**: 사용자 대면 검색 API 3종(키워드 검색 / 자동완성 / 인기검색어)을 OpenSearch + Redis 위에 얹는다.
**왜**: PRD §6.3, §8.1의 핵심 디스커버리 경로. 없으면 사용자는 카테고리 브라우징만 가능 — 검색은 H&B 커머스의 1차 진입 행동.
**As-Is → To-Be**:
- As-Is: 인덱스/동기 파이프라인은 OLV-100에서 준비됐지만 읽기 트래픽은 없음. `/api/search/*` 404.
- To-Be: `/api/search/products`는 `_score`/판매량/최신/가격/별점 정렬 + 카테고리/브랜드 필터, OS 다운 시 HTTP 503 `검색 일시 중단`. `/api/search/autocomplete`는 prefix 매칭 + 5분 Redis 캐시. `/api/search/popular`는 분-bucket ZSET을 1분 주기로 집계.

재사용/누락 핵심:
- 재사용: `OpenSearchClient`(이미 connect/socket timeout 명시) / `ProductDocument.INDEX_NAME` / `ProductPublicService` 응답 모양 (`PublicListItem`) / `PostgresIntegrationSupport` IT 베이스 / `ApiResponse`/`PageMeta` envelope.
- 누락: `SearchPublicController`, `SearchService` (OS 질의 빌더 + DB hydration), `AutocompleteService`, `SearchPopularityRecorder` + `SearchPopularityAggregator`(@Scheduled), `SearchDtos`, `ErrorCode.SEARCH_UNAVAILABLE(503)`.
- 함정: 인덱스에 categoryId/brandId/createdAt 없음 → ID→name lookup + `latest=productId desc` fallback. thumbnailUrl도 인덱스 미포함 → hit ID로 DB hydration 필요.
_세부: docs/OLV-101/explore/details.md_

## Plan Candidates

A. **Read-path만 추가 (티켓 권고)** — 매핑 변경 없이 OpenSearch 9 필드 그대로 질의. categoryId/brandId는 DB lookup으로 name 변환 후 term filter. 자동완성은 `match_phrase_prefix`+`prefix` bool. 인기검색어는 분-bucket ZSET을 1분 단위로 `ZUNIONSTORE` 집계.
B. **매핑 확장** — categoryIds/brandId/createdAt/thumbnailUrl을 인덱스에 추가하고 전량 reindex. DB hydration 불필요해 latency 우수. 단점: V8 outbox 전체 트리거 + reindex 운영 비용 + 본 ticket scope 초과.
C. **DB LIKE fallback 허용** — OS down 시 LIKE로 가짜 결과. wiki 99 §15.3 정책에 명시적으로 금지(일관성 깨짐) — 즉시 기각.

## Recommendation

**A 채택.** 이유: (1) 매핑 변경/재색인 운영 비용 없이 모든 AC 충족, (2) 응답 hydration은 size≤100 단일 batch라 N+1 위험 없음, (3) 인기/자동완성 캐시 키 설계가 wiki 99·95 정책과 일치.
구현 청크: (1) ErrorCode.SEARCH_UNAVAILABLE → (2) SearchDtos + SearchService(질의+hydration+503 매핑) → (3) AutocompleteService(+Redis 5분 캐시) → (4) SearchPopularityRecorder + Aggregator(@Scheduled, !test 프로필) → (5) SearchPublicController 3 endpoints → (6) IT 3 클래스(SearchApiIT/AutocompleteIT/PopularKeywordsIT).
**첫 실패 테스트**: `SearchApiIT#searchByKeyword_선크림_returnsOnSaleProducts` — V3 seed 선크림 enqueue + drainOnce + refresh 후 `GET /api/search/products?keyword=선크림` → 200, `$.data[0].productName contains 선크림`, `$.meta.total >= 1`. 컨트롤러 부재로 RED.

## Learnings

**무엇**: Explore 가정 vs 실제 구현 결과 비교.
**왜**: 향후 Search 관련 작업의 설계 결정에 참고.

**주요 발견**:
- **Plan A 정당성 확인**: size≤100 제한으로 단일 batch hydration이 충분 — N+1 걱정은 기우였음. DB lookup (categoryId→name, brandId→name)도 비용 무시할 수준.
- **Redis ZSET 분할 설계 유효**: 분 단위 bucket으로 누적 + 1분 주기 ZUNIONSTORE 패턴이 sweep 없이 동작. `unionAndStore`는 누락 키를 빈 ZSET으로 취급해 사전 존재 체크 불필요.
- **Clock 주입 패턴**: `SearchPopularityRecorder`가 `Clock`을 주입받아 테스트에서 시간 조작 가능 — 결정론적 IT 작성에 핵심.
- **@Profile("!test") @EnableScheduling**: OLV-100에서 배운 패턴이 그대로 적용됨. IT에서 `aggregateOnce()` 직접 호출으로 결정론적 검증.
