## Why

PRD §3.2 first-class admin requirement and the foundation for everything
sellable. Includes presigned-URL image upload so the admin UI doesn't
proxy bytes through the app.

## Scope

Endpoints under `/api/admin/products` (all require `PRODUCT_ADMIN`):

- `POST /api/admin/products` — create with options + category_ids +
  image_urls. Atomic transaction across products / product_options /
  product_category_mapping / product_images.
- `PATCH /api/admin/products/{id}` — partial update (name, description,
  prices, status). Status transitions validated against the allowed graph
  (DRAFT → ON_SALE; ON_SALE ↔ SOLD_OUT; ON_SALE → STOPPED; HIDDEN any time).
- `GET /api/admin/products` — paginated, filter by status / brand_id /
  category_id / name LIKE.
- `POST /api/admin/products/{id}/options` — add a new option.
- `PATCH /api/admin/products/{id}/options/{optionId}` — update name /
  price / status.
- `POST /api/admin/uploads/product-image` — returns `{uploadUrl, fileUrl}`
  presigned to S3 (LocalStack). The caller PUTs the bytes directly.

Cross-cutting:

- Every successful mutation publishes `ProductUpdatedEvent(productId,
  changeKind)` — wired in OLV-110.
- `BigDecimal` everywhere for money; reject negative prices with 400.
- Audit log every admin mutation with member id + before/after JSON snapshot.

## Acceptance Criteria

- [x] Creating a product with two options + two categories returns 201 and
      a follow-up GET reflects all of it.
- [x] Status transition to an illegal target (e.g., DRAFT → REFUNDED) is
      rejected with 422 (`ErrorCode.INVALID_PRODUCT_STATE_TRANSITION`).
- [x] Presigned upload: integration test acquires a URL, PUTs a tiny PNG,
      and a GET on `fileUrl` returns the bytes.
- [x] Admin edits show up in the audit log.

## Domain Brief

**무엇**: 상품 관리자 CRUD API와 S3 presigned URL 이미지 업로드 기능을 구현한다. Admin UI가 서버를 프록시 없이 S3에 직접 업로드하도록 presigned URL을 발급하고, 모든 변경 시 감사 로그를 남긴다.

**왜**: PRD §3.2 first-class admin requirement로 판매 가능한 모든 것의 기반이다. 이미지를 앱 서버를 통해 프록시하지 않고 S3에 직접 업로드하면 트래픽 부하가 줄어들고, 관리자의 모든 변경을 감사 로그로 추적할 수 있다.

**As-Is → To-Be**:
- As-Is: V3__product.sql에 테이블만 존재. Entity/Repository/Service/Controller 없음. Presigned URL 발급 기능 없음. AdminProductPlaceholderController만 있음.
- To-Be: Product/ProductOption/ProductImage Entity 3개 + Repository 3개 + ProductAdminService + Admin 컨트롤러 2개(Product + Upload) + ErrorCode 확장 + 감사 로그 통합 + IT 3건 이상.

핵심 인바리언트:
- 상태 천이: DRAFT → ON_SALE; ON_SALE ↔ SOLD_OUT; ON_SALE → STOPPED; HIDDEN 언제든지 가능
- 트랜잭션: POST는 products + options + categories + images를 원자적으로 생성
- 돈: base_price, sale_price는 BigDecimal (음수 400 거부)
- 이미지 업로드: Presigned URL → PUT 직접 업로드 → POST 시 imageUrls 포함
- 감사 로그: ADMIN_MUTATION 이벤트로 before/after JSON 스냅샷 기록

위험 R1: 상태 천이 유효성 (DRAFT → REFUNDED = 422)
위험 R2: 옵션/카테고리 동시 생성 트랜잭션
위험 R3: Presigned URL 보안 (파일 크기/타입 검증)
위험 R4: 감사 로그 JSON 직렬화 시 순환 참조

## Plan Candidates

**Option A — 단일 Service + 분리 컨트롤러 (권장)**: ProductAdminService(DB 로직) + ProductUploadService(Presigned URL) 분리. 장점: 테스트 용이, S3 의존성 격리, 확장성. 단점: 클래스 수 다소 많음.

**Option B — Product Service에 Upload 통합**: ProductAdminService에 presigned URL 발급까지 통합. 장점: 클래스 수 감소. 단점: S3 의존성 섞임, 생명주기 차이 무시, 테스트 시 S3 모킹 필수.

**Option C — Controller에서 S3Client 직접 호출**: Service는 DB만, Controller에서 S3. 장점: Service 순수 도메인. 단점: 비즈니스 로직 누출, 테스트가 Controller까지 필요.

_세부: docs/OLV-022/explore/plan-candidates.md_

## Recommendation

**Option A 채택.** ProductAdminService는 DB 로직과 트랜잭션, ProductUploadService는 S3 presigned URL 발급 — 책임 분리로 테스트와 확장성이 우수함.

**첫 실패 테스트**: `ProductAdminApiIT.createProduct_withTwoOptionsAndTwoCategories_returns201` — 옵션 2개 + 카테고리 2개 + 이미지 2개를 포함한 상품 생성 요청이 201 Created + 응답 구조 검증. Product/ProductAdminService/Controller 부재로 RED → 순차 구현으로 GREEN.

구현 순서: Product/Option/Image Entity → Repository 3개 → ProductDtos(ProductCreateRequest 등) → ProductAdminService(ProductUploadService 분리) → ProductAdminController + ProductUploadController → ErrorCode 확장 → 감사 로그 통합 → IT 3건 이상.

_세부: docs/OLV-022/explore/recommendation.md_

## Implementation

**수정 파일 목록**:

1. `Product.java` — 상태 천이 로직(`isValidTransition`), `@BatchSize` 추가로 지연 로딩 최적화
2. `ProductOption.java` — 옵션 상태 enum, 가격 유효성 검증
3. `ProductImage.java` — 이미지 URL, 정렬 순서, 썸네일 플래그
4. `ProductRepository.java` — `findByIdWithDetails` 쿼리(brand JOIN FETCH, images JOIN FETCH, options는 @BatchSize)
5. `ProductAdminService.java` — CRUD 로직, 상태 천이 검증, 감사 로그 통합
6. `ProductUploadService.java` — S3 presigned URL 발급, 파일 크기/타입 검증
7. `ProductAdminController.java` — POST/PATCH/GET 엔드포인트, 페이지네이션/필터링
8. `ProductUploadController.java` — presigned URL 발급 엔드포인트
9. `ProductDtos.java` — 요청/응답 DTO, `ImageResponse.isThumbnail`은 `sortOrder == 1`로 판단

**주요 수정사항**:
- `MultipleBagFetchException` 해결: 두 개의 `@OneToMany` 컬렉션을 동시에 fetch하지 못하는 Hibernate 제약 → `@BatchSize(size = 10)`로 지연 로딩 최적화
- `@Transactional(readOnly = true)` 추가: `get()`/`list()` 메서드에서 `LazyInitializationException` 방지
- `isThumbnail` 판단 로직: `@OrderColumn` 제거 후 DTO에서 `sortOrder == 1` 비즈니스 로직으로 판단

## Resolution

## Learnings

**무엇**: Product Admin API 구현 과정에서 Hibernate 제약사항과 상태 천이 검증 패턴을 학습함.

**왜**: 향후 유사한 도메인(Entity에 다중 `@OneToMany` 관계, 상태 머신 포함) 구현 시 동일 문제를 방지하고 일관된 패턴을 적용하기 위해.

**As-Is → To-Be**:
- As-Is: Hibernate `MultipleBagFetchException` 원인 불명, 상태 천이 로직 위치 미정
- To-Be: `@BatchSize` 패턴으로 해결, Entity 내부 `isValidTransition()`로 캡슐화

**핵심 학습**:
1. **Hibernate 제약**: 단일 쿼리에서 두 `@OneToMany`를 동시에 fetch 불가 → `@BatchSize(size = 10)`로 지연 로딩 최적화
2. **상태 천이 캡슐화**: Entity 내부에 `isValidTransition()` 메서드로 비즈니스 규칙 보존
3. **Presigned URL 패턴**: S3 직접 업로드로 앱 서버 트래픽 부하 감소, `ProductUploadService` 분리로 책임 명확
4. **감사 로그**: Spring Events로 before/after JSON 스냅샷 발행 — 순환 참조 주의 필요

### Reasoning

**Option A 선택 이유**: ProductAdminService(DB 로직) + ProductUploadService(Presigned URL) 분리로 책임 명확, 테스트 용이, S3 의존성 격리. Option B/C는 S3 의존성 혼재 또는 비즈니스 로직 누출로 기각.

**Trade-offs**:
- `@BatchSize` 추가로 지연 로딩 최적화 — N+1 쿼리는 여전히 발생하지만 배치 크기 10으로 완화
- `@OrderColumn` 제거 후 `sortOrder == 1` 비즈니스 로직으로 썸네일 판단 — 데이터 일관성은 애플리케이션 레벨에서 보증
- Docker 볼륨 이슈로 실제 앱 실행 테스트는 생략 — Testcontainers 기반 통합 테스트로 검증 완료

**Follow-ups**:
- OLV-110: `ProductUpdatedEvent` → 검색 색인 outbox + 캐시 무효화 연결
- OpenSearch 색인 동기화: outbox pattern 구현 시 연동
- Admin UI: 프론트엔드에서 presigned URL PUT 구현 필요
