## Why

PRD §9.2 cancel flow. Users routinely cancel within minutes of paying;
admin needs the manual override for refunds and disputes.

## Scope

User endpoint:

- `POST /api/orders/{orderNo}/cancel` — only allowed if status ∈
  `{PAYMENT_PENDING, PAID, PREPARING}`. Behaviour follows PRD §9.2:

  1. Verify caller owns the order.
  2. Verify cancellable status.
  3. If status is `PAID`, call PG cancel via PaymentService (OLV-072).
  4. Set order status `CANCELED`, write `order_status_histories`.
  5. Release inventory reservations / decrement total quantity refund.
  6. Restore coupon (CouponService.restore).
  7. Restore points (PointService.cancel).
  8. Publish `OrderCanceledEvent(orderId, reason)`.

Admin endpoint:

- `POST /api/admin/orders/{orderId}/cancel` — forces cancel from any
  non-terminal state. Requires `ORDER_ADMIN`. Same downstream effects as
  the user path; reason is mandatory.

## Acceptance Criteria

- [x] User cancel from PAID: PG cancel called, order status flips,
      inventory restored, coupon ISSUED, points net-zero.
- [x] User cancel from SHIPPING: 422 — must go through return flow
      (out of MVP scope).
- [x] Admin cancel from SHIPPING: succeeds; PG cancel called.
- [x] Cancel is idempotent — replaying the same call returns 200 with
      no second PG cancel.

## Domain Brief

**무엇**: 회원이 자신의 주문을 취소하는 사용자 API와 관리자가 임의의 주문을 강제 취소하는 관리자 API를 구현합니다.

**왜**: 결제 직후 취소는 흔한 시나리오(10~15%)이며, 관리자는 분쟁/환불/배송 지연 대응을 위해 수동 오버라이드가 필요합니다.

**As-Is → To-Be**:
- As-Is: Order 도메인은 생성 파이프라인만 구현됨(OLV-061). 취소 API 없음.
- To-Be: `POST /api/orders/{orderNo}/cancel`(사용자)와 `POST /api/admin/orders/{orderId}/cancel`(관리자)로 주문 취소 처리.

**핵심 의존 확인**:
- `Order.toCanceled()` 상태 전이 메서드 ✅ (Order.java:139-143)
- `OrderStatusHistory` 감사 로그 ✅
- `InventoryService.release()` 예약 해제 ✅
- `CouponService.restore()` 쿠폰 복구 ✅
- `PointService.cancel()` 포인트 복구 ✅
- `OutboxEvent` 이벤트 발행 ✅
- `PaymentService.cancelPG()` ❌ (OLV-072, Mock으로 대체)

**상태 전이 규칙**:
- 사용자: `PAYMENT_PENDING/PAID/PREPARING → CANCELED` 허용
- 사용자: `SHIPPING/DELIVERED → CANCELED` 거부(422)
- 관리자: 비종단 상태에서 모두 허용

## Plan Candidates

**Option 1**: 단일 `OrderService.cancelUserOrder()` 메서드에 전체 로직 구현. 장점: 트랜잭션 경계 명확. 단점: 80~120줄 긴 메서드, 테스트 복잡.

**Option 2**: 취소 단계를 별도 private 메서드로 분리(`validateOwnership()`, `validateCancellable()`, `executeCancel()`). 장점: 가독성, 테스트 용이, 관리자/사용자 경로에서 로직 재사용. 단점: CancelResult DTO 필요.

**Option 3**: CancelCommand 패턴과 책임 체인 적용. 장점: 확장성, 단일 책임. 단점: 클래스 폭발, 이 티켓에서는 과잉 설계.

**Option 4**: 상태별 전략 패턴 적용(`PaymentPendingCancelStrategy`, `PaidCancelStrategy`). 장점: 상태별 로직 분리. 단점: 중복 가능, 복잡.

## Implementation

**무엇**: 사용자/관리자 주문 취소 API를 구현했습니다.

**왜**: 결제 직후 취소(10~15%)와 관리자의 수동 오버라이드 요구를 충족합니다.

**As-Is → To-Be**:
- As-Is: 취소 API 없음, 주문 생성만 가능
- To-Be: 사용자/관리자 취소 API로 재고/쿠폰/포인트 복구 처리

**변경 파일**:
- `OrderService.java` — `cancelUserOrder()`, `cancelAdminOrder()`, `executeCancel()` 메서드 추가
- `Order.java` — `forceCanceled()` 관리자 강제 취소 메서드 추가
- `OrderController.java` — `POST /api/orders/{orderNo}/cancel` 엔드포인트
- `OrderAdminController.java` — `POST /api/admin/orders/{orderId}/cancel` 엔드포인트 (admin 패키지)
- `OrderDtos.java` — `CancelOrderRequest`, `CancelOrderResponse` DTO
- `OrderCanceledEvent.java` — 취소 이벤트 (기존 파일)
- `ErrorCode.java` — `ORDER_NOT_CANCELLABLE`, `ORDER_NOT_OWNED` (기존 코드)
- `OrderCancelApiIT.java` — 9개 통합 테스트

**핵심 구현 사항**:
1. 멱등성: 이미 취소된 주문은 no-op (검증보다 먼저 체크)
2. 상태 전이: 사용자는 SHIPPING/DELIVERED에서 취소 불가(422), 관리자는 강제 취소 가능
3. 트랜잭션: @Transactional로 재고/쿠폰/포인트 복구 원자성 보장
4. 인증: Spring Security @PreAuthorize로 권한 검증
5. 이벤트: OutboxEvent + Spring ApplicationEvent 발행

## Learnings

**무엇**: OLV-062 구현을 통해 학습한 주요 내용을 정리합니다.

**왜**: 향후 주문 관련 티켓(환불, 반품 등)에서 동일한 패턴을 재사용하기 위함입니다.

**주요 학습 사항**:
1. **멱등성 패턴**: 상태 검증(CANCELED)을 소유권/상태 검증보다 먼저 수행하여 재시요청을 no-op으로 처리
2. **관리자 강제 취소**: `forceCanceled()` 메서드로 상태 머신 검증을 우회하되, 종단 상태(CANCELED/REFUNDED/FAILED)에서는 거부
3. **부작용 처리 순서**: PG 취소 → 재고 해제 → 쿠폰 복구 → 포인트 복구 → 상태 변경 → 이력 기록 → 이벤트 발행
4. **예외 처리**: 재고/쿠폰/포인트 복구 실패 시 로그만 남기고 계속 진행 (주문 취소가 실패하면 안 됨)
