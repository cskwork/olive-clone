## Why

Order creation needs the coupon + point ledger schema present. Splitting
the migration off keeps the API tickets focused on validation logic.

## Scope

Create `V6__promotion.sql` matching PRD §6.8, §7.8:

- `coupons` (PRD §7.8): id, name, discount_type
  (`FIXED_AMOUNT | PERCENTAGE | FREE_SHIPPING | BUY_ONE_GET_ONE | MEMBER_GRADE`),
  discount_value DECIMAL(12,2), min_order_amount DECIMAL(12,2) NULLABLE,
  started_at, ended_at, status (`ACTIVE | INACTIVE`), max_issue_count INT
  NULLABLE, issued_count INT NOT NULL DEFAULT 0, created_at.
- `member_coupons`: id, member_id FK, coupon_id FK, status
  (`ISSUED | USED | EXPIRED | REVOKED`), issued_at, used_at NULLABLE,
  used_order_id NULLABLE FK, expires_at.
  - UNIQUE (member_id, coupon_id) when coupons are once-per-member; allow
    duplicates for multi-issue campaigns by including `id` only.
- `promotions`: id, name, type (`TIME_DEAL | EVENT | CATEGORY_SALE`),
  started_at, ended_at, discount_rule_json JSONB.
- `promotion_products`: promotion_id FK, product_id FK, PRIMARY KEY
  (promotion_id, product_id).
- `points`: id, member_id UNIQUE FK, balance DECIMAL(12,2) — a **cached
  rollup** updated by triggers / app code. Source of truth is
  `point_histories`.
- `point_histories`: id, member_id, change_type
  (`EARN | USE | CANCEL | EXPIRE | ADMIN_ADJUST`), amount DECIMAL(12,2),
  reason VARCHAR(255), order_id NULLABLE, available_at TIMESTAMP NOT NULL,
  expires_at TIMESTAMP NULLABLE, created_at.

## Acceptance Criteria
