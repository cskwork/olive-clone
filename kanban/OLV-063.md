## Why

User needs "주문 조회" (PRD §3.1); admin needs status management
(PRD §3.2 주문 상태 변경, §8.5).

## Scope

User endpoints:

- `GET /api/me/orders?status=&page=&size=` — paginated. Status filter
  optional; sorted by created_at DESC.
- `GET /api/me/orders/{orderNo}` — detail with items, payment summary,
  delivery snapshot, status history.

Admin endpoints (require `ORDER_ADMIN` or higher):

- `GET /api/admin/orders?status=&memberId=&from=&to=&page=&size=` —
  paginated, filterable. PII (recipient phone/address) masked unless the
  caller has the explicit "view PII" permission.
- `GET /api/admin/orders/{orderId}` — full detail incl. status history
  and audit trail.
- `PATCH /api/admin/orders/{orderId}/status` — allowed transitions only:
  `PAID → PREPARING`, `PREPARING → SHIPPING`, `SHIPPING → DELIVERED`,
  `DELIVERED → REFUND_REQUESTED`, etc. Body `{toStatus, reason}`.
  Refuse illegal transitions with 422.

## Acceptance Criteria

- [ ] Admin list: PII masked in JSON unless caller has the PII permission
      claim (test both views).
- [ ] Status transition validation rejects `PAID → DELIVERED` (skipped
      step) with 422.
- [ ] Each successful admin status change writes to
      `order_status_histories` and the audit log.

## Domain Brief

**무엇**: 회원은 자신의 주문 목록/상세를 조회하고, 관리자는 전체 주문을 조회하며 상태를 변경하는 API를 구현합니다.

**왜**: 회원은 구매 이력을 확인해야 하며(주문 조회), 관리자는 주문 상태를 관리하고 CS 대응을 위해 배송지 PII를 확인할 수 있어야 합니다(PRD §3.1, §3.2, §8.5).

**As-Is → To-Be**:
- As-Is: 주문 생성/취소 API만 존재(OLV-061, OLV-062). 조회 API 없음. PII 마스킹 로직 없음.
- To-Be: 회원 주문 목록/상세 API, 관리자 주문 목록(PII 마스킹)/상세/상태 변경 API 구현.

**기존 자산 확인**:
- `Order.OrderStatus` enum과 상태 전이 검증 로직 존재 (`isValidTransition()`)
- `OrderRepository.findByMemberIdOrderByCreatedAtDesc()` 있으나 페이지네이션 없음
- `order_status_histories` 테이블 구조 완비 (감사 추적용)
- 페이지네이션 패턴은 `ProductAdminController` 참조 (`Page<T>`, `PageMeta`)
- PII 대상: `member_addresses.recipient_name`, `phone`, `address_main`, `address_detail`

## Plan Candidates

**Option 1: 단일 `OrderService`에 모든 조회/상태 변경 로직 구현**
- 장점: 기존 `OrderService`와 일관성. 단일 진입점.
- 단점: 메서드 수 증가. 관리자용 PII 마스킹 로직이 서비스 레이어에 혼재.
- 복잡도: 낮음

**Option 2: `OrderQueryService` (조회) + `OrderAdminService` (관리자용) 분리**
- 장점: CQRS 패턴. 조회/명령 분리. PII 마스킹 전용 레이어.
- 단점: 클래스 수 증가. `OrderService`와의 중복 코드 가능성.
- 복잡도: 중간

**Option 3: Repository 직접 QueryDSL + Projection 인터페이스**
- 장점: 타입 안전한 쿼리. N+1 방지. 컴파일 타임 검증.
- 단점: QueryDSL 설정 필요(현재 미활용). Projection 복잡도.
- 복잡도: 높음

## Recommendation

**Option 1**을 채택합니다. 기존 `OrderService`에 조회 메서드를 추가하고, PII 마스킹은 DTO 변환 단계에서 수행합니다. 이 티켓 범위에서는 과잉 설계를 피하고 기존 패턴을 따릅니다.

## Learnings

1. **Nested Record 제약사항**: Java의 중첩 record는 바깥쪽 클래스 내부에서만 참조 가능하므로, 여러 DTO에서 공통으로 사용하려면 최상위 레벨로 정의해야 합니다. `OrderItemResponse`를 `OrderResponse` 내부에서 최상위로 이동하여 해결.

2. **JPA Entity 접근자**: `@OneToOne` 관계 필드는 기본적으로 package-private이지만, 서비스 레이어에서 직접 접근하려면 명시적인 getter가 필요합니다. Lombok을 사용하지 않는 프로젝트에서는 수동으로 추가해야 합니다.

3. **관리자 상태 전이 검증**: Order 엔티티의 `isValidTransition()`은 사용자 취소 흐름에 최적화되어 있습니다. 관리자 상태 변경은 별도 `isValidAdminTransition()` 메서드로 검증하여, 허용된 관리자 전이만 허용하도록 구현했습니다.

## As-Is → To-Be Report

### As-Is
- 회원은 자신의 주문 내역을 조회할 수 없는 상태
- 관리자는 주문 목록/상세를 조회할 수 없어 CS 대응 불가
- 주문 상태 관리가 주문 생성/취소로만 제한됨
- PII 마스킹 로직이 없어 관리자 목록에서 민감 정보 노출 위험

### To-Be
- `GET /api/me/orders` — 회원 주문 목록 조회 (상태 필터 + 페이지네이션)
- `GET /api/me/orders/{orderNo}` — 회원 주문 상세 조회 (상품, 배송지, 상태 이력 포함)
- `GET /api/admin/orders` — 관리자 주문 목록 (PII 자동 마스킹, 다중 필터)
- `GET /api/admin/orders/{orderId}` — 관리자 주문 상세 (전체 정보)
- `PATCH /api/admin/orders/{orderId}/status` — 관리자 상태 변경 (전이 검증 + 감사 로그)

### Reasoning

- **단일 OrderService 접근**: 조회와 명령을 하나의 서비스에서 관리하여 일관성 유지. 관리자용 별도 서비스를 만들지 않고 권한 검증은 컨트롤러 레벨의 `@PreAuthorize`로 처리.
- **PII 마스킹 위치**: 서비스가 아닌 DTO 변환 단계에서 마스킹하여, 상세 조회(마스킹 없음)와 목록 조회(마스킹 있음)를 자연스럽게 분리.
- **상태 전이 검증**: `isValidAdminTransition()`을 별도로 구현하여 관리자만의 허용 전이(PAID→PREPARING, PREPARING→SHIPPING 등)를 엄격히 제어.

### Trade-offs

- **PII 마스킹 세분화**: 현재는 "view PII" 권한 없이 무조건 마스킹하지만, 향후 권한 기반 마스킹을 추가하려면 `AuthenticatedUser`의 권한 체크를 서비스로 이동해야 할 수 있음.
- **상태 전이 복잡도**: 관리자 상태 변경이 `toPaid()`, `toPreparing()` 등 여러 메서드를 호출하는 구조라, 상태 머신 수정 시 영향받을 가능성. 향후 전이 전담 메서드를 Order 엔티티에 추가하는 것을 고려.
