## Why

User needs "주문 조회" (PRD §3.1); admin needs status management
(PRD §3.2 주문 상태 변경, §8.5).

## Scope

User endpoints:

- `GET /api/me/orders?status=&page=&size=` — paginated. Status filter
  optional; sorted by created_at DESC.
- `GET /api/me/orders/{orderNo}` — detail with items, payment summary,
  delivery snapshot, status history.

Admin endpoints (require `ORDER_ADMIN` or higher):

- `GET /api/admin/orders?status=&memberId=&from=&to=&page=&size=` —
  paginated, filterable. PII (recipient phone/address) masked unless the
  caller has the explicit "view PII" permission.
- `GET /api/admin/orders/{orderId}` — full detail incl. status history
  and audit trail.
- `PATCH /api/admin/orders/{orderId}/status` — allowed transitions only:
  `PAID → PREPARING`, `PREPARING → SHIPPING`, `SHIPPING → DELIVERED`,
  `DELIVERED → REFUND_REQUESTED`, etc. Body `{toStatus, reason}`.
  Refuse illegal transitions with 422.

## Acceptance Criteria

- [ ] Admin list: PII masked in JSON unless caller has the PII permission
      claim (test both views).
- [ ] Status transition validation rejects `PAID → DELIVERED` (skipped
      step) with 422.
- [ ] Each successful admin status change writes to
      `order_status_histories` and the audit log.
