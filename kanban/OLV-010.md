## Why

OLV-011 needs the member schema present before signup/login can write
anything. Splitting the migration off keeps the API ticket focused on
behaviour.

## Scope

Create `V2__member.sql` matching PRD §7.1 exactly for `members`, plus the
related tables described in `llm-wiki/10-member-domain.md`:

- `members` (PRD §7.1): id, email UNIQUE, password_hash, name, phone,
  status, grade_id, created_at, updated_at.
- `member_addresses`: id, member_id FK, recipient_name, phone,
  zipcode, address_main, address_detail, is_default BOOLEAN, created_at.
- `member_grades`: id, name, discount_rate DECIMAL(5,2), point_rate
  DECIMAL(5,2), benefit_description, sort_order.
- `member_login_histories`: id, member_id, login_at, ip_address,
  user_agent, success BOOLEAN, failure_reason.
- `member_refresh_tokens` (for OLV-005 contract): id, member_id, token_hash
  CHAR(64), issued_at, expires_at, revoked_at NULLABLE.
- Seed three default member grades (BRONZE / SILVER / GOLD) in the same
  migration.

## Acceptance Criteria

- [ ] `./gradlew flywayInfo` shows V2 applied with no errors.
- [ ] Repository integration test inserts and reads back a member, address,
      and login history row.
- [ ] `EXPLAIN ANALYZE SELECT * FROM members WHERE email = ?` uses an index
      (UNIQUE constraint creates it automatically — verify it's there).

## Hints

- Use `BIGSERIAL` for id (Postgres) — not MySQL's AUTO_INCREMENT syntax.
- All `created_at` / `updated_at` default to `now()`. Use a trigger or
  JPA `@PrePersist`/`@PreUpdate` (the latter is simpler) for the updated_at
  refresh.
- Add an index on `member_login_histories.member_id, login_at DESC` for
  the audit query path.
