## Why

OLV-011 needs the member schema present before signup/login can write
anything. Splitting the migration off keeps the API ticket focused on
behaviour.

## Scope

Create `V2__member.sql` matching PRD §7.1 exactly for `members`, plus the
related tables described in `llm-wiki/10-member-domain.md`:

- `members` (PRD §7.1): id, email UNIQUE, password_hash, name, phone,
  status, grade_id, created_at, updated_at.
- `member_addresses`: id, member_id FK, recipient_name, phone,
  zipcode, address_main, address_detail, is_default BOOLEAN, created_at.
- `member_grades`: id, name, discount_rate DECIMAL(5,2), point_rate
  DECIMAL(5,2), benefit_description, sort_order.
- `member_login_histories`: id, member_id, login_at, ip_address,
  user_agent, success BOOLEAN, failure_reason.
- `member_refresh_tokens` (for OLV-005 contract): id, member_id, token_hash
  CHAR(64), issued_at, expires_at, revoked_at NULLABLE.
- Seed three default member grades (BRONZE / SILVER / GOLD) in the same
  migration.

## Acceptance Criteria

- [ ] `./gradlew flywayInfo` shows V2 applied with no errors.
- [ ] Repository integration test inserts and reads back a member, address,
      and login history row.
- [ ] `EXPLAIN ANALYZE SELECT * FROM members WHERE email = ?` uses an index
      (UNIQUE constraint creates it automatically — verify it's there).

## Hints

- Use `BIGSERIAL` for id (Postgres) — not MySQL's AUTO_INCREMENT syntax.
- All `created_at` / `updated_at` default to `now()`. Use a trigger or
  JPA `@PrePersist`/`@PreUpdate` (the latter is simpler) for the updated_at
  refresh.
- Add an index on `member_login_histories.member_id, login_at DESC` for
  the audit query path.

## As-Is -> To-Be Report

### As-Is

- 회원 도메인 테이블 0개. V1 (init baseline placeholder) 만 적용된 상태.
- OLV-011 (signup/login API) 가 진입할 storage 가 없어서 blocked_by 체인이 끊김.

### To-Be

- `V2__member.sql` 신규 — 5 테이블 (members / member_addresses / member_grades / member_login_histories / member_refresh_tokens) + `set_updated_at()` trigger 함수 + 3건의 등급 시드.
- `MemberSchemaIntegrationTest` 신규 — `@DataJpaTest` + `PostgresIntegrationSupport` 위에서 5 케이스 GREEN.
- `PostgresIntegrationSupport` 정정 — `@Testcontainers`/`@Container` per-class lifecycle 을 JVM-singleton 정적 초기화로 교체. OLV-002 의 `RepositoryIntegrationTest` 와 본 티켓의 `MemberSchemaIntegrationTest` 두 `@DataJpaTest` 가 같은 컨테이너를 공유하면서도 첫 클래스 종료 시 컨테이너 stop 으로 두 번째 클래스가 죽는 문제를 회피.

### Reasoning

- **append-only V2** 채택 (옵션 A). V1 placeholder 를 편집해서 도메인 테이블을 합치는 옵션 B 는 OLV-002 의 hard rule (`이미 main 에 커밋된 Flyway 파일은 절대 편집 금지`) 위반 — 운영 DB 이관 시 checksum 깨짐.
- **member_refresh_tokens 를 본 티켓에 포함** — kanban 본문 §Scope 에 명시됐고 OLV-005 의 JWT contract (token_hash CHAR(64), revoked_at NULLABLE) 가 본 마이그레이션에서 만족돼야 OLV-011 이 signup/login 안에서 refresh 토큰을 안전하게 발행/회수할 수 있다. 분리 마이그레이션은 OLV-011 의 blocked_by 를 두 단계로 늘림.
- **트리거 vs JPA `@PreUpdate`** — kanban Hints 는 둘 다 허용. 트리거를 채택한 이유: native SQL 업데이트 경로 (예: 배치 잡, admin 콘솔) 에서도 `updated_at` 이 일관되게 갱신됨. JPA 만 신뢰하면 native query 가 우회.
- **partial unique index** `uniq_member_addresses_default_per_member` — 회원당 default 배송지 1건 invariant 를 DB 가 강제. 어플리케이션 레이어 race condition 으로 두 행이 default=TRUE 로 들어가는 사고 차단.
- **`member_login_histories.member_id` 가 NULL 허용 + ON DELETE SET NULL** — UNKNOWN_EMAIL 실패 (해당 회원 없음) 도 감사 로그가 남아야 PRD §16.2 충족. 회원 삭제 시 감사 행은 보존 (anonymize) 하는 GDPR-friendly 동작.
