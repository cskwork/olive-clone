## Why

Cache, distributed lock, search, and image upload tickets all reference
infra that must already exist. Adding it once now saves five repeat docker
edits later.

## Scope

Extend `docker-compose.yml` with:

- `redis:7-alpine` (port 6379, no auth in dev).
- `localstack/localstack:latest` (port 4566) preconfigured with one
  S3 bucket `commerce-images-local`.
- `opensearchproject/opensearch:2` (port 9200) single-node, security plugin
  disabled in dev (`DISABLE_SECURITY_PLUGIN=true`).

Wire Spring config for `local` profile:

- `spring.data.redis.host/port` for Redis.
- `aws.s3.endpoint=http://localhost:4566`, region `ap-northeast-2`, signing
  with anonymous-equivalent credentials suitable for LocalStack.
- `opensearch.uris=http://localhost:9200`.

Add a `common.config` module with one bean per integration
(`RedisTemplate<String,String>`, `S3Client`, `OpenSearchClient`) so domain
tickets just `@Autowired` them.

## Acceptance Criteria

- [ ] `docker compose up -d` brings all four services up.
- [ ] `./gradlew test` includes one Testcontainers integration test per
      service that proves the bean is healthy: Redis SET/GET, S3 PUT/GET,
      OpenSearch index existence check.
- [ ] Boot logs show all three clients initialized at startup with no
      stack traces.

## Hints

- Use `org.springframework.boot:spring-boot-starter-data-redis`
  (Lettuce client by default).
- For S3, use the AWS SDK v2 (`software.amazon.awssdk:s3`).
- For OpenSearch, use the official `opensearch-rest-client` +
  `opensearch-java` library.
- Add Testcontainers modules: `redis`, `localstack`, `opensearch`.

## Plan Candidates

**Option A — `@ServiceConnection` 적극 활용 (Spring Boot 3.1+ auto-wire)**: Redis/LocalStack/OpenSearch 컨테이너에 `@ServiceConnection`을 붙이면 Spring Boot가 spring.data.redis.* / aws.* 자동 매핑. 코드 최소.

**Option B — `@DynamicPropertySource` 명시 주입 (선택)**: 각 IT 베이스가 컨테이너 host/port를 읽어 우리가 정한 키(`spring.data.redis.host/port`, `aws.s3.endpoint`, `olive.opensearch.uris`)에 박는다. SDK 빈 정의를 우리 손으로 통제.

**Option C — 통합 IT 한 클래스로 묶기**: 한 `@SpringBootTest`에 3 컨테이너 모두. CI 시간 절약하지만 한 컨테이너 실패 → 3 AC 모두 RED.

## Recommendation

**Option B** 채택. 이유: ① OpenSearchClient는 `@ServiceConnection` 자동 빈이 없어 어차피 수동 정의 필요 — 셋 다 동일 패턴이 가독성 ↑. ② LocalStack `@ServiceConnection`은 spring-cloud-aws 한정 — AWS SDK v2 직접 사용 시 효과 없음. ③ 빈 3개 모두 `common.config`에서 우리가 통제하는 정의가 도메인 티켓의 디버깅 수월.

## Learnings

**무엇**: Spring 자동설정과의 협업·외부 호출의 부팅 가드·테스트 컨텍스트 슬라이스 3건이 Explore 단계의 가정을 뒤집었다.
**왜**: baseline 단계에서 못 박지 않으면 OLV-030/090/095/110 등 후속 5+ 티켓이 같은 함정을 반복한다.
**As-Is → To-Be**:
- As-Is: 빈 직접 정의가 default 가정, S3 부팅 시 외부 호출 가능, 테스트 컨텍스트는 풀 컨텍스트 가정.
- To-Be: 자동설정 위임이 default, 외부 호출은 LocalStack mode 한정, 테스트는 좁은 슬라이스 + `@Configuration` 명시.

- Spring `RedisAutoConfiguration`의 `stringRedisTemplate`은 이미 `RedisTemplate<String,String>`(서브클래스)을 만족 — 우리가 같은 이름으로 빈을 다시 정의하면 `BeanDefinitionOverrideException`. 자동설정에 위임 + init log만 남기는 것이 baseline 패턴.
- AWS SDK v2는 endpoint 미설정 시 실제 AWS 엔드포인트로 호출 — 부팅 시 `headBucket`/`createBucket`이 익명 자격으로 운영 AWS에 닿아 403. baseline 정책: **endpoint 명시(=LocalStack)일 때만 warm-up**, 운영은 devops가 외부에서 버킷 생성.
- `@SpringBootTest(classes = X.class)`의 X는 **반드시 `@Configuration` 메타데이터**가 있어야 `@Import` / `@EnableConfigurationProperties` / `@ImportAutoConfiguration`이 처리됨 — 누락 시 컨텍스트 로드는 성공하지만 빈이 등록되지 않음(silent failure).
- LocalStack init script + Spring `ensureBucket` 이중 안전망 — 도커 마운트 환경 차이로 ready.d hook이 가끔 무시되어도 부팅 시점에 자동 복구.

### Reasoning

- Spring 자동설정 `StringRedisTemplate`이 이미 `RedisTemplate<String,String>` 서브클래스 — 새 빈 정의 대신 자동설정에 위임. 1차 라운드에서 `BeanDefinitionOverrideException`(18 IT 회귀) 후 학습.
- S3 bucket warm-up은 LocalStack mode 한정. 운영 AWS는 devops가 외부에서 버킷 생성 — 부팅 시 외부 호출이 시한폭탄이 되는 구조 차단.
- OpenSearch transport는 legacy `RestClient` + `RestClientTransport` 채택 (안정성 우선). `ApacheHttpClient5TransportBuilder`도 가능하나 baseline은 검증된 경로.
