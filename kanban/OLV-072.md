## Why

This is the closest thing to a money-touching method in the system. Every
hard rule in PRD §20.4 / §14.4 / §15.1 is enforced HERE.

## Scope

`POST /api/payments/confirm` (PRD §8.4). Body:

```json
{ "orderNo": "ORD202605100001", "paymentKey": "pg_payment_key", "amount": 35000 }
```

The 8-step PRD pipeline:

1. **주문 조회**: load order by `orderNo`. 404 if missing. 422 if status
   ≠ `PAYMENT_PENDING`.
2. **결제 금액 검증**: `payments.requested_amount == request.amount`
   AND `== orders.final_payment_amount`. Mismatch → 422
   `PAYMENT_AMOUNT_MISMATCH` + audit log + alert.
3. **PG사 결제 승인 API 호출**: via `PgClient.confirmPayment`. On
   `PgTimeoutException`: leave order in `PAYMENT_PENDING`, return 504 with
   guidance to retry. On PG `FAILED`: set `payments.status=FAILED`, set
   `orders.status=PAYMENT_PENDING` (per PRD §9.3), release reservation
   (no — leave reservation per PRD §15.1; release only on confirmed PG
   FAIL via reconciliation batch).
4. **결제 성공 시 주문 상태 PAID 변경**: also update `payments.status=APPROVED`,
   set `approved_at`, `approved_amount`.
5. **재고 선점 확정**: `InventoryService.commit(orderId)`.
6. **쿠폰 사용 처리**: `CouponService.markUsed(memberCouponId, orderId)`.
7. **포인트 사용 처리**: `PointService.use(memberId, point_used_amount,
   orderId)`. Also schedule earned points via
   `PointService.earnScheduled(...)` with available_at = +14d (배송 완료
   기준은 OLV-081 이벤트로 다시 갱신).
8. **주문 완료 알림 발송**: publish `PaymentApprovedEvent(orderId)` via
   outbox; subscribers in OLV-110 fan out.

**Idempotency rules (PRD §20.4)**:

- Accept `Idempotency-Key` header. If a `payment_transactions` row already
  exists with `(payment_id, kind=APPROVE, idempotency_key)`, return the
  cached prior response — do NOT re-call PG.
- Even without an idempotency header: a second confirm call against an
  order already in `PAID` returns 200 with the original `paymentKey`,
  not an error.

## Acceptance Criteria

- [ ] Happy path: PAYMENT_PENDING → PAID, inventory committed, coupon
      used, points earned-pending. Single `PaymentApprovedEvent` in the
      outbox.
- [ ] Amount mismatch: PG NOT called, order stays PAYMENT_PENDING, 422
      returned, audit log row written.
- [ ] PG returns FAIL: `payments.status=FAILED`, order stays
      `PAYMENT_PENDING`, reservation NOT released here (kept until
      reconciliation batch confirms FAIL).
- [ ] PG timeout: 504, no state change.
- [ ] Replay with same Idempotency-Key: returns cached response, PG not
      called twice (assert call count = 1).
- [ ] Replay without Idempotency-Key on already-PAID order: 200 with
      cached response, PG not called.

## Hints

- The reconciliation/release piece is OLV-120's job. Don't release
  reservations here on PG-side FAIL — it muddies the audit trail and
  breaks the batch reconciliation invariant.
- Wrap steps 4-7 in a single `@Transactional`; step 8 publishes
  AFTER_COMMIT.

- V9__payment.sql에 idempotency 보호를 위한 UNIQUE 제약 `(payment_id, kind, idempotency_key)`이 이미 존재

- ErrorCode에 PAYMENT_AMOUNT_MISMATCH, PG_TIMEOUT, PG_FAILED, IDEMPOTENCY_CONFLICT가 이미 정의됨
- InventoryService.commit(), CouponService.markUsed(), PointService.use/earnScheduled() 메서드가 이미 존재

## Learnings

1. **멱등성 구현**: idempotencyKey로 payment_transactions 테이블의 UNIQUE 제약 `(payment_id, kind, idempotency_key)`을 활용하면 DB 레벨에서 중복 요청을 방지할 수 있음
2. **PG 타임아웃 처리**: PgTimeoutException을 Controller에서 504로 변환하여 클라이언트가 재시도할 수 있도록 함
3. **금액 검증**: 이중 검증(requested_amount vs order.final_payment_amount)으로 데이터 정합성 보장
4. **포인트 시각**: static final로 선언하면 모든 인스턴스가 같은 시각을 공유하므로 주의 필요

## As-Is -> To-Be Report

### As-Is
- Payment 스키마(V9__payment.sql)와 PgClient/MockPgClient만 존재
- POST /api/payments/confirm 엔드포인트 없음
- 결제 확인 로직 미구현

### To-Be
- POST /api/payments/confirm 엔드포인트 동작
- 8단계 파이프라인: 주문 조회 → 금액 검증 → PG 승인 → 상태 변경 → 재고/쿠폰/포인트 처리 → outbox 이벤트
- 멱등성 보장 (Idempotency-Key 헤더)
- PG 타임아웃 시 504 반환

### Reasoning
- **기존 패턴 준수**: OrderService/CouponService와 동일한 일체형 Service 패턴 사용
- **단일 트랜잭션**: Step 4-7을 @Transactional로 묶어 원자성 보장
- **멱등성**: payment_transactions 테이블의 UNIQUE 제약 활용

### Trade-offs
- **포인트 적립 실패**: 주문을 실패시키지 않고 로그만 남김 (업무적 결정)
- **PG FAILED 시 재고 확정 보류**: OLV-120 reconciliation 배치에 위임
