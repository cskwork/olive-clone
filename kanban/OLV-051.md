## Triage

티켓이 실행 가능한 상태입니다. 의존성(OLV-050, OLV-005)이 완료되었고, AC가 명확합니다. Explore 단계로 진행합니다.

## Why

Order creation pipeline (OLV-061) depends on `CouponService.validate()` and
`CouponService.use()`. Land them with their own tests so the order ticket
inherits a known-good API.

## Scope

Admin endpoints under `/api/admin/coupons` (require `PRODUCT_ADMIN` or
`SUPER_ADMIN`):

- `POST /api/admin/coupons` — create coupon definition.
- `GET /api/admin/coupons` — list with filters (status, discount_type).
- `PATCH /api/admin/coupons/{id}/status` — suspend / re-activate.
- `POST /api/admin/coupons/{id}/issue` — bulk-issue to a member id list
  (atomically respects `max_issue_count`).

User endpoints:

- `GET /api/me/coupons?status=ISSUED` — list owned coupons.
- (Coupon use is not a standalone endpoint — it happens during order
  creation. Expose `CouponService.tryReserve(memberId, couponId, cartTotal)`
  for OLV-061 to call.)

Service contract for OLV-061 to consume:

- `validate(memberId, couponId, orderAmount, productIds): ValidatedCoupon`
  — throws `BusinessException(COUPON_INVALID, reason)` with reason ∈
  `EXPIRED | NOT_OWNED | ALREADY_USED | MIN_AMOUNT_NOT_MET | NOT_APPLICABLE_PRODUCT`.
- `markUsed(memberCouponId, orderId)` — called inside the order-create txn.
- `restore(memberCouponId, orderId)` — called on order cancel.

## Acceptance Criteria

- [ ] Bulk issue: 1000 member ids + max_issue_count=500 → exactly 500
      `member_coupons` rows + `coupons.issued_count = 500`. Concurrent
      issue calls do not exceed the cap.
- [ ] `validate()` returns the right `BusinessException` reason for each
      failure mode (parameterized test).
- [ ] `markUsed` then `restore` is idempotent and the coupon ends in
      ISSUED status.

## Hints

- Use a row-level lock (`SELECT ... FOR UPDATE`) on `coupons` during bulk
  issue to atomically check + bump `issued_count`.
- Audit log every admin coupon mutation.

## Domain Brief

**무엇**: 쿠폰 관리자 API(생성/조회/상태변경/대량발급)와 회원 쿠폰 API(목록조회/사용)를 구현합니다. 주문 생성 파이프라인(OLV-061)에서 호출할 `validate()`, `markUsed()`, `restore()` 서비스 메서드도 함께 제공합니다.

**왜**: OLV-061(주문 생성)이 쿠폰 검증과 사용 로직에 의존하므로, 먼저 독립적으로 테스트된 쿠폰 서비스를 완성해야 주문 티켓이 안정적으로 진행됩니다.

**As-Is → To-Be**:
- As-Is: V6 마이그레이션으로 `coupons`, `member_coupons` 테이블만 존재하고, 도메인 엔티티/서비스/컨트롤러가 없음
- To-Be: 쿠폰 생성/발급/검증/사용 전체 라이프사이클을 처리하는 `CouponService`와 Admin/User API가 완성됨

**핵심 발견**:
1. **스키마 제약조건 활용**: `coupons.issued_count <= max_issue_count` 체크는 DB constraint로 강제되지만, 동시성 제어를 위해 `SELECT FOR UPDATE`가 필요함
2. **중복 발급 방지**: `member_coupons(member_id, coupon_id)` WHERE status='ISSUED' 유니크 인덱스로 한 회원이 같은 쿠폰을 중복 발급받지 못함
3. **에러 코드 확장**: 기존 `COUPON_INVALID` 외에 `COUPON_NOT_FOUND`, `COUPON_EXPIRED`, `COUPON_ISSUE_LIMIT_EXCEEDED` 등 추가 필요
4. **OLV-061 계약**: `validate()`는 `BusinessException(COUPON_INVALID, reason)`을 던지며, reason ∈ `EXPIRED | NOT_OWNED | ALREADY_USED | MIN_AMOUNT_NOT_MET | NOT_APPLICABLE_PRODUCT`

## Learnings

1. **@DataJpaTest 트랜잭션 경계와 동시성 테스트 충돌**: `@DataJpaTest`는 각 테스트를 트랜잭션으로 감싸고 테스트 후 롤백합니다. 하지만 동시성 테스트에서 별도 스레드를 생성하면 해당 스레드는 원본 트랜잭션 컨텍스트를 볼 수 없어 데이터 조회가 실패합니다. 해결: 동시성 테스트 메서드에서 `@Transactional`을 제거하고 `@BeforeEach`에서 생성한 데이터는 커밋되도록 처리.
2. **CopyOnWriteArrayList for thread-safe results**: 여러 스레드에서 결과를 수집할 때 `ArrayList`는 `ConcurrentModificationException`을 발생시킬 수 있습니다. `CopyOnWriteArrayList`를 사용하면 스레드 안전성이 보장됩니다.
3. **SELECT FOR UPDATE 패턴의 중요성**: 대량 발급에서 `max_issue_count`를 초과하지 않으려면 DB 레벨 락이 필수입니다. JPA의 `@Lock(PESSIMISTIC_WRITE)`을 사용하면 `SELECT ... FOR UPDATE` 쿼리가 생성되어 원자성을 보장합니다.
