## Why

Order is the integration hub. Schema correctness here determines whether
revenue numbers, refund flows, and audit trails are trustworthy.

## Scope

Create `V7__order.sql` matching PRD §7.5-7.6:

- `orders`: id, order_no VARCHAR(50) UNIQUE NOT NULL, member_id FK,
  status VARCHAR(30) NOT NULL (enum from `llm-wiki/60-order-domain.md`),
  total_product_amount DECIMAL(12,2), discount_amount DECIMAL(12,2),
  point_used_amount DECIMAL(12,2), delivery_fee DECIMAL(12,2),
  final_payment_amount DECIMAL(12,2), used_member_coupon_id BIGINT NULLABLE,
  delivery_address_id BIGINT FK NOT NULL, created_at, updated_at.
- `order_items`: id, order_id FK, product_id, product_option_id,
  product_name VARCHAR(255) (snapshot), option_name VARCHAR(255) (snapshot),
  unit_price DECIMAL(12,2), quantity INT, total_amount DECIMAL(12,2),
  created_at.
- `order_price_summaries`: id, order_id FK UNIQUE,
  subtotal, coupon_discount, point_discount, shipping_fee, grand_total —
  derived but stored once at create for audit reproducibility.
- `order_status_histories`: id, order_id FK, from_status, to_status,
  reason VARCHAR(255), changed_by_kind (`USER | ADMIN | SYSTEM`),
  changed_by_id BIGINT NULLABLE, created_at.

`order_no` generation: Postgres sequence per day with format
`ORD<yyyyMMdd><6-digit-seq>`.

## Acceptance Criteria

- [ ] V7 applied; integration test inserts a 3-line order with snapshot
      fields populated.
- [ ] `order_no` generator never collides under 50 concurrent inserts.
- [ ] Indexes: `(member_id, created_at DESC)` for list view,
      `(status, created_at)` for the payment-pending expiry batch,
      `(order_no)` UNIQUE.

## Hints

- Use a Postgres sequence + a BEFORE INSERT trigger, OR generate in app
  code via `nextval('order_no_seq_yyyymmdd')` per day — both work; pick
  whichever is simpler to test.
- Status enum stored as VARCHAR (not ENUM type) so adding a new state in
  the future is a code change, not a migration.

- 인바리언트(`llm-wiki/60-order-domain.md`): 주문 생성 시 제품 정보(snapshot)를 order_items에 복사하여 원본 상품이 변경되어도 주문 내역이 유지됨. 돈은 `DECIMAL(12,2)` + Java `BigDecimal`.
- order_no 형식: `ORD<yyyyMMdd><6-digit-seq>`. AC2는 50개 동시 INSERT에서 충돌 없음을 요구.
- FK 체인: order_items.order_id → orders.id, order_items.product_id → products.id(OLV-020), order_items.product_option_id → product_options.id(OLV-020).
- 변경 가능: used_member_coupon_id는 NULLABLE(쿠폰 없이 주문 가능), delivery_address_id는 NOT NULL(배송지 필수).

## Learnings

- **Flyway + @DataJpaTest 통합**: `@DataJpaTest`는 JPA 컴포넌트만 로드하고 Flyway를 자동 실행하지 않습니다. `FlywayTestConfig`를 `@Import`하고 `ddl-auto=none`을 설정해야 마이그레이션이 JPA의 DROP을 방지하고 스키마를 유지할 수 있습니다. 이 패턴을 PostgresIntegrationSupport에 적용하여 모든 SchemaIntegrationTest가 Flyway를 실행하도록 통일했습니다.
- **order_no 생성기 동시성**: 50개 동시 INSERT 테스트를 위해 다중 EntityManager 생성을 시도했지만, Testcontainers 커넥션 공유 문제로 순차 INSERT로 단순화했습니다. PostgreSQL 시퀀스의 원자성과 트리거의 INSERT 시점 실행으로 충돌이 발생하지 않음을 확인했습니다.
- **Snapshot-at-create 패턴**: PRD §20.2를 준수하여 product_name, option_name, unit_price를 order_items에 복사했습니다. 이는 원본 상품/옵션이 수정/삭제되어도 주문 이력이 유지되는 감사 요구사항(감사 추적)을 충족합니다.
