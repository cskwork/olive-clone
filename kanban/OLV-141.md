## Why

PRD §16.3 sets latency / error-rate thresholds; without a baseline we
can't tell when we cross them. Establish the load profile + the
metric snapshots.

## Scope

`infra/k6/` directory:

- `product-list.js` — 50 VUs for 2 min, GET `/api/products?categoryId=1`
  with random sort. Goal: p95 < 300ms.
- `order-create.js` — 20 VUs for 2 min, login + cart + order create.
  Includes a "stock-out tail" test: the last 30s drives 50 VUs against a
  10-stock SKU to verify no oversell.
- A `runload.sh` wrapper that hits the locally-running app and saves
  results under `docs/OLV-141/qa/k6-{scenario}-{timestamp}.json`.

A short `infra/k6/README.md` documents:
- prerequisite (k6 installed),
- target environment assumption (local docker-compose stack),
- how to interpret pass/fail (latency bands, oversell = ANY 200 beyond
  stock = fail).

## Acceptance Criteria

- [ ] `k6 run infra/k6/product-list.js` finishes; p95 captured in summary.
- [ ] `k6 run infra/k6/order-create.js` finishes; the stock-out tail
      reports exactly 10 successful orders, the rest 422
      `INSUFFICIENT_INVENTORY`, zero 500s.
- [ ] Prometheus / Grafana dashboard from OLV-130 shows the load spike
      and the post-load steady state.

## Recommendation

**선택**: 옵션 1 - 독립 스크립트 + 공통 라이브러리 구조

**이유**: 티켓 요구사항 명시, 각 시나리오 독립 실행 용이, 코드 재사용성

**첫 번째 실패 테스트 (k6 script 기준)**:
```javascript
// product-list.js
import http from 'k6/http';
import { check } from 'k6';

export const options = {
    stages: [{ duration: '2m', target: 50 }],
    thresholds: { http_req_duration: ['p(95)<300'] }
};

export default function() {
    const res = http.get('http://localhost:8080/api/products?categoryId=1&sort=LATEST');
    check(res, { 'status is 200': (r) => r.status === 200 });
}
```

## Learnings

1. **k6 SharedArray**: `SharedArray`는 VU 간 데이터 공유를 위해 순차적 접근을 보장합니다. 일반 배열과 달리 내부적으로 락이 구현되어 있어 동시성 문제를 방지합니다.
2. **k6 환경변수**: `__ENV` 객체를 통해 쉘 환경변수에 접근할 수 있습니다. 이를 통해 타겟 URL, 모드 전환 등을 런타임에 제어할 수 있습니다.
3. **k6 handleSummary**: 테스트 종료 후 `handleSummary()` 함수로 결과를 요약하고, 여러 형식(JSON, 텍스트)으로 저장할 수 있습니다.
4. **재고 고갈 테스트 설계**: 재고 10개인 SKU에 대해 50 VU를 동시에 보내면, 정확히 10건만 성공하고 나머지는 422를 반환해야 합니다. 이를 통해 Redis 분산 락과 DB 트랜잭션 격리가 올바르게 동작하는지 검증할 수 있습니다.

### Reasoning

- **k6 선택**: Go 기반 단일 바이너리로 설치 간편, JavaScript 스크립트로 작성 가능, Grafana와 연동 용이

- **모듈 분리**: 공통 기능을 `lib/` 폴더로 분리하여 코드 재사용성 확보
- **환경변수**: `BASE_URL`, `STOCK_OUT_MODE` 등으로 런타임 설정 제어

### Trade-offs

- **선택**: k6 vs JMeter vs Gatling → k6는 코드로 작성 가능, CI/CD 파이프라인에 통합 용이
- **선택**: runload.sh 래퍼 vs 직접 k6 실행 → 래퍼는 서비스 확인, 결과 저장 등 부가 기능 제공
