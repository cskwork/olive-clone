## Why

Bridges payment-complete to "사용자 화면에서 배송 추적이 보임". Without
this, the order lifecycle dead-ends at PAID.

## Scope

Schema (`V9__delivery.sql`):

- `deliveries`: id, order_id FK, delivery_address_id FK, carrier_name,
  invoice_no NULLABLE, status (enum from `llm-wiki/80-delivery-domain.md`),
  created_at, updated_at.
- `delivery_status_histories`: id, delivery_id FK, from_status, to_status,
  reason, created_at.
- `delivery_retry_queue`: id, delivery_id, request_kind
  (`ISSUE_INVOICE | FETCH_STATUS`), payload_json, retry_count,
  next_retry_at, last_error, status (`PENDING | DONE | DEAD`).

Service:

- `DeliveryService.prepareForOrder(orderId)` — invoked by
  `PaymentApprovedEvent` subscriber. Creates one `deliveries` row per
  warehouse (MVP: one per order). Status `READY`.
- `CarrierClient` interface + `MockCarrierClient`:
  - `issueInvoice(deliveryId)` — returns a fake `invoice_no`. Mock
    auto-flips status `READY → INVOICE` immediately and `INVOICE →
    SHIPPING` after 1 min (test-overridable).
  - `fetchStatus(invoiceNo)` — returns current status. Mock walks
    `SHIPPING → DELIVERED` after another 1 min.
- Failed carrier calls land in `delivery_retry_queue`.

User endpoint:

- `GET /api/me/orders/{orderNo}/deliveries` — list deliveries with current
  status + invoice_no.

Admin endpoints (require `ORDER_ADMIN`):

- `POST /api/admin/deliveries/{id}/issue-invoice` — manual replay.
- `GET /api/admin/deliveries?status=&page=&size=` — operational view.

## Acceptance Criteria

- [x] After a successful payment confirm (OLV-072), exactly one
      `deliveries` row exists for the order.
- [x] Mock carrier walk: READY → INVOICE → SHIPPING → DELIVERED with
      history rows for each transition.
- [x] Carrier failure: row lands in `delivery_retry_queue`, admin can
      manually replay.
- [x] `DeliveryCompletedEvent` published when status reaches DELIVERED;
      OLV-110 wires the subscriber.

## Hints

- Use `@Async` + `@Scheduled` (1 min interval) for the mock walk so the
  e2e ticket (OLV-140) can verify the full payment → delivery loop.
- ShedLock for the scheduled walker to avoid duplicate execution on
  multi-instance deploys.

## Domain Brief

**무엇**: 결제 완료(PAID) 이후 배송 준비부터 배송 완료까지의 라이프사이클을 관리하는 배송 도메인 구현
**왜**: 주문이 결제 완료로 종료되는 현상을 방지하고, 사용자가 배송 추적이 가능하도록 함
**As-Is → To-Be**:
- As-Is: 결제 승인(PaymentApprovedEvent) 후 배송 관련 기능이 없어 주문 라이프사이클이 PAID에서 멈춤
- To-Be: 결제 완료 시 자동으로 배송 준비(READY)가 시작되고, Mock Carrier가 1분 간격으로 상태를 전이시켜 배송 완료(DELIVERED)까지 도달

배송 상태: READY(준비) → INVOICE(운송장 발급) → SHIPPING(배송중) → DELIVERED(완료). Carrier API 실패 시 delivery_retry_queue에 기록되고 관리자가 수동 재처리합니다. DeliveryCompletedEvent는 향후 OLV-110에서 구독하여 포인트 적립/리뷰 작성 가능 처리에 사용합니다.

## Plan Candidates

### Option 1: ShedLock 기반 스케줄러 (추천)
- **장점**: 멀티 인스턴스 배포에서 안전한 실행 중복 방지, 별도 DB 테이블로 락 관리
- **단점**: 의존성 추가 필요 (net.javacrumbs.shedlock)
- **복잡도**: 중간
- **적용 사례**: Spring Batch 대용어로 많이 사용

### Option 2: DB 락 기반 스케줄러 (SELECT FOR UPDATE SKIP LOCKED)
- **장점**: 별도 의존성 없음, 이미 Postgres 사용 중
- **단점**: 직접 구현 필요, 락 관리 로직이 복잡해질 수 있음
- **복잡도**: 높음
- **적용 사례**: Outbox worker 패턴 (이미 프로젝트에 존재)

### Option 3: 단일 인스턴스 가정 (profile 분리)
- **장점**: 구현이 가장 간단
- **단점**: 멀티 인스턴스 배포 시 중복 실행 위험
- **복잡도**: 낮음
- **적용 사례**: 초기 프로토타입

## Recommendation

**Option 1 선택**: ShedLock 기반 스케줄러와 MockCarrierClient 분리

**이유**:
1. 프로젝트가 이미 모듈형 모놀리스로 확장 가능성을 고려하고 있음 (wiki/00-architecture-overview)
2. 힌트에서 ShedLock 언급 ("multi-instance deploys")
3. @ConditionalOnProperty 패턴이 이미 PgClient에서 사용되어 팀에 익숙함
4. 테스트에서 ShedLock을 비활성화하고 Mock 즉시 모드 사용 가능

**첫 번째 실패 테스트**:
```java
@Test
@DisplayName("PaymentApprovedEvent 수신 시 배송 준비 상태로 Delivery 생성")
void whenPaymentApproved_thenCreatesDeliveryWithReadyStatus() {
    // given
    Long orderId = 1L;
    PaymentApprovedEvent event = new PaymentApprovedEvent(
        this, orderId, "ORD001", 1L, "pay-key", BigDecimal.valueOf(50000)
    );

    // when
    deliveryService.prepareForOrder(orderId);

    // then
    List<Delivery> deliveries = deliveryRepository.findByOrderId(orderId);
    assertThat(deliveries).hasSize(1);
    assertThat(deliveries.get(0).getStatus()).isEqualTo(DeliveryStatus.READY);
}
```

## As-Is -> To-Be Report

### As-Is
- 결제 승인(PaymentApprovedEvent) 후 배송 관련 기능이 없어 주문 라이프사이클이 PAID 상태에서 종료됨
- 사용자가 배송 추적을 할 수 없음

### To-Be
- 결제 완료 시 `DeliveryEventListener`가 `PaymentApprovedEvent`를 수신하여 자동으로 배송 생성 (READY 상태)
- `MockCarrierClient`가 1분 간격으로 배송 상태를 전이시켜 배송 완료까지 도달
- 사용자가 `/api/me/orders/{orderNo}/deliveries` API로 배송 현황 조회 가능
- 관리자가 `/api/admin/deliveries` API로 배송 관리 및 수동 재시도 가능

### Reasoning

- **ShedLock 도입**: 멀티 인스턴스 배포 환경에서 스케줄러 중복 실행 방지
- **MockCarrierClient 분리**: `@ConditionalOnProperty`로 Mock/Real 구현 분리, 테스트에서 즉시 모드 지원
- **상태 전이 머신**: `Delivery` 엔티티에 상태 유효성 검증 로직 포함
- **감사 추적**: 모든 상태 변경이 `delivery_status_histories` 테이블에 기록됨

## Learnings

1. **MockMvc 트랜잭션 격리 문제**: 테스트 메서드와 MockMvc 요청이 다른 트랜잭션 컨텍스트에서 실행될 때 데이터 가시성 문제가 발생할 수 있음. 해결 방법은 `@BeforeEach`에서 `txTemplate.executeWithoutResult`로 데이터를 커밋하고, 테스트에서 실제 저장된 엔티티의 값을 사용함.
2. **orderNo 자동 생성**: orders 테이블에 트리거가 있어 orderNo가 자동으로 생성됨. 테스트에서 설정한 값이 덮어쓰여지므로, 저장 후 `order.getOrderNo()`로 실제 값을 읽어서 사용해야 함.
3. **ShedLock 설정**: `spring.scheduling.enabled=true` 프로필에서만 스케줄러 활성화되도록 설정하여 테스트 환경에서의 중복 실행 방지.
