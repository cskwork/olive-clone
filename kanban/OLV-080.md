## Why

Bridges payment-complete to "사용자 화면에서 배송 추적이 보임". Without
this, the order lifecycle dead-ends at PAID.

## Scope

Schema (`V9__delivery.sql`):

- `deliveries`: id, order_id FK, delivery_address_id FK, carrier_name,
  invoice_no NULLABLE, status (enum from `llm-wiki/80-delivery-domain.md`),
  created_at, updated_at.
- `delivery_status_histories`: id, delivery_id FK, from_status, to_status,
  reason, created_at.
- `delivery_retry_queue`: id, delivery_id, request_kind
  (`ISSUE_INVOICE | FETCH_STATUS`), payload_json, retry_count,
  next_retry_at, last_error, status (`PENDING | DONE | DEAD`).

Service:

- `DeliveryService.prepareForOrder(orderId)` — invoked by
  `PaymentApprovedEvent` subscriber. Creates one `deliveries` row per
  warehouse (MVP: one per order). Status `READY`.
- `CarrierClient` interface + `MockCarrierClient`:
  - `issueInvoice(deliveryId)` — returns a fake `invoice_no`. Mock
    auto-flips status `READY → INVOICE` immediately and `INVOICE →
    SHIPPING` after 1 min (test-overridable).
  - `fetchStatus(invoiceNo)` — returns current status. Mock walks
    `SHIPPING → DELIVERED` after another 1 min.
- Failed carrier calls land in `delivery_retry_queue`.

User endpoint:

- `GET /api/me/orders/{orderNo}/deliveries` — list deliveries with current
  status + invoice_no.

Admin endpoints (require `ORDER_ADMIN`):

- `POST /api/admin/deliveries/{id}/issue-invoice` — manual replay.
- `GET /api/admin/deliveries?status=&page=&size=` — operational view.

## Acceptance Criteria

- [ ] After a successful payment confirm (OLV-072), exactly one
      `deliveries` row exists for the order.
- [ ] Mock carrier walk: READY → INVOICE → SHIPPING → DELIVERED with
      history rows for each transition.
- [ ] Carrier failure: row lands in `delivery_retry_queue`, admin can
      manually replay.
- [ ] `DeliveryCompletedEvent` published when status reaches DELIVERED;
      OLV-110 wires the subscriber.

## Hints

- Use `@Async` + `@Scheduled` (1 min interval) for the mock walk so the
  e2e ticket (OLV-140) can verify the full payment → delivery loop.
- ShedLock for the scheduled walker to avoid duplicate execution on
  multi-instance deploys.
