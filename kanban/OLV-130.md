## Domain Brief

**무엇**: 커머스 백엔드의 Observability(관측 가능성) 구축 — Prometheus 메트릭, 구조화된 JSON 로깅, Docker 이미지 빌드
**왜**: 시스템 상태를 실시간으로 모니터링하여 장애 조기 탐지 및 성능 병목 파악
**As-Is → To-Be**:
- As-Is: Actuator는 있으나 Prometheus endpoint 비활성화, 도메인 메트릭 없음, Dockerfile 없음
- To-Be: `/actuator/prometheus`에서 모든 메트릭 노출, Grafana 대시보드로 시각화, Docker 이미지로 컨테이너 배포 가능

### 기존 상태 확인

1. **Spring Boot Actuator**: `spring-boot-starter-actuator` 의존성 존재 (`build.gradle.kts:40`)
2. **Logstash JSON 로깅**: `logstash-logback-encoder` 이미 추가됨 (OLV-004), `logback-spring.xml`에 `AUDIT_FILE` appender 설정됨
3. **메트릭 부재**: Micrometer Prometheus 모듈 없음 → `/actuator/prometheus` endpoint 비활성
4. **Dockerfile 없음**: 애플리케이션 컨테이너화 미지원

### 구현 범위 (PRD §16)

**메트릭 (Micrometer + Prometheus)**:
- 내장: `http_server_requests_seconds` (Tomcat/JVM metrics 자동 노출)
- 커스텀: 주문 생성/취소, 결제 시도/승인/실패, 재고 예약 실패, 검색 지연/빈결과, 캐시 적중률, DB 풀, Outbox 대기수

**로깅**: `olive.audit` logger가 JSON 포맷으로 `audit-YYYY-MM-DD.log`에 기록됨 (OLV-004 완료)

**인프라**:
- Dockerfile: 멀티 스테이지 (jdk21-alpine → jre21-alpine)
- docker-compose: Prometheus + Grafana 추가
- Alert 규칙: PRD §16.3 예시 (결제 실패율 10% 이상 시 알림)

## Plan Candidates

### Option 1: Micrometer + Prometheus (선택)
- **장점**: Spring Boot Actuator와 완벽 통합, Prometheus 생태계 방대, Grafana 대시보드 풍부
- **단점**: 별도 스크래핑 인프라 필요
- **구현**: `micrometer-registry-prometheus` 의존성 추가, application.yml 설정

### Option 2: OpenTelemetry + Prometheus
- **장점**: 표준화된 추적/메트릭/로그 통합, 벤더 중립
- **단점**: 설정 복잡, Spring Boot 3.x 아직 초기 지원
- **구현**: `opentelemetry-spring-boot-starter` 추가

### Option 3: CloudWatch Metrics (AWS)
- **장점**: AWS 네이티브, 별도 인프라 불필요
- **단점**: 벤더 락인, 비용, 로컬 개발 환경 불리
- **구현**: `spring-cloud-starter-aws-metrics` 추가

## Recommendation

**Option 1 (Micrometer + Prometheus)** 채택.

**이유**:
1. Spring Boot Actuator와 자동 통합 (의존성 하나 추가면 끝)
2. docker-compose로 로컬 개발 환경에서 완전 재현 가능
3. Prometheus/Grafana는 업계 표준 오픈소스 스택
4. 향후 Kubernetes 이주 시 서비스 디스커버리(Prometheus Operator) 활용 가능

**첫 번째 실패 테스트**:
```

    given().port(8080)
        .when().get("/actuator/prometheus")

}
```

## Learnings

1. **Micrometer 동적 태그 처리**: `Counter.builder().tag()`는 매번 새 인스턴스를 생성하므로, `MeterRegistry.counter()`를 직접 호출하는 것이 더 효율적임
2. **Spring Boot Actuator 자동 설정**: `management.endpoints.web.exposure.include=prometheus`만 추가하면 Micrometer가 자동으로 Prometheus endpoint를 노출함
3. **Grafana 프로비저닝**: `datasources/`와 `dashboards/` YAML 파일로 컨테이너 시작 시 자동 설정 가능 → 수동 설정 불필요
4. **HikariCP 메트릭**: Spring Boot 3.x는 `hikaricp_connections_*` 메트릭을 자동으로 노출하므로 별도 Gauge 불필요

## As-Is -> To-Be Report

### As-Is
- `/actuator/prometheus` endpoint 존재하지 않음 (404)
- 도메인 메트릭 없음 (주문 생성, 결제, 재고, 검색)
- Dockerfile 없음 (컨테이너 이미지 빌드 불가)
- Prometheus + Grafana 인프라 없음

### To-Be
- `/actuator/prometheus`에서 모든 메트릭 노출:
  - `http_server_requests_seconds` (내장)
  - `commerce_orders_created_total`, `commerce_payments_total`, `commerce_inventory_reservation_failures_total`, `commerce_orders_canceled_total`, `commerce_search_request_seconds`, `commerce_search_empty_result_total` (커스텀)
  - `cache_redis_hit_ratio`, `outbox_pending_count`, `hikaricp_connections_active` (Gauge)
- `docker build .`로 실행 가능한 이미지 생성
- `docker compose up -d prometheus grafana`로 모니터링 스택 실행
- Grafana 대시보드 자동 프로비저닝 (http://localhost:3000)
- Alert 규칙 적용 (`infra/prometheus/alerts.yml`)

### Reasoning
- **Micrometer + Prometheus 선택**: Spring Boot Actuator와 완벽 통합, docker-compose로 로컬 개발 환경 재현 가능
- **이벤트 리스너 방식**: 도메인 코드와 메트릭 수집 로직 분리 → 결합도 최소화
- **Grafana 프로비저닝**: `datasources/`, `dashboards/` YAML로 컨테이너 시작 시 자동 설정 → 수동 설정 불필요
- **멀티 스테이지 Dockerfile**: 빌드 스테이지와 런타임 스테이지 분리 → 최종 이미지 크기 최소화

### Trade-offs
- **선택**: Prometheus vs OpenTelemetry → Prometheus는 더 성숙하고 Spring Boot 통합이 용이
- **선택**: 이벤트 리스너 vs 직접 호출 → 이벤트 리스너는 비동기 처리 가능하지만 실패 시 메트릭 소실 가능성 (수락: 메트릭은 best-effort)

## Why

PRD §16 names the exact metrics, logs, and alert thresholds. Wiring them
once gives every later ticket a place to add new gauges without
re-architecting.

## Scope

Metrics (Micrometer + Prometheus):

- All required indicators from PRD §16.1 exposed at `/actuator/prometheus`:
  - `http_server_requests_seconds` (built-in) — covers API 응답 시간 / 오류율.
  - `commerce_orders_created_total` (counter, tags: status).
  - `commerce_payments_total` (counter, tags: status, pg_provider).
  - `commerce_inventory_reservation_failures_total` (counter, tags:
    optionId).
  - `commerce_orders_canceled_total` (counter).
  - `commerce_search_request_seconds` (timer).
  - `commerce_search_empty_result_total` (counter).
  - `cache_redis_hit_ratio` (gauge).
  - `db_pool_active` (gauge from HikariCP).
  - `outbox_pending_count` (gauge).

Logging (logstash-encoder JSON layout already wired in OLV-004):

- All categories from PRD §16.2 emit structured logs with consistent
  fields (`event=login_failure`, `member_id=...`, `request_id=...`).

Dockerfile + docker-compose addition for Prometheus:

- App `Dockerfile` (multi-stage build: jdk21-alpine).
- Add Prometheus + Grafana to `docker-compose.yml` with a basic
  `prometheus.yml` scraping `/actuator/prometheus` and a Grafana
  pre-provisioned dashboard JSON checked into `infra/grafana/`.
