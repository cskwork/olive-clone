## Why

PRD §13: DB LIKE works at MVP scale, but search quality and latency stop
holding once catalog grows. Build the OpenSearch index + the sync pipeline
NOW so we don't have to dual-write later.

## Scope

- Index `products` with the document shape from PRD §13.2 / `llm-wiki/95-search-domain.md`.
  Mapping: `productName` (text, Korean analyzer when available, else
  standard), `tags` (keyword), `categoryNames` (keyword), `brandName`
  (keyword), `salePrice` (long), `rating` (float), `salesCount` (long),
  `reviewCount` (long), `status` (keyword).
- Initial bulk import job: `./gradlew reindexProducts` (a Spring command
  bean) walks `products` and bulk-indexes them.
- Outbox sync:
  - On `ProductUpdatedEvent` (from OLV-022) → enqueue an `outbox_events`
    row of type `PRODUCT_INDEX_SYNC` with the productId.
  - A `@Scheduled(fixedDelay = 1000)` worker drains the outbox in batches
    of 100, bulk-indexes via OpenSearch, marks rows processed.
  - Failure handling: increment `attempt_count`; after 5 failures, move
    to `dlq=true`. Admin endpoint `POST /api/admin/search/reindex/{productId}`
    for manual replay.

## Acceptance Criteria

- [ ] `./gradlew reindexProducts` indexes the seed catalog; OpenSearch
      shows the expected count.
- [ ] Updating a product price emits a sync event; within 5 seconds the
      OpenSearch document reflects the new price.
- [ ] Stopping OpenSearch keeps the outbox draining attempts and never
      crashes the app; restart drains successfully.

## Hints

- Use the OpenSearch Java client's `BulkRequest`. Batch by 100 documents.
- Choose `nori` analyzer if available in the local image; otherwise
  document the limitation as a follow-up.
