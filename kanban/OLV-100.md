## Domain Brief

**무엇**: 상품 검색용 OpenSearch 인덱스를 만들고, 상품이 바뀔 때마다 outbox 큐를 통해 OpenSearch 문서를 자동 업데이트하는 파이프라인.
**왜**: DB LIKE 검색은 카탈로그가 커지면 응답 속도와 검색 품질이 무너짐. 나중에 옮기면 dual-write 부담이 커지므로 지금 인프라를 깐다.
**As-Is → To-Be**:
- As-Is: `OpenSearchClient` 빈만 떠 있고 `products` 인덱스/sync 경로는 없음. `ProductUpdatedEvent`는 캐시 무효화에만 쓰임.
- To-Be: 부팅 시 `products` 인덱스 자동 생성, admin 변경 → outbox row → 1초 폴링 워커가 bulk index, `./gradlew reindexProducts`로 전체 재색인.

핵심 재사용/누락:
- 재사용: `OpenSearchConfig.openSearchClient`(`common/config/OpenSearchConfig.java:27`), `ProductUpdatedEvent`(`product/ProductUpdatedEvent.java:11`), `PostgresIntegrationSupport`(`common/persistence/PostgresIntegrationSupport.java:25`), `OpensearchContainer` 테스트 패턴(`common/config/OpenSearchIntegrationTest.java:29`).
- 누락: `outbox_events` 테이블(`wiki/96-eventing.md §Decision log`만 존재, V8 Flyway 신규), `OutboxEvent`/`OutboxEventRepository`, `ProductDocument` 페이로드, `SearchIndexInitializer`(매핑 부트스트랩), `ProductIndexer`(DB→bulk), `OutboxIndexerWorker`(@Scheduled), `ReindexProductsCommand`(`ApplicationRunner` + Gradle task), `SearchAdminController`(reindex endpoint), `ErrorCode.SEARCH_REINDEX_*`.
- 함정: `Product`는 options+images 두 컬렉션 → MultipleBagFetchException 회피용 brand-only fetch. nori 분석기 미포함(`opensearchproject/opensearch:2.15.0`), `standard` fallback + follow-up 명시.
_세부: docs/OLV-100/explore/details.md_

## Plan Candidates

A. **단일 outbox + 1초 폴링 (티켓 권고)** — V8 outbox 테이블 + `@Scheduled(fixedDelay=1000)` 워커. 100건 배치 bulk. 단순/검증된 패턴, wiki 96 정의 그대로.
B. **트랜잭션 직후 동기 인덱싱** — `@TransactionalEventListener(AFTER_COMMIT)`에서 직접 OpenSearchClient 호출. 즉시성 ↑ but OS 장애 시 이벤트 손실, AC3(다운 시 영구 큐)와 충돌.
C. **외부 CDC (Debezium/Kafka Connect)** — 가장 견고. 인프라/운영 비용 과대, MVP scope 초과 (wiki 96 §Decision log: "swap for Debezium when traffic warrants").

## Recommendation

**A 채택.** 이유: (1) wiki 96-eventing §Producer pattern + 95-search-domain §Decision log와 정확히 일치, (2) AC3("OpenSearch 다운에도 앱이 죽지 않고 outbox에 attempt 누적, 재기동 시 드레인")를 자연스럽게 충족, (3) Kafka/Debezium 대비 의존성 0 추가.
구현 청크: (1) V8 outbox 마이그레이션 → (2) `OutboxEvent` 엔티티/리포지토리 → (3) `ProductDocument` + 인덱스 매핑 부트스트랩 → (4) `ProductIndexer.indexBulk` + outbox 워커 → (5) `ProductAdminService` 4곳 outbox enqueue 추가 → (6) `reindexProducts` Gradle task + `ReindexProductsCommand` → (7) `SearchAdminController.reindex` 엔드포인트.
**첫 실패 테스트**: `ProductSearchSyncIT#updatingProductPrice_emitsOutboxRow_and_indexerReflectsInOpenSearch` — admin update 호출 → 5초 안에 OpenSearch `products/_doc/{id}.salePrice == 새 가격` 검증.

## Learnings

**무엇**: 구현 중 깨달은 invariant와 explore 단계 가정의 어긋남을 정리.
**왜**: 다음 ticket이 같은 함정에 빠지지 않도록.
**As-Is → To-Be**:
- As-Is: 가정 검증 없이 패턴만 차용.
- To-Be: 운영 안전성 함정 2가지가 wiki에 기록되어 재발 방지.

- **OpenSearch client는 명시적 타임아웃 없이는 hang.** `pause`된 노드를 `bulk()`로
  치면 Apache HttpClient 4의 default("무제한")가 워커 스레드를 영구히 막아 fixedDelay
  폴링이 무의미해짐. 본 작업에서 connect=2s, socket=3s를 박음 — `OpenSearchConfig.java:30-33`.
- **`@EnableScheduling` + JVM-싱글톤 Postgres = 테스트 간섭.** 다른 SpringBootTest
  컨텍스트의 워커가 PENDING row를 가로채 OLV-100 IT가 비결정적으로 실패. `SchedulingConfig
  @Profile("!test")`로 분리하고 IT는 `worker.drainOnce()`를 수동 호출 — 결정론적이고 빠름.
- **`Product`의 두 컬렉션(options/images) 동시 fetch 불가** — `MultipleBagFetchException`.
  인덱서는 brand만 fetch + 카테고리는 단일 native 쿼리로 분리해 N+1 + bag 충돌 회피.
- **31개 기존 회귀**: `:param IS NULL OR ...` JPA 패턴은 Hibernate 6에서 NULL 타입 추론
  실패. 별도 ticket 후보 (BrandRepository / ProductRepository / CategoryRepository 등).

## Why

PRD §13: DB LIKE works at MVP scale, but search quality and latency stop
holding once catalog grows. Build the OpenSearch index + the sync pipeline
NOW so we don't have to dual-write later.

## Scope

- Index `products` with the document shape from PRD §13.2 / `llm-wiki/95-search-domain.md`.
  Mapping: `productName` (text, Korean analyzer when available, else
  standard), `tags` (keyword), `categoryNames` (keyword), `brandName`
  (keyword), `salePrice` (long), `rating` (float), `salesCount` (long),
  `reviewCount` (long), `status` (keyword).
- Initial bulk import job: `./gradlew reindexProducts` (a Spring command
  bean) walks `products` and bulk-indexes them.
- Outbox sync:
  - On `ProductUpdatedEvent` (from OLV-022) → enqueue an `outbox_events`
    row of type `PRODUCT_INDEX_SYNC` with the productId.
  - A `@Scheduled(fixedDelay = 1000)` worker drains the outbox in batches
    of 100, bulk-indexes via OpenSearch, marks rows processed.
  - Failure handling: increment `attempt_count`; after 5 failures, move
    to `dlq=true`. Admin endpoint `POST /api/admin/search/reindex/{productId}`
    for manual replay.

## Acceptance Criteria

- [ ] `./gradlew reindexProducts` indexes the seed catalog; OpenSearch
      shows the expected count.
- [ ] Updating a product price emits a sync event; within 5 seconds the
      OpenSearch document reflects the new price.
- [ ] Stopping OpenSearch keeps the outbox draining attempts and never
      crashes the app; restart drains successfully.

## Hints

- Use the OpenSearch Java client's `BulkRequest`. Batch by 100 documents.
- Choose `nori` analyzer if available in the local image; otherwise
  document the limitation as a follow-up.
