## Why

Same rationale as OLV-051: order creation needs a stable point service
contract. Point math is also where audit teams will look first — keep the
ledger pure.

## Scope

Service:

- `PointService.earnScheduled(memberId, amount, orderId, availableAt,
  expiresAt)` — writes EARN row with future `available_at`. The point is
  not spendable until then.
- `PointService.use(memberId, amount, orderId)` — atomically validates
  spendable balance ≥ amount, writes USE row.
- `PointService.cancel(memberId, orderId)` — for cancelled orders, writes
  CANCEL row that reverses both EARN and USE rows tied to that order.
- `PointService.expire(memberId, asOf)` — writes EXPIRE rows for any
  unused EARN with `expires_at <= asOf`.
- `PointService.spendableBalance(memberId, asOf)` — sum of EARN+CANCEL
  amounts where `available_at <= asOf < (expires_at OR ∞)`, minus
  USE+EXPIRE amounts in the same window.

Endpoints:

- `GET /api/me/points` — current spendable balance + 30-day-pending list.
- `GET /api/me/points/history?page=&size=` — paged history with reasons.

## Acceptance Criteria

- [ ] Earn 1000 with `available_at = now+30d`. Spendable balance = 0
      until 30d clock passes (test by injecting an as-of timestamp).
- [ ] Use 500 against a 700 spendable balance succeeds; 800 returns 422.
- [ ] Cancel an order that earned 1000 + used 200 → both rows reversed,
      net effect 0.
- [ ] `points.balance` rollup column matches the recompute from history
      after every test (run a parity check at teardown).
- [ ] Concurrent uses on a 100-balance account: 10 threads each try to
      use 20. Exactly 5 succeed.

## Hints

- For concurrent use, lock on `SELECT * FROM points WHERE member_id=? FOR UPDATE`
  inside the txn — keeps the rollup column consistent.
- Don't expose the rollup column externally; always derive
  `spendableBalance` from the history view to avoid stale-rollup bugs.

**As-Is → To-Be**:

**핵심 발견**:
1. **스키마**: `point_histories`가 source of truth, `points.balance`는 트리거로 동기화되는 캐시
2. **change_type enum**: EARN(적립), USE(사용), CANCEL(취소), EXPIRE(소멸), ADMIN_ADJUST(관리자 조정)
3. **동시성 제어**: use()에서 `SELECT FOR UPDATE`로 points 행 잠금
4. **spendable 계산**: SUM(amount) WHERE change_type IN (EARN, CANCEL, ADMIN_ADJUST) AND available_at <= asOf AND (expires_at IS NULL OR expires_at > asOf) - SUM(amount) WHERE change_type IN (USE, EXPIRE) AND 동일 윈도우
5. **OLV-051 패턴**: 단일 Service + User 컨트롤러, sealed interface DTO

## Learnings

**학습 1**: `SELECT FOR UPDATE` 동시성 제어는 DB 락을 사용하므로 트랜잭션 내에서 유효합니다. 테스트에서는 `@Transactional`이 있어야 락이 작동합니다.

**학습 2**: 포인트 사용 내역(USE)은 현재 시간에 생성되지만, 적립(EARN)은 미래 시간(`available_at`)을 가질 수 있습니다. 잔액 조회 시 `asOf` 파라미터로 기준 시점을 명확히 해야 합니다.

**학습 3**: Pending 포인트 조회는 `asOf < available_at <= cutoff` 범위를 찾아야 합니다. 단순 `available_at > cutoff`는 cutoff 이후의 포인트만 반환하여 빈 결과가 나옵니다.
