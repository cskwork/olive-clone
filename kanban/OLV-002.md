## Why

Every domain ticket from OLV-010 onward needs Postgres available locally for
both runtime and Testcontainers. Flyway is mandatory for the
"migrations are append-only" Hard rule.

## Scope

- `docker-compose.yml` at project root with services: `postgres:16-alpine`
  (port 5432, db `commerce`, user `commerce`, password `commerce`).
- `application.yml` profile `local` pointing at it; `application.yml`
  profile `test` configured for Testcontainers (resolved by
  `org.testcontainers:postgresql` in test scope).
- Flyway baseline migration `V1__init_baseline.sql` (empty / placeholder
  comment — domain tables ship in their own tickets).
- Flyway plugin in Gradle wired so `./gradlew flywayInfo` works.
- Re-enable JPA auto-config that OLV-001 disabled.

## Acceptance Criteria

- [ ] `docker compose up -d postgres` brings the DB up; `psql` connects.
- [ ] `./gradlew bootRun --args='--spring.profiles.active=local'` boots
      without DB errors and Flyway logs `Successfully applied 1 migration`.
- [ ] `./gradlew flywayInfo` lists `V1__init_baseline` as applied.
- [ ] A new `RepositoryIntegrationTest` (Testcontainers) runs against a
      throwaway Postgres and passes.

## Hints

- Use Spring Boot's built-in Flyway autoconfig — set
  `spring.flyway.locations=classpath:db/migration`.
- For Testcontainers, prefer the `@ServiceConnection` annotation (Spring
  Boot 3.1+) for zero-config wiring.
- Add the `Dockerfile` for the app in OLV-130 (observability), not here.

## Domain Brief

**무엇**: 로컬 개발(`docker compose`)과 통합 테스트(Testcontainers)에서 PostgreSQL이 항상 동일한 스키마로 떠 있도록, Flyway 마이그레이션 골격과 JPA 자동설정을 처음으로 켠다.
**왜**: OLV-010 이후 모든 도메인 티켓이 Postgres + JPA + Flyway 위에서 동작 — 이 셋이 깔리지 않으면 후속 티켓이 매번 자기 환경을 새로 설정해야 하고, "migrations are append-only" Hard rule을 강제할 baseline V1이 없으면 V2/V3가 어디 위에 쌓이는지 불명확해진다.
**As-Is → To-Be**:
- As-Is: `application.yml`이 `DataSourceAutoConfiguration`/`HibernateJpaAutoConfiguration` exclude. 로컬 Postgres 0개, Flyway 0개, Testcontainers 미연결.
- To-Be: `docker compose up -d postgres`로 5432 노출, `bootRun --args=local`이 V1 1건 적용 후 정상 기동, `./gradlew flywayInfo`가 V1을 `Success`로 표시, `RepositoryIntegrationTest`가 throwaway PG에서 통과.

- 인바리언트(`llm-wiki/00-architecture-overview.md:11`): **One Postgres schema** — Flyway 위치는 단일 `classpath:db/migration/`로 통일.
- OLV-001에서 `spring-boot-starter-data-jpa`는 의존성에는 포함되어 있으므로(`build.gradle.kts:22`), 본 티켓은 (1) `org.postgresql:postgresql` 런타임 추가, (2) `flyway-core` + `flyway-database-postgresql` 추가(Flyway 10.x는 PG dialect가 분리됨), (3) Testcontainers BOM + postgres + spring-boot-testcontainers 추가, (4) `application.yml`의 두 줄 exclude 제거의 단순 합.
- `@ServiceConnection`(Spring Boot 3.1+)을 쓰면 `@DataJpaTest` + `PostgreSQLContainer`만으로 `spring.datasource.*`가 자동 wiring — 별도 `@DynamicPropertySource` 불필요.
- 위험 R1: `flyway-core`만 넣고 `flyway-database-postgresql`을 빠뜨리면 부팅 시 `Unsupported Database: PostgreSQL`로 실패 — 두 의존성 모두 명시 필요.
- 위험 R2: 로컬/테스트 설정이 한 파일에 섞이면 우선순위 충돌 — `application.yml`(공통) + `application-local.yml`(로컬 DB만) 분리, 테스트는 `@ServiceConnection`으로 동적 주입.

## Plan Candidates

- **A. Spring Boot Flyway autoconfig + `@ServiceConnection` Testcontainers (티켓 Hints 라인)**: 의존성 추가 + `application-local.yml` + `V1__init_baseline.sql` + `flyway` Gradle plugin 블록 + `RepositoryIntegrationTest`(@DataJpaTest + @ServiceConnection). 부팅 시 Flyway 자동 실행, 테스트는 throwaway PG. **권장**.
- **B. Testcontainers JDBC URL 모드(`jdbc:tc:postgresql:16:///commerce`)**: `application-test.yml`에 특수 JDBC URL만 박으면 자동으로 컨테이너가 뜨는 매직 모드. 장점: 코드 0줄. 단점: `@ServiceConnection` 대비 Spring Boot 3.x 권장 경로에서 벗어나며, Flyway 통합/디버깅이 더 불투명. 티켓 Hints의 `@ServiceConnection` 명시와도 충돌.
- **C. Embedded Postgres(zonkyio/embedded-postgres) + 별도 dev-loader**: Docker 미보유 환경 친화적. 장점: 호스트에 Docker 불필요. 단점: PRD §18 잠금 스택을 벗어남(Testcontainers가 표준), 바이너리 다운로드 캐시·플랫폼별 호환성 이슈, 향후 Redis/OpenSearch 통합 테스트와 일관성 깨짐.

## Recommendation

**Option A 채택.** 티켓 §Hints가 `@ServiceConnection` + `spring.flyway.locations=classpath:db/migration`을 직접 명시했고, 이것이 Spring Boot 3.x의 권장 경로 + 후속 티켓(Redis/OpenSearch도 같은 패턴)과의 일관성을 보장한다. B는 매직성이 강해 디버깅 부담이 크고, C는 PRD §18 스택 잠금을 위반.
**첫 실패 테스트**: `com.olive.commerce.common.persistence.RepositoryIntegrationTest` — `@DataJpaTest` + `@Testcontainers` + `PostgreSQLContainer<?>` (`@ServiceConnection`). `entityManager`가 `SELECT 1`을 PostgreSQL dialect로 실행하고 Flyway가 V1을 적용했는지(`flyway_schema_history`에 row 1건) assert. 의존성/yml/마이그레이션 파일이 없으면 RED, 갖추면 GREEN.
