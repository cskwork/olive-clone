## Why

This is the most-trafficked read endpoint in the system (PRD §11.1
caches main page lists). Wiring cache-aside here sets the pattern for the
category, brand, and search caches that come later.

## Scope

Endpoints under `/api/products`:

- `GET /api/products?categoryId=&brandId=&sort=&page=&size=` — paginated
  list. `sort` ∈ `popular | latest | price_asc | price_desc | rating`.
  Returns the shape from PRD §8.1 (productId, brandName, productName,
  salePrice, originalPrice, discountRate, thumbnailUrl, rating,
  reviewCount).
- `GET /api/products/{productId}` — full detail: product, all options
  (with current `available_quantity` joined from inventory — but tolerate
  inventory rows missing during the bootstrap window), images, category
  path, brand summary, aggregate rating + review_count.

Caching (cache-aside per PRD §11.2):

- Detail: `cache:product:detail:{productId}`, TTL 60s, invalidated on
  `ProductUpdatedEvent`.
- List by `(categoryId, sort, page, size)`: TTL 30s, invalidated by a
  versioned key (bump a counter in Redis on any product write so existing
  list keys become unreachable without enumerating them).

## Acceptance Criteria

- [ ] First call to `/api/products/{id}` populates Redis; second call
      returns sub-10ms latency in a Testcontainers smoke.
- [ ] `PATCH /api/admin/products/{id}` invalidates the cached detail and
      bumps the list version within one transaction-after-commit hook.
- [ ] HIDDEN / STOPPED / DRAFT products are excluded from the public list.
- [ ] Sort by `rating` falls back gracefully when review_count is 0 (use
      `(rating, salesCount)` as composite tiebreaker).

## Hints

- For the list response, project directly into a DTO with QueryDSL or a
  JPQL `SELECT new ...` to avoid N+1.
- Don't cache the inventory in Redis here — leave that to OLV-031, which
  has its own freshness requirements.

## Domain Brief

**무엇**: 공개 상품 목록/상세 API + Redis cache-aside 패턴 구현

**왜**: 시스템에서 가장 트래픽이 높은 읽기 엔드포인트. 여기서 cache-aside 패턴을 확립하면 category/brand/search 캐시가 동일 패턴을 따름

**As-Is → To-Be**:
- As-Is: `GET /api/products` 없음. Admin API만 존재 (OLV-022). 캐싱 없음
- To-Be: 공개 API + Detail cache(TTL 60s) + List versioned cache(TTL 30s). Event 기반 무효화

**핵심 제약**:
- HIDDEN/STOPPED/DRAFT 상품 제외 (AC3)
- Inventory 미구현(OLV-031) → `available_quantity`는 NULL 허용
- Rating/review_count 컬럼 없음 → 정렬 fallback만 구현

## Plan Candidates

**옵션 1: Category 모방 (직접 cache 무효화)**
- `ProductAdminService`에서 `redisTemplate.delete()` 직접 호출
- 장점: 단순. 단점: 강결합, eventing 원칙 위반

**옵션 2: Spring Events + @EventListener (권장)**
- `ProductUpdatedEvent` 발행 → `@TransactionalEventListener(phase=AFTER_COMMIT)` 수신
- 장점: PRD §96 준수, decoupling, transaction-after-commit 보장
- 단점: Event record 1개 추가 (코드량 미미)

**옵션 3: Outbox 패턴**
- 미보류 (OLV-096 eventing 티켓에서)

_세부: docs/OLV-023/explore/plan-candidates.md_

## Recommendation

**선택**: 옵션 2 (Spring Events + @EventListener)

**이유**:
1. PRD §96 eventing 원칙 준수 (category/brand/search와 통일 패턴)
2. AC2 "transaction-after-commit"을 Spring이 보장
3. 향후 Search index sync listener 추가 시 확장 가능

**첫 번째 실패 테스트** (`ProductPublicApiIT.java`):
```java
@Test
void getProductDetail_firstCallMissesSecondCallHitsCache() {
    // Given: Flyway seeds product id=1
    // When: First GET → cache miss, DB 조회, Redis 저장
    mockMvc.perform(get("/api/products/1"))
        .andExpect(status().isOk());

    // Then: Redis에 캐시됨
    String cached = redisTemplate.opsForValue().get("cache:product:detail:1");
    assertThat(cached).isNotNull();

    // When: Second GET → cache hit (<10ms in prod, here just verify 200)
    mockMvc.perform(get("/api/products/1"))
        .andExpect(status().isOk());
}
```
