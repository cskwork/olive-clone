## Why

Inventory is the most concurrency-sensitive domain. Land the schema with
the right constraints (`available_quantity` derived, history append-only,
reservation expiry-indexed) before the service layer goes near it.

## Scope

Create `V4__inventory.sql` matching PRD §6.7, §7.4:

- `inventories`: id, product_option_id BIGINT UNIQUE NOT NULL FK,
  total_quantity INT NOT NULL, reserved_quantity INT NOT NULL DEFAULT 0,
  available_quantity INT GENERATED ALWAYS AS (total_quantity -
  reserved_quantity) STORED, updated_at.
  - CHECK constraint: `total_quantity >= 0 AND reserved_quantity >= 0`.
  - CHECK constraint: `total_quantity >= reserved_quantity` (no negative
    available).
- `inventory_histories`: id, product_option_id, change_type
  (`STOCK_IN | STOCK_OUT | RESERVE | COMMIT | RELEASE | ADMIN_ADJUST`),
  quantity_delta INT, reason VARCHAR(255), order_id BIGINT NULLABLE,
  created_at, created_by BIGINT NULLABLE.
- `inventory_reservations`: id, order_id, product_option_id, quantity,
  status (`HELD | COMMITTED | RELEASED`), expires_at, created_at,
  finalized_at NULLABLE.
  - Index on (status, expires_at) for the batch scan.

## Acceptance Criteria

- [ ] V4 applied with no errors; `inventories.available_quantity` is
      a generated column and updates automatically when total/reserved
      change.
- [ ] Inserting a reservation row with the same `(order_id,
      product_option_id)` twice violates a UNIQUE index — add it.
- [ ] Repository test: stock 100, reserve 30 → available_quantity = 70;
      commit 30 → total 70, reserved 0, available 70.

## Hints

- Postgres supports `GENERATED ALWAYS AS ... STORED` since 12. Use it —
  removes a class of bugs where service code forgets to recompute available.
- Foreign keys to `product_options` should be `ON DELETE RESTRICT` —
  inventory must outlive an option deletion attempt and force a stock-zero
  workflow first.
