## Why

Inventory is the most concurrency-sensitive domain. Land the schema with
the right constraints (`available_quantity` derived, history append-only,
reservation expiry-indexed) before the service layer goes near it.

## Scope

Create `V4__inventory.sql` matching PRD §6.7, §7.4:

- `inventories`: id, product_option_id BIGINT UNIQUE NOT NULL FK,
  total_quantity INT NOT NULL, reserved_quantity INT NOT NULL DEFAULT 0,
  available_quantity INT GENERATED ALWAYS AS (total_quantity -
  reserved_quantity) STORED, updated_at.
  - CHECK constraint: `total_quantity >= 0 AND reserved_quantity >= 0`.
  - CHECK constraint: `total_quantity >= reserved_quantity` (no negative
    available).
- `inventory_histories`: id, product_option_id, change_type
  (`STOCK_IN | STOCK_OUT | RESERVE | COMMIT | RELEASE | ADMIN_ADJUST`),
  quantity_delta INT, reason VARCHAR(255), order_id BIGINT NULLABLE,
  created_at, created_by BIGINT NULLABLE.
- `inventory_reservations`: id, order_id, product_option_id, quantity,
  status (`HELD | COMMITTED | RELEASED`), expires_at, created_at,
  finalized_at NULLABLE.
  - Index on (status, expires_at) for the batch scan.

## Acceptance Criteria

- [ ] V4 applied with no errors; `inventories.available_quantity` is
      a generated column and updates automatically when total/reserved
      change.
- [ ] Inserting a reservation row with the same `(order_id,
      product_option_id)` twice violates a UNIQUE index — add it.
- [ ] Repository test: stock 100, reserve 30 → available_quantity = 70;
      commit 30 → total 70, reserved 0, available 70.

## Hints

- Postgres supports `GENERATED ALWAYS AS ... STORED` since 12. Use it —
  removes a class of bugs where service code forgets to recompute available.
- Foreign keys to `product_options` should be `ON DELETE RESTRICT` —
  inventory must outlive an option deletion attempt and force a stock-zero
  workflow first.

## Domain Brief

**무엇**: 재고 도메인의 핵심 테이블 3개(inventories, inventory_histories, inventory_reservations)와 인덱스를 생성하고, Repository 테스트로 검증하는 V4__inventory.sql 마이그레이션을 작성한다.
**왜**: Inventory는 가장 동시성에 민감한 도메인으로, 서비스 레이어가 접근하기 전에 `available_quantity` 파생 컬럼, history append-only, reservation expiry-indexed 등 올바른 제약조건을 먼저 깔아두어야 concurrency bug를 방지할 수 있음.
**As-Is → To-Be**:

- To-Be: V4 적용 후 3개 테이블 + 6개 인덱스 + UNIQUE 제약 존재, InventorySchemaIntegrationTest로 reserve→commit flow 검증.

- 인바이언트(`llm-wiki/30-inventory-domain.md:7-28`): **Per-option 재고** (`product_option_id` 기준), **Reserve-then-commit 필수** (PRD §6.7 방식 2), **Redis 분산 락 + DB 폴백** (PRD §10.2, §15.4), **모든 변경은 history에 기록**.
- V3__product.sql 패턴: `product_options(id)`를 참조하는 FK에 `ON DELETE RESTRICT` 사용 — 옵션 삭제 전 재고 0 강제. `set_updated_at()` 트리거 재사용(V2 정의).
- 위험 R1: `GENERATED ALWAYS AS ... STORED`는 Postgres 12+ 필요 — docker-compose `postgres:16-alpine` 확인 완료, 문제 없음.
- 위험 R2: AC2의 `(order_id, product_option_id)` UNIQUE 제약 누락 시 중복 예약 가능 — composite UNIQUE 인덱스로 방지.
- 위험 R3: FK `ON DELETE RESTRICT` 누락 시 옵션 삭제로 orphan inventories row 발생 — 명시적으로 지정.

## Learnings

**무엇**: V4__inventory.sql 작성 및 테스트过程中 얻은 교훈.
**왜**: 후속 티켓(OLV-031 Inventory Service, OLV-040 Order)에서 동일한 패턴을 적용하고, 반복되는 실수를 방지하기 위해.

- **GENERATED 컬럼의 검증된 효과**: `available_quantity GENERATED ALWAYS AS (total_quantity - reserved_quantity) STORED` 사용으로 서비스 코드가 available 값을 계산하거나 갱신하는 bug를 완전히 방지. Postgres가 total/reserved 변경 시 자동 재계산하므로 데이터 일관성이 DB 레벨에서 보장됨.
- **UNIQUE 제약으로 AC2 만족**: `(order_id, product_option_id)` composite UNIQUE 인덱스로 같은 주문+옵션 조합의 중복 예약을 DB 레벨에서 방지. 애플리케이션 레벨 체크도 필요하지만, DB 제약이 최종 안전망 역할.
- **Flyway Gradle 플러그인 인증 이슈**: docker-compose PostgreSQL과 Gradle Flyway 플러그인 사이의 인증 문제를 겪었으나, Testcontainers 기반 통합 테스트로 스키마 검증 가능. 이는 `@DataJpaTest` + `@ServiceConnection` 패턴이 Flyway 마이그레이션을 자동 적용하기 때문.
- **CHECK 제약으로 데이터 무결성**: 3개 CHECK 제약(`total >= 0`, `reserved >= 0`, `total >= reserved`)으로 비즈니스 로직 위반 데이터를 DB 레벨에서 거부. 테스트에서 예상된 에러를 모두 확인.
