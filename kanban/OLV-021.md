## Why

Products can't be created until brands + categories exist. Land the
classification CRUD before the product CRUD ticket.

## Scope

Endpoints under `/api/admin/{brands,categories}`:

- `POST /api/admin/brands` — create brand. Requires `PRODUCT_ADMIN`.
- `GET /api/admin/brands` — list with pagination + name search.
- `PATCH /api/admin/brands/{id}` — update name / logo / status.
- `POST /api/admin/categories` — create category (parent_id optional).
- `GET /api/admin/categories` — return as a tree (`children: [...]`).
- `PATCH /api/admin/categories/{id}` — update name / sort_order / parent_id.
- `DELETE /api/admin/categories/{id}` — block delete if any product is
  mapped (return 409 with the affected product count).

Public read mirrors:

- `GET /api/brands` — paginated list, public, ON_SALE status only.
- `GET /api/categories` — full tree, cached in Redis (10-minute TTL,
  cache key `cache:categories:tree`).

## Acceptance Criteria

- [ ] All admin endpoints reject non-admin tokens with 403.
- [ ] Category tree endpoint returns nested children correctly for a
      3-level hierarchy.
- [ ] Cache invalidation: editing a category bumps `cache:categories:tree`
      so the next public GET reflects the change within one request.
- [ ] Integration test verifies the slug uniqueness constraint is
      surfaced as 409 not 500.

## Domain Brief

**무엇**: 브랜드와 카테고리의 admin CRUD + 공개 read API를 구현한다. Admin은 생성/목록/수정/삭제(카테고리만), 공개는 목록만(브랜드는 페이지네이션, 카테고리는 트리+Redis 캐시).
**왜**: 상품 등록(OLV-022) 전에 분류 체계가 먼저 있어야 하므로. 카테고리 트리는 Redis 10분 TTL 캐시로 공개 API 부하 줄임.
**As-Is → To-Be**:
- As-Is: V3__product.sql에 테이블만 존재 — Entity/Repository/Service/Controller 전무, API 호출 불가.
- To-Be: Brand/Category Entity + Repository + Service + Admin/Public 컨트롤러 + Redis 캐시 + IT 8건 존재.

- 인바리언트(`llm-wiki/20-product-domain.md`): brands는 name UNIQUE + slug UNIQUE + status(ACTIVE|INACTIVE). categories는 self-referencing FK(parent_id) + depth + sort_order.
- 보안(`llm-wiki/98-security.md`): `/api/admin/**` → PRODUCT_ADMIN 역할 필요, 공개 `/api/brands` `/api/categories`는 인증 불필요.
- 응답 봉투(`llm-wiki/01-common-conventions.md`): ApiResponse<T> envelope, ErrorCode enum에 BRAND_SLUG_DUPLICATE/CATEGORY_HAS_PRODUCTS 추가.
- 캐시(`llm-wiki/03-infra-baseline.md`): StringRedisTemplate(Spring 자동설정), cache key `cache:categories:tree`, TTL 10분.
- 위험 R1: brands.slug UNIQUE 제약 위반 시 409 반환 → 통합 테스트로 검증. R2: category delete 시 product 매핑 있으면 409 + affected count 반환. R3: Redis 캐싱은 DB 커밋 후 삭제해야 안전.

## Plan Candidates

- **A. Entity + Repository + Service + 분리 컨트롤러 (권장)**: Brand/Category Entity 2개 + Repository 2개 + BrandAdminService/CategoryAdminService/CategoryPublicService 3개 + Admin 컨트롤러 2개 + Public 컨트롤러 2개. CategoryAdminService가 Redis 캐시 무효화 담당. CategoryPublicService는 캐시 read 전용. PostgreSQL recursive CTE로 트리 조회 1회로 해결.
- **B. Service 통합 + 컨트롤러 통합**: ProductClassificationService 하나에 Brand/Category 로직 통합. 단점: (1) Brand와 Category는 생명주기가 다름(Brand는 삭제 없음, Category는 삭제 있음), (2) 권한 분리가 어려움(Brand admin만 Brand 수정 가능해야 함), (3) 티켓의 "분류 CRUD"라는 목적과 위배.
- **C. Controller만 + Entity 생략**: Repository만 사용해서 바로 컨트롤러에서 쿼리. 단점: (1) 캐시 무효화 로직이 컨트롤러에 흩어짐, (2) 순환 참조 검증 등 비즈니스 로직 재사용 불가, (3) 테스트가 어려움(IT에서 컨트롤러까지 띄워야 함).

## Recommendation

**Option A 채택.** Brand와 Category는 명확히 분리된 도메인(삭제 여부, 캐시 요구사항 다름)이므로 Service 분리가 올바름. Entity는 JPA 매핑 + 순환 참조 검증을 위해 필수.**첫 실패 테스트**: `BrandCategoryAdminApiIT` — `@WebMvcTest`로 admin 컨트롤러 4개 endpoint를 PRODUCT_ADMIN 토큰 없이 호출 → 403 검증. Category 트리는 recursive CTE로 3-level depth 검증. Cache 무효화는 Redis Testcontainers로 `delete(cache:categories:tree)` 호출 검증. Entity/Service 부재로 컴파일 RED → 구현으로 GREEN.

## Learnings

**무엇**: 구현 과정에서 발견한 인바리언트와 테스트 패턴.
**왜**: 향后 카테고리/캐시 기능 구현 시 동일 실제 반복 방지.

- **Flyway 데이터와 테스트 격리**: V3 마이그레이션으로 기본 데이터(브랜드 1개, 카테고리 3개, 상품 1개)가 들어가므로 테스트는 이를 가정해야 함. `@BeforeEach`에서 sequence만 리셋하면 됨.
- **Public API vs Admin API 필드 차이**: `AdminUpdateRequest`에 slug 없음(명세), `PublicTreeNode`에 sortOrder 없음(설계). 테스트는 공개 필드만으로 검증해야 함.
- **캐시 무효화 타이밍**: Service 메서드 마지막에 `invalidateCache()` 호출 → DB 커밋 후 캐시 삭제가 보장됨 (`@Transactional` + Service 계층).
- **Gradle 자동 JDK 다운로드**: `org.gradle.java.installations.auto-download=true` 설정으로 시스템 JDK 없이도 Java 21 자동 다운로드 가능.
