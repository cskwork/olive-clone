## Why

Every member, admin, and order endpoint needs auth applied consistently.
Wiring Spring Security + JWT once means OLV-011 (member signup/login) just
issues tokens, and OLV-022/063/072 (admin product/order/payment endpoints)
just declare `@PreAuthorize`.

## Scope

- Add `spring-boot-starter-security`, `io.jsonwebtoken:jjwt-*` (or
  `nimbus-jose-jwt`) dependencies.
- `SecurityFilterChainConfig` with:
  - JWT bearer token resolver (`OAuth2 Resource Server` config or custom
    filter — pick one and document why).
  - Public endpoints: `/actuator/health`, `/api/auth/**`,
    `/api/products/**`, `/api/search/**`.
  - Authenticated: everything else under `/api/**` requires `USER` or
    higher role.
  - Admin: `/api/admin/**` requires the role hierarchy from
    `llm-wiki/98-security.md` — `PRODUCT_ADMIN`, `ORDER_ADMIN`,
    `CS_MANAGER`, `SUPER_ADMIN` (configure `RoleHierarchy` so
    `SUPER_ADMIN` ⊇ all).
- `JwtTokenProvider` with `issueAccess(memberId, role)` (TTL 30 min) and
  `issueRefresh(memberId)` (TTL 14 days, JTI persisted hashed in
  `member_refresh_tokens` — table created by OLV-011 migration).
- `AuthenticationPrincipal` resolver that exposes
  `record AuthenticatedUser(long memberId, MemberRole role)`.

## Acceptance Criteria

- [x] Calling `/actuator/health` without auth returns 200.
- [x] Calling `/api/cart` without auth returns 401.
- [x] Calling `/api/cart` with a JWT for a `USER` returns 200 (placeholder
      controller stub is fine — the real cart ships in OLV-040).
- [x] Calling `/api/admin/products` (placeholder POST) with a `USER`
      JWT returns 403; with a `PRODUCT_ADMIN` JWT returns 200/201.
- [x] Unit tests for `JwtTokenProvider.issueAccess()` round-trip and
      reject expired tokens.

## Hints

- Sign with RS256 using a keypair generated to `src/main/resources/keys/`
  (gitignored) for dev. Document key generation in `README.md`.
- Use `MethodSecurityExpressionHandler` with the configured
  `RoleHierarchyImpl` so `@PreAuthorize("hasRole('PRODUCT_ADMIN')")`
  passes for SUPER_ADMIN tokens.

## Domain Brief

**무엇**: 모든 API에 인증/인가를 일관되게 적용할 Spring Security + JWT 골격을 한 번에 깐다.
**왜**: 후속 티켓(OLV-011 회원 로그인, OLV-022 상품 admin, OLV-063 주문 admin 등)이 토큰 발급 / `@PreAuthorize` 만으로 끝낼 수 있어야 하기 때문.
**As-Is → To-Be**:
- As-Is: `SecurityConfig`가 `/actuator/health|info` permitAll + 그 외 authenticated 만 처리. JWT 검증/발급 / role hierarchy / 401·403 envelope 부재.
- To-Be: RS256 JWT 발급기(`JwtTokenProvider`)와 Spring Security Resource Server 기반 검증 체인이 켜지고, USER/CS_MANAGER/PRODUCT_ADMIN/ORDER_ADMIN/SUPER_ADMIN 5단계 hierarchy로 `/api/admin/**` 를 보호한다.

- 5개 역할 (`USER`, `CS_MANAGER`, `PRODUCT_ADMIN`, `ORDER_ADMIN`, `SUPER_ADMIN`) — `llm-wiki/10-member-domain.md:13-21`. `SUPER_ADMIN`은 다른 admin 셋을 모두 포함, 이들은 `USER`를 포함.
- access 토큰 TTL 30분, refresh 14일 (`llm-wiki/98-security.md:8-10`).
- `member_refresh_tokens` 테이블은 OLV-011 마이그레이션 — 본 티켓은 *발급기*만, persistence는 미포함.
- 401/403 envelope은 `ApiResponse.failure(...)` 형태(`llm-wiki/01-common-conventions.md`); `GlobalExceptionHandler`는 보안 예외를 rethrow → SecurityFilterChain의 EntryPoint/AccessDeniedHandler가 책임.
- `RequestIdFilter` (`HIGHEST_PRECEDENCE`) 가 SecurityFilterChain보다 먼저 traceId를 MDC에 채우므로 401/403 응답에도 traceId가 들어간다.

## Plan Candidates

- **Option A — Resource Server (권장)**: `spring-boot-starter-oauth2-resource-server` + `nimbus-jose-jwt`로 검증을 라이브러리에 위임, 발급은 동일 nimbus로 직접 작성. claim → `AuthenticatedUser` 변환은 `JwtAuthenticationConverter`. 의존성 +1, 표준 검증(`exp`/`iss`/`aud`)은 자동.
- **Option B — 커스텀 OncePerRequestFilter + jjwt**: 최소 의존, 토큰 처리 경로 가시화. 단점: jose 검증 코드를 직접 작성 → 보안 사각지대 위험, 단위 테스트 부담 큼.
- **Option C — 커스텀 필터 + nimbus**: 두 강점 절충안이지만 결국 검증 코드는 직접 작성 — Option A 대비 이득 거의 없음.

## Recommendation

Option A 채택. Resource Server는 jose 검증을 라이브러리에 맡겨 "401/403 분기 + role hierarchy + EntryPoint envelope"라는 본 티켓의 *Spring Security 통합* 부분에 집중할 수 있다. 발급은 nimbus 한 라이브러리만 쓰므로 의존성도 1개. 첫 실패 테스트: `JwtTokenProviderTest#issueAccess_roundTrip_decodesSameClaims` — `JwtTokenProvider` 부재로 컴파일 RED. 이어 `SecurityFilterChainIT` 가 AC 4건(`/actuator/health` 200, `/api/cart` 401/200, `/api/admin/products` 403/200) 을 MockMvc로 검증.

## Learnings

**무엇**: Explore brief vs 실제 구현 차이 점검.
**왜**: 다음 보안/세션 티켓이 같은 함정을 다시 밟지 않게 wiki 에 흔적을 남긴다.
**As-Is → To-Be**:
- As-Is: brief 는 *Resource Server + Custom 401/403 envelope* 정도까지 예측.
- To-Be: 4 가지 비-자명한 invariant 가 검증되어 wiki 에 기록됨.

- **Spring 6 의 RoleHierarchy 자동 적용**: `RoleHierarchy` 빈만 있으면 URL 매처와 `@PreAuthorize` 둘 다 자동으로 hierarchy 를 따른다. 5.x 처럼 `WebExpressionVoter` 에 직접 주입 불필요.
- **RS256 base64url 마지막 char padding**: 2048-bit 서명의 마지막 base64url char 는 6 비트 중 4 비트가 padding — 위조 테스트는 *중간* char 를 변경해야 안정적.
- **401/403 envelope 의 traceId 보장**: `RequestIdFilter(HIGHEST_PRECEDENCE)` 가 SecurityFilterChain 보다 먼저 MDC 에 채워주므로, EntryPoint/AccessDeniedHandler 가 MDC.get 만으로 traceId 채울 수 있다.
- **Docker Desktop on macOS sock 경로**: `~/.docker/run/docker.sock` 은 CLI proxy 라 server info 비어 있어 Testcontainers 가 거부. 진짜 데몬은 `~/Library/Containers/com.docker.docker/Data/docker.raw.sock`. build.gradle.kts 가 자동 탐지.

### Reasoning
- **Resource Server vs Custom Filter**: jose 검증을 라이브러리에 위임해 보안 사각지대를 줄임. 발급은 nimbus 한 라이브러리에서 직접 — 의존성 1개, 라이브러리 분산 없음.
- **RS256 vs HS256**: 운영의 다중 서비스 / 키 회전 / 외부 검증자(JWKS 노출 가능성) 를 고려해 비대칭. dev 키는 `src/main/resources/keys/` (gitignored), 운영은 K8s Secret 마운트.
- **5단계 hierarchy 구조**: `SUPER_ADMIN` ⊇ `{PRODUCT_ADMIN, ORDER_ADMIN, CS_MANAGER}` ⊇ `USER`. 형제 admin 끼리는 inherit 안 함 — least-privilege (`PRODUCT_ADMIN` 은 product 만, `ORDER_ADMIN` 은 order 만) 를 hierarchy 만으로 강제 가능.
- **placeholder 컨트롤러 vs 도메인 본 컨트롤러**: 본 티켓은 *Spring Security 골격* 만 검증 대상. 실제 cart (OLV-040), admin product (OLV-022), auth (OLV-011) 는 placeholder 를 import-replace 한다.
- **트레이드오프 — Testcontainers Postgres 미사용**: 보안 골격은 DB 와 무관하므로 `SecurityFilterChainIT` 는 DataSource/JPA/Flyway 자동설정 3 종을 끄고 띄움. 결과: docker 가용성과 독립적인 빠른 테스트 (~1.5 초). 도메인 티켓은 여전히 `PostgresIntegrationSupport` 사용.
- **연기된 follow-up**:
  - refresh rotation / blacklist (`member_refresh_tokens` 테이블 + 회전 흐름) — OLV-011 회원 로그인.
  - audience claim 검증 — 외부 호출자 도입 시 OLV-011 또는 후속.
  - SUPER_ADMIN 2FA — feature flag 뒤 별도 티켓 (`llm-wiki/98-security.md:11`).
  - 키 회전 (`kid` + JWKSource 다중 키) — 단일 키만 충분, 추후.
