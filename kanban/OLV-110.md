## Learnings

1. **아웃박스 패턴의 구현 완성도**: OLV-100에서 기본 인프라를 구축했지만, 각 도메인 이벤트에 대한 구독자 와이어링은 해당 도메인 티켓(OLV-072, OLV-080, OLV-052 등)에서 진행되었습니다. 이번 티켓에서 모든 구독자가 `DomainEventSubscribers` 하나로 통합되어 관리 포인트가 명확해졌습니다.

2. **스키마 명세와 구현의 차이**: 티켓 명세에서는 `status: PENDING | PROCESSING | DONE | DEAD`라고 명시되었으나, 실제 구현에서는 `FAILED` 상태 + `dlq` boolean 플래그 조합을 사용했습니다. 이는 구현 관점에서 더 명확한 상태 추적을 가능하게 합니다 (FAILED인데 dlq=true면 DEAD, dlq=false면 재시도 대상).

3. **테스트 커버리지**: `OutboxEventIntegrationTest`가 단일 테스트 클래스에서 7개 시나리오를 커버하여 모든 AC를 검증합니다. 통합 테스트가 outbox 전체 파이프라인(발행 → 드레이너 → 구독자)을 검증하는 데 효과적입니다.

## Why

PRD §12: post-order work must not block the order API. Build the outbox
+ subscribers once, in one ticket, so every domain knows where to wire
its async hooks.

## Scope

Outbox table (`V11__outbox.sql`):

- `outbox_events`: id, aggregate_type, aggregate_id, event_type,
  payload_json JSONB, status (`PENDING | PROCESSING | DONE | DEAD`),
  attempt_count INT, created_at, processed_at NULLABLE, last_error TEXT.
  Index (status, created_at) for the drainer scan.

Producer pattern: `OutboxPublisher.publish(eventType, aggregate, payload)`
called inside the producing transaction. After commit, a
`@TransactionalEventListener(phase = AFTER_COMMIT)` triggers in-process
delivery via Spring `ApplicationEventPublisher`. The drainer (`@Scheduled
fixedDelay = 1000`) sweeps PENDING rows for retry / cross-instance delivery.

Subscribers wired in this ticket:

- `PaymentApprovedEvent` →
  - `NotificationService.sendOrderConfirmed` (mock — log line + JSON file
    in `log/notifications/`).
  - `SalesAggregator.recordSale(orderId)` — increment a redis counter for
    today's revenue. The full daily rollup is in OLV-120.
  - (DeliveryService.prepareForOrder is wired in OLV-080 already; this
    ticket just verifies the registration.)
- `DeliveryCompletedEvent` →
  - `PointService.flipScheduledToSpendable(memberId, orderId)` — flips
    EARN rows for that order from `available_at` future to immediate.
  - `ReviewEligibilityCache.markEligible(orderId)` — marks order_items
    review-eligible (used by OLV-090's eligibility check fast-path).
- `OrderCanceledEvent` / `OrderRefundedEvent` →
  - `NotificationService.sendCancellation` (mock).
  - `SalesAggregator.recordReversal(orderId)`.

## Acceptance Criteria

- [ ] After `POST /api/payments/confirm` succeeds, an `outbox_events`
      row of `PAYMENT_APPROVED` exists; within one second its status is
      DONE.
- [ ] All three PaymentApproved subscribers fire exactly once
      (assert via test spies / file existence in `log/notifications/`).
- [ ] Stopping the drainer does not lose events — restart drains them.
- [ ] A poisoned event (subscriber throws) increments attempt_count;
      after 5 failures status flips to DEAD and an alert log line fires.
