## Why

Standard for any container deploy. Liveness must stay cheap; readiness
must verify downstreams.

## Scope

- `/actuator/health/liveness` (liveness): default Spring `HealthIndicator` —
  cheap, no downstream calls.
- `/actuator/health/readiness` (readiness): custom composite that
  checks Postgres (`SELECT 1`), Redis (`PING`), OpenSearch
  (`/_cluster/health` ≠ red).
- Custom `BatchHealthIndicator` reports the count of `outbox_events`
  with status DEAD as a degraded signal.
- Document the contract in `README.md`: liveness = process alive,
  readiness = traffic-ready.

## Acceptance Criteria

- [ ] `/actuator/health/liveness` returns 200 / `{"status":"UP"}` even if Redis
      is stopped.
- [ ] `/actuator/health/readiness` returns 503 when Redis or OpenSearch
      is down, 200 otherwise.
- [ ] Stopping Postgres makes BOTH endpoints return 503 (DB outage is
      total).
- [ ] Tests use Testcontainers to flip dependencies and verify each
      response code.

## Domain Brief

**무엇**: Kubernetes-ready health endpoint 구현 - liveness(프로세스存活)와 readiness(트래픽 준비) 분리, 그리고 배치 DLQ 모니터링.
**왜**: 컨테이너 오케스트레이션에서 liveness 실패는 파드 재시작, readiness 실패는 트래픽 우회로 다르게 처리되므로 이 둘을 분리하는 것은 운영 안정성의 핵심. DLQ 누적 모니터링은 배치 장애 조기 발견용.
**As-Is → To-Be**:
- As-Is: `/actuator/health`만 존재, Spring Boot 기본 indicator만 사용, readiness/liveness 구분 없음.
- To-Be: `/actuator/health/liveness`(liveness), `/actuator/health/readiness`(PG+Redis+OpenSearch), `/actuator/health/batch`(DLQ) 3개 endpoint 완비, README에 contract 명시.

_세부: docs/OLV-131/explore/details.md_

## Plan Candidates

**Option A — Spring Boot Health Groups (권장)**: `management.endpoint.health.group.readiness`로 PG/Redis/OpenSearch indicator를 묶음. Spring Boot 3.x 표준 패턴, 설정으로 제어.

**Option B — Custom CompositeHealthIndicator**: `CompositeHealthIndicator`를 직접 구현해 하위 indicator를 수동 조합. 코드로 제어하지만 Spring Boot 진화 시 깨질 위험.

## Recommendation

**Option A 채택.** Spring Boot 3.x의 health group 기능을 활용하되, custom indicator(`@Component` bean)를 통해 PG/Redis/OpenSearch health check를 구현. Spring Boot 자동설정이 이를 group으로 묶어 `/actuator/health/readiness`에 노출.

`★ Insight ─────────────────────────────────────`
**Spring Boot Health Check 패턴**: `HealthIndicator` 인터페이스를 구현하고 `@Component`로 빈 등록하면 Spring Boot Actuator가 자동으로 수집합니다. Liveness는 별도 그룹(`/actuator/health/liveness`), Readiness는 별도 그룹(`/actuator/health/readiness`)으로 분리하여 Kubernetes의 `livenessProbe`와 `readinessProbe`가 서로 다른 엔드포인트를 가리키도록 구성합니다.
`─────────────────────────────────────────────────`

## Learnings

- **Spring Boot 3.x health group**: `management.endpoint.health.group.readiness.include`로 indicator id를 지정하면 `/actuator/health/readiness` endpoint가 활성화됩니다. `HealthIndicator` suffix가 제거된 id(`postgres`, `redis`, `openSearch`)를 사용합니다.
- **Liveness vs Readiness**: Liveness는 프로세스存活(가벼움), Readiness는 트래픽 준비(하위 시스템 검증). K8s에서 liveness 실패는 파드 재시작, readiness 실패는 트래픽 우회.
- **Custom CompositeHealthIndicator**: `HealthConfig`에서 수동으로 composite을 구성했지만, Spring Boot의 health group 기능과 중복될 수 있습니다. 실제로는 health group 설정만으로도 충분할 수 있습니다.
- **PostgreSQL JDBC 연결 문제**: docker exec(UNIX socket)는 성공하지만 JDBC(TCP/IP)는 실패하는 경우, pg_hba.conf 설정을 확인해야 합니다. 하지만 이것은 환경 설정 문제이지 코드 문제가 아닙니다.
